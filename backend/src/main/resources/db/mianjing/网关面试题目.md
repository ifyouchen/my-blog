## 一、项目整体介绍类（必须流畅）

### Q1. 请介绍一下你做的这个项目

<strong>【标准回答】</strong>

xy-ccp-gateway 是美团消费金融的**外部机构通信网关**，核心职责是在美团内部信贷系统和 20+ 家外部金融机构（银行、支付公司、导流平台）之间做协议翻译和安全通信。

项目解决三个核心问题：

1. **协议适配**：各机构报文格式不同（XML、JSON、Form 键值对），网关统一抹平差异
2. **安全通信**：各机构加密方式不同（AES、RSA、SM4、MAC 等），网关统一加解密
3. **高可用网络**：金融专线的故障自动切换，保障资金链路不中断

架构上是一个标准的 Maven 多模块 Spring Boot 项目，对内通过 **Thrift RPC** 暴露服务，对外通过 **RESTful HTTP** 接收机构回调。技术栈包括 Spring Boot、MyBatis+Zebra（美团分布式数据库中间件）、CAT 监控、Mafka 消息队列、Crane 分布式任务调度。

<strong>【可能追问】</strong>

- 你在项目中负责哪些模块？→ 重点说新接口体系（`newservice` 包）的设计与实现，以及专线高可用改造

### Q2. 项目最大的技术挑战是什么？你是怎么解决的？

<strong>【标准回答】</strong>

最大挑战是**接入机构数量多（20+）、协议差异大，如何保证可扩展性同时不让存量代码越来越乱**。

早期系统用大量 `if-else` 按 `BankType` 枚举分支处理，每接一个新机构就要改主流程，测试风险高，代码越来越难维护。

我们重新设计了一套**工厂 + 策略模式**体系，把报文格式、加密算法、响应解析、回调处理、路由规则这五个维度都抽象成 `Strategy Interface + Factory`：

```java
// 以加密为例，所有加密实现都实现 EncryptionService 接口public interface EncryptionService {
    String encryptMessage(String message, PartnerSecretKey key, TradeCode tradeCode, ProductNo productNo);
    String decryptMessage(String message, PartnerSecretKey key, TradeCode tradeCode, ProductNo productNo);
    EncryptionType getEncryptionType(); // 标识自己支持哪种加密
}

// 工厂通过枚举匹配，由 Spring 注入所有实现的 List@Servicepublic class EncryptionServiceFactory {
    @Resource
    private List<EncryptionService> encryptionServiceList;

    public EncryptionService getEncryptionService(EncryptionType encryptionType) {
        for (EncryptionService service : encryptionServiceList) {
            if (encryptionType == service.getEncryptionType()) {
                return service;
            }
        }
        return null;
    }
}

```

新接一个机构只需：

1. 新增一个 `XxxEncryptionService implements EncryptionService`，加 `@Service`
2. 在数据库里配置该机构用哪种加密类型

主流程代码 `PartnerAppService` 完全不需要改动。

<strong>【可能追问与拓展】</strong>

- <strong>为什么用枚举 match 而不是 Map？</strong>
枚举 match 编译期安全，不会出现 key 写错的情况；而且 `EncryptionType` 枚举本身就是机构配置的一部分，语义更清晰。
- **Spring 注入 **`List<T>`<strong> 是什么原理？</strong>
Spring 会自动收集容器中所有实现了 `EncryptionService` 接口的 Bean，按声明顺序注入到 List 中。这依赖 Spring 的类型匹配注入能力，本质上是 `ApplicationContext.getBeansOfType()` 的封装。
- <strong>有没有更好的实现方式？</strong>
可以用 `Map<EncryptionType, EncryptionService>`，启动时遍历所有实现 Bean 构建 Map，查找效率从 O(n) 提升到 O(1)。但当前机构数量不多（14个加密实现），O(n) 实际影响可忽略不计。

## 二、设计模式类（高频）

### Q3. 你们是怎么用工厂模式的？和简单工厂、抽象工厂有什么区别？

<strong>【标准回答】</strong>

我们项目里用了**两种工厂模式**，根据场景不同而选择：

<strong>方式一：Spring 注入 List + 枚举匹配（</strong>`MessageServiceFactory`<strong>、</strong>`EncryptionServiceFactory`<strong>）</strong>

```java
// MessageServiceFactory.java@Servicepublic class MessageServiceFactory {
    @Resource
    private List<MessageService> messageServiceList; // Spring 自动收集所有实现

    public MessageService getMessageService(PartnerMessageConfig config) {
        MessageType messageType = config.getMessageType();
        for (MessageService service : messageServiceList) {
            if (messageType == service.getMessageType()) {
                return service;
            }
        }
        return null;
    }
}

```

<strong>方式二：ApplicationContext 动态发现 + 接口方法匹配（</strong>`ProductServiceFactory`<strong>，老版本）</strong>

```java
// ProductServiceFactory.java - 实现 ApplicationContextAwarepublic static <T extends ProductAware> T getService(ProductNo productNo, Class<T> commandClazz) {
    Map<String, T> id2Bean = applicationContext.getBeansOfType(commandClazz);
    for (T bean : id2Bean.values()) {
        // 每个实现类通过 serverProducts() 声明自己服务哪些产品
        if (bean.serverProducts().contains(productNo)) {
            return bean;
        }
    }
}

```

**方式三：Stream + **`isFit()`<strong> 方法的责任链变体（</strong>`RequestPartnerRouteFactory`<strong>）</strong>

```java
// RequestPartnerRouteFactory.javapublic RequestPartnerRouteService getRequestPartnerRouteService(
        ProductNo productNo, PartnerCode partnerCode, TradeCode tradeCode) {
    return requestPartnerRouteServiceList.stream()
        .filter(service -> service.isFit(productNo, partnerCode, tradeCode))
        .findFirst()
        .orElse(null);
}

```

<strong>三种工厂模式对比：</strong>


|  | 简单工厂 | 工厂方法 | 抽象工厂 |
|---|---|---|---|
| 扩展方式 | 改工厂类 | 新增工厂子类 | 新增工厂族 |
| 开闭原则 | ❌不满足 | ✅满足 | ✅满足 |
| 本项目 | 早期 `if-else` 就是简单工厂 | 当前 List+枚举方式接近工厂方法 | 不涉及 |

<strong>【可能追问】</strong>

- <strong>策略模式和工厂模式有什么关系？</strong>
工厂模式负责**创建/选择**哪个策略，策略模式定义**怎么执行**。两者通常配合使用：工厂选策略，策略做业务。
- <strong>项目里有没有用到模板方法模式？</strong>
有，`AbsNBMessageService` 和 `AbsNBMessageServiceV2` 就是模板方法——父类定义报文组装的骨架流程，子类重写差异化的字段填充逻辑。

### Q4. 你们的责任链模式是怎么实现的？

<strong>【标准回答】</strong>

在请求发送环节，我们用了责任链模式处理特殊场景（如文件上传、Form 表单提交等），这些场景不适合走通用的 JSON/XML HTTP 发送逻辑：

```java
// RequestService.java - 责任链核心逻辑public String sendRequestMessage(BusinessRequest businessRequest, ...) {
    // 遍历所有 RequestSender 实现，找第一个"认领"当前请求的
    for (RequestSender requestSender : requestSenders) {
        GatewayRequest gatewayRequest = new GatewayRequest(
            businessRequest, partnerMessageConfig, partnerTradeConfig, urlMap, requestJson);
        if (requestSender.fit(gatewayRequest)) {
            // 找到匹配的 Sender，直接处理并返回
            return requestSender.sendRequest(gatewayRequest).getResponseContent();
        }
    }
    // 所有特殊 Sender 都不认领，走通用降级逻辑
    return sendRequestMessage(requestJson, partnerTradeConfig, urlMap);
}

```

每个 `RequestSender` 实现类只需关注 `fit()` 方法（自己是否适配）和 `sendRequest()` 方法（具体怎么发），对其他 Sender 完全透明。

<strong>【可能追问】</strong>

- <strong>责任链和策略模式有什么区别？</strong>
策略模式是**选一个**执行；责任链是**依次判断**，直到某个节点处理（或全部不处理）。本项目 `RequestSender` 更接近责任链——先找能处理的，找不到再走默认逻辑。
- <strong>责任链有什么缺点？</strong>
链过长时性能损耗，且调试复杂。本项目 `RequestSender` 实现类不多，可以接受。

## 三、Spring 与框架类

### Q5. InitializingBean 是什么？你们是怎么用的？

<strong>【标准回答】</strong>

`InitializingBean` 是 Spring 提供的生命周期回调接口，实现了 `afterPropertiesSet()` 方法的 Bean 会在**所有属性注入完成后、Bean 被放入容器之前**自动调用该方法。

我们的 `InitialConfig` 用它在系统**启动时预加载所有关键配置**到内存：

```java
@Configurationpublic class InitialConfig implements InitializingBean {
    // 四个静态 Map，存放运行时频繁访问的配置
    private static Map<String, RSAPrivateCrtKey> tlpRSAKeyMap = new HashMap<>();
    private static Map<String, String> secretKeyMap = new HashMap<>();
    private static Map<String, TradeCode> tradeCodeMap = new HashMap<>();
    private static Map<String, PartnerCode> partnerCodeMap = new HashMap<>();

    @Override
    public void afterPropertiesSet() {
        initTLPConfig();    // 加载通联支付商户 P12 证书 → RSAPrivateCrtKey 对象
        initPartnerConfig(); // 加载所有机构密钥、TradeCode 映射
    }

    public void initTLPConfig() {
        List<TlpPayMerchant> merchants = tlpPayMerchantRepository.getAll();
        for (TlpPayMerchant merchant : merchants) {
            // 从 KMS 获取证书内容（已 AES 加密），解密后解析 P12 证书
            String certContent = AesConfig.getNewKMSValue(merchant.getMerchantSecretKey());
            RSAPrivateCrtKey key = ReadRSAKeyHelper.getRSAPrivateKey(certContent, p12Password);
            tlpRSAKeyMap.put(merchant.getMerchantSecretKey(), key);
        }
    }
}

```

<strong>为什么要预加载到静态 Map？</strong>

- 每次请求都加密/解密，不能每次都查数据库（性能问题）
- KMS 调用是远程调用，也不能每次都发起
- 静态 Map + `volatile` 引用替换保证热更新时的线程安全

**配置不可用时快速失败**：

```java
public static Map<String, RSAPrivateCrtKey> getRSAKeyMap() {
    if (CollectionUtils.isEmpty(tlpRSAKeyMap)) {
        // 配置为空直接抛异常，而不是返回 null 让下游 NPE
        throw new GateWayInternalParamException("路由配置表初始化时为空, 请检查！");
    }
    return tlpRSAKeyMap;
}

```

<strong>【可能追问与八股延伸】</strong>

- <strong>Bean 的生命周期说一下？</strong>
实例化 → 属性填充（@Resource/@Autowired） → `postProcessBeforeInitialization`（BeanPostProcessor）→ `afterPropertiesSet`（InitializingBean）→ `init-method` → `postProcessAfterInitialization`（AOP代理在此生成）→ 使用 → `destroy-method`
- `@PostConstruct`** 和 **`InitializingBean`<strong> 有什么区别？</strong>`@PostConstruct` 是 JSR-250 注解，是 Java 标准，不依赖 Spring；`InitializingBean` 是 Spring 接口，与框架耦合。执行顺序：`@PostConstruct` 先于 `afterPropertiesSet()`。
- **如果 **`afterPropertiesSet()`<strong> 里抛异常会怎样？</strong>
Spring 容器启动失败，服务无法启动。这是我们**故意的设计**——配置缺失宁可不启动，也不要带着错误配置运行。

### Q6. 你们项目里 @Resource 和 @Autowired 用的是哪个？有什么区别？

<strong>【标准回答】</strong>

项目里统一用 `@Resource`（JSR-250 标准注解）。

核心区别：


|  | `@Autowired` | `@Resource` |
|---|---|---|
| 来源 | Spring 专有 | JDK 标准（JSR-250） |
| 默认匹配 | **按类型**（byType） | **按名称**（byName），找不到再 byType |
| 指定 Bean | `@Qualifier("beanName")` | `@Resource(name="beanName")` |
| 注入 `null` | `required=false` 可允许 | 默认找不到直接报错 |

实际场景中区别体现在注入 `List<EncryptionService>` 时：

- `@Autowired` 会按类型收集所有 `EncryptionService` 实现
- `@Resource` 此时需要指定名称，否则会报"找到多个 Bean"

<strong>【可能追问】</strong>

- **你们为什么选 **`@Resource`** 而不是 **`@Autowired`<strong>？</strong>
团队规范，JSR 标准注解减少对 Spring 的硬依赖，理论上更易迁移。在实际大型项目里两者差异不大，遵循团队规范最重要。

## 四、并发与线程安全类

### Q7. 你们配置热更新是怎么保证线程安全的？为什么不用 ConcurrentHashMap？

<strong>【标准回答】</strong>

这是项目里一个很经典的并发设计决策。

**问题背景**：`InitialConfig` 里的静态 Map 在系统运行时会被高频读（每次请求都需要查密钥），偶尔写（触发 Crane 任务热更新）。

<strong>我们的方案：</strong>`volatile`** 引用 + 全量 Map 替换**

```java
// 声明时用 volatile 保证可见性private static volatile Map<String, String> secretKeyMap = new HashMap<>();

// 热更新时：不是 put() 增量更新，而是创建新 Map 全量赋值public void initPartnerConfig() {
    Map<String, String> newSecretKeyMap = new HashMap<>();  // 新 Map
    for (PartnerSecretKey key : partnerSecretRepository.queryAll()) {
        String value = AesConfig.getNewKMSValue(key.getSecretKey());
        newSecretKeyMap.put(key.getSecretKey(), value);     // 填入新 Map
    }
    secretKeyMap = newSecretKeyMap;  // 原子性引用替换（volatile 保证可见性）
}

```

<strong>为什么不用 ConcurrentHashMap？</strong>

`ConcurrentHashMap` 能保证单次 `put/get` 操作的线程安全，但不能保证**批量更新的原子性**：

```text
场景：更新 50 个机构的密钥
ConcurrentHashMap 方案：
put(key1, newVal1)  ← 此时已可见！
put(key2, newVal2)  ← 此时 key1 新 key2 旧，状态不一致！
...
put(key50, newVal50) ← 整个更新期间读到的都是"半新半旧"状态

```

而全量替换方案：

```text
方案：
1. 构建完整的 newMap（此时对外不可见）
2. secretKeyMap = newMap（一次引用赋值，JVM 保证引用赋值原子性）
3. 赋值完成后，所有后续读都看到完整的新 Map
```

`volatile` 的作用：禁止指令重排，保证新 Map 完全构建好之后引用才对其他线程可见。

<strong>【可能追问与八股延伸】</strong>

- <strong>volatile 能保证原子性吗？</strong>
不能。`volatile` 只保证**可见性**和<strong>有序性（禁止重排序）</strong>，不保证原子性。`i++` 这种复合操作需要用 `AtomicInteger`。
- <strong>ConcurrentHashMap 的底层实现？</strong>
JDK 8 以后：`Node[]` 数组 + 链表 + 红黑树。锁粒度细化到**数组桶级别**（CAS + synchronized），读操作通常无锁（`volatile Node`），写操作只锁当前桶，并发性远好于 `HashTable`。
- <strong>读写锁（ReadWriteLock）适用这个场景吗？</strong>
理论上也可以，读多写少场景适合用 `ReentrantReadWriteLock`。但引用替换方案更简单，读操作完全无锁开销，性能更好，是更优解。

## 五、消息队列类

### Q8. 你们专线故障自动切换是怎么做的？Mafka 消费中怎么处理延迟消息的问题？

<strong>【标准回答】</strong>

**整体链路**：

```text
Falcon（监控系统）检测到专线 ping 不通
  ↓ 生成 FalconMessage，推送到 Mafka Topic
ZhuanXianSwitchService（Mafka Consumer）消费
  ↓
  1. 过滤非关注指标（careMetric 白名单）
  2. 过滤非 prod 环境消息
  3. 调用 switchZxConfigStatus() → 更新数据库 private_net 表状态为 OFF
  4. 发送大象 IM 通知 RD
  5. 旧版兼容：同步更新 MCC 配置中心

```

**延迟消息问题与解决方案**：

```java
// ZhuanXianSwitchService.javaprivate boolean faultDeplyTime(String time, String alertZXCode) {
    Date alertTime = DateUtil.changeToDate_yyyyMMddHHmmss(time); // 告警发生的时间
    Long diffTime = System.currentTimeMillis() - alertTime.getTime(); // 当前时间 - 告警时间
    // 超过 5 分钟（getFaultDelayTime()），认为是积压的过期消息，不切换
    if (ZhuanXianContant.getFaultDelayTime().compareTo(diffTime) >= 0) {
        return true;
    }
    LOG.error("专线故障时间和消费到的时间差超过5分钟，不切换...");
    return false;
}

```

<strong>为什么需要这个设计？</strong>

场景：消息队列积压时，专线已经故障了 10 分钟，运维手动恢复了专线，但积压的故障告警消息此时才被消费到 → 如果不过滤，会把刚恢复的专线再切回故障状态，导致误切换。

5 分钟窗口的含义：如果消息延迟超过 5 分钟才被消费到，说明这条消息已经"过时"，现实中专线状态可能已经发生变化，由人工确认更安全。

<strong>【可能追问与八股延伸】</strong>

- <strong>Kafka 如何保证消息不丢失？</strong>
生产者：`acks=all`（等待所有 ISR 副本确认）；Broker：`min.insync.replicas≥2`；消费者：手动 commit offset（处理完再提交，避免消费但未处理就提交）。
- <strong>Kafka 如何保证消息不重复消费（幂等性）？</strong>
消息本身幂等（业务逻辑保证同一告警多次处理结果一致）；或者用消息 ID 做去重，消费时先查 Redis/DB 有没有处理过。本项目用时间窗口过滤本质上也是一种去重策略。
- <strong>消息积压了怎么处理？</strong>
扩大消费者数量（增加 Consumer Group 成员，前提是分区数足够）；或者临时增加分区；如果消息不重要可以跳过历史消息直接消费最新的（reset offset）。

## 六、加密与安全类

### Q9. 你们系统支持这么多加密方式，AES 和 RSA 分别用在哪里？有什么区别？

<strong>【标准回答】</strong>

<strong>AES（对称加密）</strong> — 用于大多数机构的报文加解密：

```java
// AesEncryptionService.java// AES/CBC/PKCS5Padding 模式，需要 secretKey（16/24/32字节）+ IV（16字节）Cipher cipher = Cipher.getInstance("AES/CBC/PKCS5Padding");
SecretKeySpec skeySpec = new SecretKeySpec(secretKey.getBytes(), "AES");
IvParameterSpec iv = new IvParameterSpec(ivParameterKey.getBytes());
cipher.init(Cipher.ENCRYPT_MODE, skeySpec, iv);
byte[] encrypted = cipher.doFinal(message.getBytes("utf-8"));
// 结果 Base64 编码后传输return new BASE64Encoder().encode(encrypted);

```

<strong>RSA（非对称加密）</strong> — 用于通联支付、光大部分接口的数字签名：

通联支付不是用 RSA 做加密，而是用来**签名验签**：

```java
// TLPaymentRSAEncryptionService.java// 发送报文：XML 内容 + 我方私钥签名（P12 证书）String encryptMessage = invokeAifPayService.signXml(xmlContent, merchantId);

// 接收响应：验证通联支付返回内容的签名（用通联公钥）if (!invokeAifPayService.verifySign(response)) {
    throw new GateWayInternalParamException("验签失败");
}

```

**国密 SM4** — 用于部分要求国密标准的机构：

```java
// SM4EncryptionService.java - 基于 BouncyCastle 实现static {
    Security.addProvider(new BouncyCastleProvider()); // 注册 BC Provider
}
// SM4/CBC/PKCS5Padding，与 AES 逻辑类似，但算法是国产标准Cipher cipher = Cipher.getInstance("SM4/CBC/PKCS5Padding", BouncyCastleProvider.PROVIDER_NAME);

```

<strong>核心区别：</strong>


|  | AES | RSA |
|---|---|---|
| 类型 | 对称（加解密同一个密钥） | 非对称（公钥加密/私钥解密 或 私钥签名/公钥验签） |
| 速度 | 快，适合大数据量 | 慢，适合小数据量（如密钥交换、签名） |
| 用途 | 报文内容加密 | 数字签名、密钥分发 |
| 密钥分发 | 难（双方都需要安全传递密钥） | 易（公钥公开，私钥保密） |

<strong>【可能追问与八股延伸】</strong>

- <strong>HTTPS 用的是什么加密？</strong>
混合加密：TLS 握手阶段用 RSA 或 ECDH 交换对称密钥，之后数据传输用 AES 对称加密。既保证密钥安全分发，又保证传输效率。
- <strong>SM4 和 AES 有什么区别？</strong>
算法结构类似（都是分组对称加密），但 SM4 是中国国家密码局标准（GM/T 0002），分组长度和密钥长度都是 128 位，在金融行业受监管要求。
- <strong>密钥为什么不能硬编码？</strong>
代码会被 clone、打包、部署到多环境，硬编码密钥有泄露风险；密钥轮换时需要改代码重新部署，运维代价大。我们用 KMS（密钥管理服务）统一托管，密钥加密存储在数据库，运行时通过 `AesConfig.getNewKMSValue()` 动态解密获取。

## 七、高可用设计类

### Q10. 你们专线的主备切换是怎么设计的？如果主线挂了，怎么自动切到备线？

<strong>【标准回答】</strong>

专线选择的核心逻辑在 `NetLineService.getPrivateNet()`：

```java
// NetLineService.javaprivate PrivateNet selectPrivateNet(PartnerCode partnerCode, List<PrivateNet> availableNets) {
    // 先按策略分类
    List<PrivateNet> masterNets = new ArrayList<>();
    List<PrivateNet> slaveNets = new ArrayList<>();
    List<PrivateNet> randomNets = new ArrayList<>();

    for (PrivateNet net : availableNets) {
        switch (net.getNetStrategy()) {
            case NET_MASTER: masterNets.add(net); break;
            case NET_SLAVE: slaveNets.add(net); break;
            case NET_RANDOM: randomNets.add(net); break;
        }
    }

    // 随机模式：只有 random 列表，直接随机选
    if (masterNets.isEmpty() && slaveNets.isEmpty() && !randomNets.isEmpty()) {
        return randomSelect(randomNets);
    }

    // 主备模式：优先选主线（前提：主线在可用列表中）
    if (!masterNets.isEmpty() && !slaveNets.isEmpty()) {
        return randomSelect(masterNets); // 多主线时随机
    }

    throw new GateWayInternalParamException("专线主备属性配置错误");
}

```

**主线故障自动 Failover 的关键**：

1. Falcon 告警 → Mafka 消息 → `ZhuanXianSwitchService` 消费后把主线的 `private_net.status = OFF`
2. 下次 `getPrivateNet()` 查询时，先过滤 `isNetAvailable()`，主线不可用所以不会出现在列表里
3. 此时 `masterNets` 为空，进入 `selectPrivateNet()` 后只剩 `slaveNets`，分支判断失效
4. **实际上会触发 **`GateWayInternalParamException`（配置错误）——这是设计保障，提醒运维主动处理

> 更优的方案（可以主动提出作为改进点）：主线故障时自动 Failover 到备线，而不是直接报错。当前需要运维手动触发备线切主，或通过 Crane Job 维护可用性。

**三种专线拓扑对比**：


| 模式 | `private_net` 配置 | 适用场景 |
|---|---|---|
| 单专线 | 1条记录 | 低成本，单点 |
| 双活随机 | 多条，全部 `NET_RANDOM` | 多条等价专线负载均衡 |
| 主备 | 主线 `NET_MASTER`，备线 `NET_SLAVE` | 高可用，主线优先 |

<strong>【可能追问】</strong>

- <strong>如果两条专线都挂了怎么办？</strong>
抛出 `NoAvailablePrivateNetException`，调用方会收到明确的错误，不会超时等待。同时 CAT 有专线连接异常的独立告警看板，RD 会立即收到通知。
- <strong>为什么用数据库存专线状态而不是用 Redis？</strong>
专线状态需要持久化，服务重启后状态不应该丢失；同时专线状态变更不频繁，数据库的读性能完全够用。

## 八、数据库与持久层类

### Q11. 你们 MyBatis 用的过程中有没有踩过什么坑？

<strong>【标准回答】</strong>

有几个真实踩过的坑：

<strong>坑一：</strong>`#{}`** vs **`${}`** 导致的 SQL 注入风险**

```xml
<!-- 危险：${}直接字符串拼接，外部输入可能注入 SQL -->
SELECT * FROM partner_secret WHERE partner_code = '${partnerCode}'

<!-- 安全：#{}预编译，参数作为占位符 -->
SELECT * FROM partner_secret WHERE partner_code = #{partnerCode}

```

项目中所有外部传入参数统一用 `#{}`。`${}` 只在**排序字段名**（非用户输入）场景下使用。

**坑二：大对象查询内存问题**

`partnerMessageConfigRepository.queryAll()` 会把所有机构配置加载到内存，如果表很大需要分批查询。项目启动时通过 `InitializingBean` 一次性加载是可控的，但要注意表数据量的增长。

**坑三：MyBatis 一级缓存的坑**

MyBatis 默认开启 SqlSession 级别的一级缓存。在同一个 SqlSession 内，相同查询会命中缓存。但如果第一次查询后数据库被其他线程修改了（比如专线状态变更），再次查询仍然返回旧数据。

解决方案：专线状态是强实时性数据，查询时可以用 `@Options(flushCache = Options.FlushCachePolicy.TRUE)` 或直接在 SQL 级别禁用缓存。

<strong>【可能追问与八股延伸】</strong>

- <strong>MyBatis 的一级缓存和二级缓存区别？</strong>
一级缓存：SqlSession 级别，默认开启，Session 关闭自动清除；二级缓存：Mapper/namespace 级别，跨 Session 共享，需要手动开启且实体需要实现 `Serializable`。大多数互联网项目不用二级缓存（数据一致性问题）。
- <strong>Zebra（美团分库分表中间件）是怎么用的？</strong>
Zebra 对应用透明地做读写分离和分库分表。配置分片键（如 `partner_code`），Zebra 自动把请求路由到对应的物理库/表。类似开源的 ShardingSphere。

## 九、异常处理与可观测性类

### Q12. 你们是怎么设计异常体系的？出问题了怎么快速定位是哪层的问题？

<strong>【标准回答】</strong>

我们定义了**四层异常**，每层对应不同的问题域，在 CAT 上有独立的监控维度：

```java
// PartnerInvokeRpcServiceImpl.java - Thrift 层异常翻译try {
    String result = partnerAppService.invokePartner(businessRequest);
} catch (IllegalArgumentException e) {
    throw new TIllegalArgumentException(e.getMessage());  // → 上游参数错误
} catch (PartnerBusinessRuntimeException e) {
    throw new TPartnerBusinessException(e.getMessage());  // → 机构业务拒绝（正常）
} catch (PartnerConnectException e) {
    throw new TGateWayRuntimeException(e.getMessage());   // → 网络连通性问题
} catch (Exception e) {
    throw new TGateWayRuntimeException(e.getMessage());   // → 网关内部错误
}

```


| 异常类型 | 含义 | 对应 CAT 事件 | 处理优先级 |
|---|---|---|---|
| `GateWayInternalParamException` | 网关配置缺失/数据库配置错误 | `GATEWAY_INTERNAL_PROCESS` | P0，立即处理 |
| `PartnerConnectException` | 专线/公网不通 | `PARTNER_CONNECTION` | P0，检查网络 |
| `DecryptionException` | 密钥配置错误或证书过期 | `PARTNER_QUEST_RETURN_DECRYPTION` | P1，检查密钥 |
| `PartnerBusinessRuntimeException` | 机构侧正常拒绝（如额度不足） | `BUSINESS_INVOKE` | P2，业务处理 |

**CAT 监控维度**：

```java
// MonitorService.java// 双维度 Transaction：业务耗时 + 机构接口耗时
Cat.newTransaction(productNo.getNo(), partnerCode.getPartnerCode()); // 业务维度
Cat.newTransaction(partnerCode.getPartnerCode(), tradeCode.getTradeCode()); // 机构维度

// MetricHelper 流量大盘（三维度）
MetricHelper.build().name("产品-资方-业务流量大盘")
    .tag("产品号", productNo.getName())
    .tag("业务类型", tradeCode.getTradeName())
    .tag("外部机构", partnerCode.getPartnerName()).count(1);

```

**定位问题的步骤**：

1. CAT 看 Transaction 成功率 → 确认是哪个机构/哪条链路出问题
2. 看对应的异常类型 → 判断是网络层、密钥层还是业务层
3. 看 CAT 日志 → 找到具体的请求和响应报文（已脱敏）
4. 专线问题 → 直接看 Falcon 告警

<strong>【可能追问】</strong>

- <strong>CAT 和 SkyWalking 有什么区别？</strong>
CAT 是美团内部 APM，功能侧重问题诊断（Transaction 耗时、Event 统计）；SkyWalking 是开源全链路追踪，更侧重分布式 Trace 追踪（TraceId 贯穿全链路）。本质上都是 APM 工具，原理类似。
- <strong>全链路日志脱敏是怎么做的？</strong>`DesenseUtil.desensitizeJson()` 对 JSON 中的敏感字段（手机号、身份证、银行卡）做正则 mask（如 `138****8888`）。所有含用户信息的日志打印强制经过脱敏处理，代码 Review 时重点检查。

## 十、架构与系统设计类

### Q13. 如果让你来优化这个网关的性能，你会从哪些方向入手？

<strong>【标准回答】</strong>

结合项目现有架构，我会从以下几个方向优化：

<strong>1. HTTP 连接池复用（当前最可能的优化点）</strong>

现有代码向外部机构发 HTTP 请求，需要确认是否复用了连接池。如果每次请求都新建连接，对于高频的放款/查询接口影响明显：

```java
// 应该使用 HttpClientBuilder 配置连接池CloseableHttpClient httpClient = HttpClients.custom()
    .setMaxConnTotal(200)          // 最大连接数
    .setMaxConnPerRoute(20)        // 每个 Host 最大连接数
    .setConnectionTimeToLive(60, TimeUnit.SECONDS) // 连接存活时间
    .build();

```

**2. 查询热点配置引入本地缓存**

`partnerMessageConfigRepository.getByProductNoAndPartnerCode()` 在每次请求时都查数据库。机构配置变更频率极低（只在接入新机构时），完全可以用 Guava Cache 加本地缓存：

```java
Cache<String, PartnerMessageConfig> cache = CacheBuilder.newBuilder()
    .expireAfterWrite(30, TimeUnit.MINUTES)
    .maximumSize(500)
    .build();

```

**3. 异步化非核心链路**

CAT 埋点、大象 IM 通知、日志打印等非核心操作，不应阻塞主链路。可以用线程池异步处理。

**4. 数据库索引优化**

`private_net` 表按 `partner_code` 和 `net_code` 双查询场景，需要确认两个字段都有索引。

<strong>【可能追问】</strong>

- <strong>Guava Cache 和 Redis 有什么区别？</strong>
Guava Cache 是本地缓存，数据在 JVM 堆内，访问极快（纳秒级），但多实例间不共享；Redis 是分布式缓存，多实例共享，但有网络 IO 延迟（毫秒级）。配置类数据用本地缓存更合适（访问频率高、一致性要求低）。

### Q14. Thrift 是什么？和 HTTP 接口有什么区别？你们为什么内部通信用 Thrift？

<strong>【标准回答】</strong>

Thrift 是 Facebook 开源的**跨语言 RPC 框架**，通过 IDL（接口定义语言）生成各语言的客户端/服务端代码。

我们内部系统之间用 Thrift，外部机构回调用 HTTP，原因如下：


|  | Thrift RPC | HTTP REST |
|---|---|---|
| 协议 | 二进制协议（TBinaryProtocol） | 文本协议（JSON） |
| 性能 | 序列化体积小，速度快 | 体积较大，速度慢 |
| 契约 | IDL 强约束，接口不匹配编译报错 | 无强约束，兼容性靠文档 |
| 适用场景 | 内部高频服务调用 | 开放接口、外部对接 |

**项目中的具体体现**：

```java
// TPartnerInvokerRpcService 是 Thrift IDL 生成的接口// 内部业务系统（信贷核心）通过 Thrift Client 调用网关public interface TPartnerInvokerRpcService {
    String invokePartner(TBusinessRequest request)
        throws TIllegalArgumentException, TPartnerBusinessException, TGateWayRuntimeException;
}

// 网关实现这个接口，向外暴露@Servicepublic class PartnerInvokeRpcServiceImpl implements TPartnerInvokerRpcService {
    @Override
    public String invokePartner(TBusinessRequest tBusinessRequest) {...}
}

```

外部机构不会用 Thrift，所以回调接口用 HTTP，通过 `PartnerCallBackController` 暴露 RESTful API。

<strong>【可能追问】</strong>

- <strong>gRPC 和 Thrift 有什么区别？</strong>
都是高性能 RPC 框架，gRPC 基于 HTTP/2 + Protobuf，更现代，支持流式传输和双向流；Thrift 更老牌，支持的语言更多，美团内部有深厚的 Thrift 生态积累。

## 十一、新老接口迁移类（系统演进经验）

### Q15. 你们是怎么做新老接口平滑迁移的？有没有遇到什么问题？

<strong>【标准回答】</strong>

这个迁移是我做过的比较有挑战的工作。背景是：老接口基于 `BankType` 枚举硬编码，扩展性差；新接口基于数据库配置驱动，灵活可扩展。但老接口已经稳定运行，不能一次性切换，必须**逐机构灰度迁移**。

**核心策略**：MCC 配置中心维护"已迁移机构白名单"

```java
// PartnerCallBackController.java - 回调入口的兼容判断@RequestMapping(value = "/{partnerCallBackName}/credit", method = RequestMethod.POST)public String partnerCallBack(@RequestBody String callBackMessage,
                              @PathVariable String partnerCallBackName) {
    // 查 MCC 配置：该机构是否已迁移到新接口
    if (!GateWayMccUtil.isNewCallbackInterface(partnerCallBackName)) {
        LOG.error("【未迁移机构】走到了新接口，自动降级到老接口处理");
        return useOldCallbackInterface(partnerCallBackName, callBackMessage);
    }
    // 已迁移 → 走新逻辑
    NotifyResult result = partnerCallBackService.invokeTradeService(...);
    ...
}

```

**迁移步骤**（每个机构重复）：

1. 在新接口框架下新增该机构的 `MessageService`、`EncryptionService` 实现
2. 在测试环境验证新接口逻辑
3. 将该机构加入 MCC 白名单（此时线上新接口生效）
4. 观察 CAT 监控：`monitorOldOrNewInvocation()` 同时上报 `newAPI` 和 `oldAPI` 流量占比
5. 确认 `newAPI` 流量 100% 后，从老接口代码中删除该机构逻辑

**遇到的问题**：光大银行的 GBK 编码问题。老接口和新接口处理编码的方式不同，迁移过程中出现了回调报文乱码。

```java
// 光大银行特殊处理：HTTP 容器默认 ISO-8859-1 接收 Body，需要字节级重新解释
callBackMessage = new String(
    StringEscapeUtils.unescapeHtml(callBackMessage).getBytes("iso8859-1"),
    StandardCharsets.UTF_8);

```

这个问题在 UAT 环境没复现（UAT 用的是光大测试报文，没有中文），生产才出现，排查了半天。教训：编码问题必须用生产级别的报文测试。

<strong>【可能追问】</strong>

- <strong>灰度发布还有哪些方式？</strong>
配置中心开关（本项目方案）；Feature Flag；服务路由（不同流量打到不同版本的服务）；蓝绿发布（同时维护两个版本，DNS 切流）；金丝雀发布（小比例流量打到新版本）。
- <strong>如果迁移出问题了怎么回滚？</strong>
直接在 MCC 配置中心把该机构从白名单移除，立即生效，回滚零停机。

## 十二、综合问题

### Q16. 你在这个项目里最有成就感的一件事是什么？

<strong>【参考回答（根据自身情况调整）】</strong>

最有成就感的是**专线自动故障切换系统**的设计与实现。

在这之前，专线故障的处理流程是：Falcon 告警 → RD 收到报警短信 → 登录后台手动切换配置 → 通知相关方。整个过程至少需要 10-15 分钟人工介入，期间所有走专线的请求都会失败。

上线后，专线故障的响应时间从 10+ 分钟缩短到 1 分钟内，而且全程无需人工介入。这也是我第一次把消息队列用在运维自动化场景中，收获很多。

## 十三、KMS 密钥管理类

### Q17. 你们的加密密钥是怎么管理的？为什么不能硬编码在代码里？

<strong>【标准回答】</strong>

我们所有加密密钥都通过 <strong>KMS（密钥管理服务）</strong> 托管，完全没有硬编码。整体链路是这样的：

```text
数据库 partner_secret_key 表存储密钥引用（不是明文密钥，是 KMS Key 名称）
     ↓
运行时 AesConfig.getNewKMSValue(kmsKeyName)
     ↓ 向 KMS 发起请求
KMS 返回解密后的密钥明文（密钥本身在 KMS 里用 master key 加密存储）
     ↓
InitialConfig.afterPropertiesSet() 把解密后的密钥加载到内存 Map
     ↓
加解密操作从内存 Map 中取密钥，不再每次请求 KMS
```

<strong>为什么不能硬编码：</strong>

1. **代码会被 Clone 和分发**：代码仓库、打出来的 Jar 包、各个环境的部署产物，任何一个泄露，所有机构的通信密钥就全暴露了。金融场景下这是极其严重的安全事故。
2. **密钥轮换成本高**：金融监管要求定期更换密钥。硬编码的话每次轮换都要改代码、走发布流程，而用 KMS 只需在密钥管理后台更新，触发 Crane 热更新即可，无需重启服务。
3. **多环境隔离**：dev / staging / prod 各自有独立的密钥，硬编码无法做到这点，KMS 可以按环境隔离。

<strong>P12 证书（通联支付）的特殊处理：</strong>

```java
// InitialConfig.java// P12 证书本身也加密存储在 KMS，取出后用密码解包，提取 RSAPrivateCrtKey 对象String certContent = AesConfig.getNewKMSValue(merchant.getMerchantSecretKey());
RSAPrivateCrtKey key = ReadRSAKeyHelper.getRSAPrivateKey(certContent, p12Password);
tlpRSAKeyMap.put(merchant.getMerchantSecretKey(), key);
// 缓存的是解析好的 Key 对象（不是证书字符串），避免每次请求都解析 P12
```

<strong>【可能追问】</strong>

- <strong>密钥热更新的时候，正在处理中的请求怎么办？</strong>
全量 Map 替换是瞬时的（一次引用赋值），正在进行的请求持有的是旧 Map 的引用，不受影响；新来的请求取到新 Map，用新密钥。新老密钥有一个短暂的"双活窗口"，对于 AES 对称加密来说这是安全的。
- <strong>如果 KMS 不可用（网络故障），系统能正常运行吗？</strong>
能。KMS 只在启动时和热更新时调用，日常请求直接从内存 Map 取密钥，不依赖 KMS 的实时可用性。但如果重启服务时 KMS 不可用，`afterPropertiesSet()` 会抛异常导致启动失败——这是合理的 Fail-Fast 设计。

## 十四、通联支付多商户路由类

### Q18. 你们通联支付的商户路由是怎么设计的？为什么要单独处理？

<strong>【标准回答】</strong>

通联支付是美团消费金融对接的一个重要支付通道。特殊之处在于：<strong>美团平台下有多个产品线都用通联支付通道，但通联支付按商户号（</strong>`MERCHANT_ID`<strong>）区分业务归属，不同产品线必须使用不同的商户号</strong>，否则资金会混在一起，对账会乱。

<strong>整体设计：</strong>

```text
数据库 tlp_pay_merchant 表
  ├── product_no (产品线)
  ├── merchant_id (通联商户号)
  └── merchant_secret_key (KMS 中 P12 证书的引用)

系统启动时 InitialConfig.initTLPConfig():
  → 按 product_no 遍历所有商户记录
  → KMS 解密 P12 证书 → 解析 RSAPrivateCrtKey
  → 缓存到 tlpRSAKeyMap<merchantSecretKey, RSAPrivateCrtKey>

请求时 RequestService.parseTlpUrl():
  → TlpParseUtil.getMerchantId(businessRequest) 从 productNo 查表找商户号
  → URL 拼接 ?MERCHANT_ID=xxx&REQ_SN=yyy（通联要求在 URL Query String 里传）

```

<strong>URL 动态构造代码：</strong>

```java
// RequestService.javaprivate String parseTlpUrl(String url, BusinessRequest businessRequest) {
    String merchantId = TlpParseUtil.getMerchantId(businessRequest); // 按产品号路由商户
    String reqNo = TlpParseUtil.getRequestNo(businessRequest.getTradeJson()); // 从业务参数提取流水号
    if (StringUtils.isEmpty(merchantId) || StringUtils.isEmpty(reqNo)) {
        throw new GateWayInternalParamException("解析商户号和流水号异常");
    }
    return url + "?MERCHANT_ID=" + merchantId + "&REQ_SN=" + reqNo;
}

```

<strong>为什么流水号也要放 URL 里？</strong>
通联支付的接口设计要求用 `MERCHANT_ID + REQ_SN` 做幂等控制，这是通联支付的协议约定，网关必须适配。

<strong>【可能追问】</strong>

- <strong>如果新增一个产品线用通联支付，需要改什么代码？</strong>
不需要改代码，只需：1. 在通联支付申请一个新商户号；2. 在 `tlp_pay_merchant` 表里插一条记录；3. 触发 Crane 热更新 `InitialConfig.initTLPConfig()`。完全配置驱动，零代码改动。
- <strong>多商户的 RSA 私钥各不相同，怎么知道用哪个私钥签名？</strong>`TLPaymentRSAEncryptionService.encryptMessage()` 从报文里解析出 `MERCHANT_ID`（报文组装阶段已经把商户号填入），再去 `tlpRSAKeyMap` 里找对应的私钥。

## 十五、Thrift 泛化调用类

### Q19. 你们回调通知下游系统用的是什么机制？Thrift 泛化调用是什么？

<strong>【标准回答】</strong>

网关接收到外部机构回调后，需要通知下游的信贷核心系统。常规做法是网关直接依赖信贷核心系统的 Client 包，调用具体方法。

但这样有个问题：**网关是基础设施服务，不应该强依赖各个业务系统的具体接口**——如果每个下游系统都往网关里打进依赖，网关会变成大杂烩，且任何下游的接口变更都可能影响网关重新发版。

我们用了 MT-Thrift 的**泛化调用（Generic Service）**来解耦：

```java
// PartnerInvokeGenericRemoteService.java// 不需要导入下游系统的 Thrift Client 包GenericService genericService = (GenericService) thriftProxyFactory.getProxy(config).getObject();

// 通过方法名字符串 + 参数列表 动态调用，而不是调用具体的接口方法
List<String> paramValues = buildParamValues(message, partnerMessageConfig, tradeCode);
String response = genericService.$invoke(config.getMethod(), null, paramValues);

```

<strong>泛化调用的原理：</strong>
Thrift 框架在二进制协议层面，调用任何方法最终都是序列化一个包含「方法名 + 参数列表」的请求包。泛化调用绕过了 IDL 生成的强类型客户端，直接构造这个请求包，实现在**不导入 IDL 的情况下调用任意 Thrift 方法**。

<strong>路由配置（</strong>`service_list_config`<strong> 表）：</strong>

```text
service_list_config 表存储：
  - trade_code：业务交易码（如授信回调、风控回调）
  - appkey：下游系统的 Thrift AppKey
  - method：要调用的方法名
  - 超时时间、重试次数等

```

不同的回调业务（授信/风控/还款）路由到不同的下游服务，只需改数据库配置，网关代码不动。

<strong>【可能追问】</strong>

- <strong>泛化调用和正常 Thrift 调用的优缺点对比？</strong>|  | 正常调用 | 泛化调用 |
|---|---|---|
| 依赖 | 需要 Client 包 | 无需 Client 包 |
| 类型安全 | 编译期检查 | 运行时才知道方法对不对 |
| 性能 | 略好（少一次反射） | 略低 |
| 适用场景 | 稳定的强依赖调用 | 平台层、低耦合调用 |
- <strong>如果下游系统改了方法签名，怎么知道？</strong>
运行时报错——这是泛化调用的缺点。所以我们在 `ServiceListConfig` 表里记录方法签名，下游改接口时需要同步更新配置，并有集成测试覆盖。

## 十六、回调双版本协议类

### Q20. 你们回调 V1 和 V2 协议有什么区别？为什么要维护两套？

<strong>【标准回答】</strong>

回调 V1 和 V2 的核心区别是**响应体的丰富程度**：

```java
// V1：NotifyResult（简单 STATUS + MESSAGE）public class NotifyResult {
    private String STATUS;   // S=成功, F=失败
    private String MESSAGE;  // 附加信息
}

// V2：NotifyResponseResult（包含业务数据，支持富体响应）public class NotifyResponseResult {
    private String status;
    private String message;
    private Map<String, Object> data; // 额外业务字段，如还款金额、账单号等
}

```

<strong>为什么需要 V2：</strong>
早期回调只需要告诉机构"成功/失败"，V1 够用。但随着业务发展，部分机构（如银行侧还款申请）需要网关在响应里返回更多业务字段（如实际扣款金额、银行流水号），V1 的固定字段无法满足，所以引入了 V2。

<strong>两套 Factory 并行：</strong>

```java
// PartnerCallbackService.java// V1 工厂@Resourceprivate CallBackFactory callBackFactory;

// V2 工厂@Resourceprivate CallBackFactoryV2 callBackFactoryV2;

// 回调处理时按 tradeCode 找对应版本的处理器public NotifyResult invokeTradeService(...) {  // V1
    TradeCallBackService service = callBackFactory.getCallBackTradeService(tradeCode);
    return service.invokeTradeService(hasDecryptedMessage, partnerMessageConfig);
}

public NotifyResponseResult invokeTradeServiceV2(...) {  // V2
    TradeCallBackServiceV2 service = callBackFactoryV2.getCallBackTradeService(tradeCode);
    return service.invokeTradeService(hasDecryptedMessage, partnerMessageConfig);
}

```

<strong>为什么不直接升级到 V2，还要保留 V1：</strong>
存量机构的回调接口已经上线运行，机构侧的接收方也依赖当前的响应格式。贸然改成 V2 会导致机构侧解析失败。按"新业务用 V2，存量业务保持 V1"的原则，逐步迁移。

<strong>【可能追问】</strong>

- <strong>这种双版本设计会不会导致代码维护困难？</strong>
确实有这个问题，这是历史债务。理想状态是 V2 的 `NotifyResponseResult` 设计为向后兼容 V1（V1 字段是 V2 的子集），这样只需一套 Factory 就够了。当前是可以接受的中间状态，随着机构迁移到 V2 后逐步废弃 V1。

## 十七、导流平台路由类

### Q21. 你们有导流平台的对接，导流和直连有什么区别？编码映射是怎么处理的？

<strong>【标准回答】</strong>

<strong>直连 vs 导流的区别：</strong>


|  | 直连 | 导流 |
|---|---|---|
| 资金归属 | 美团直接和金融机构合作，资金由美团管理 | 美团只做流量入口，资金由导流平台（如拍拍贷、马上消费）管理 |
| 接入复杂度 | 需要金融牌照和资金托管 | 美团不需要牌照，风险低 |
| 报文协议 | 由金融机构定义 | 由导流平台定义（与直连机构有所不同） |
| 编码体系 | 用 partnerCode（机构编码） | 用"导流业务编码"（如 `00012`） |

<strong>编码映射问题（最复杂的地方）：</strong>

美团内部用**导流业务编码**（如 `00012`）标识一个导流平台，但实际查密钥、专线、URL 配置时，用的是**机构编码**（如 `L0001201`）。两套编码体系共存，必须做双向映射：

```java
// GatewayConstants.java - 静态初始化块维护双向映射static {
    // 导流业务编码 → 机构编码（请求时用）
    diversionPlatform.put("00012", "L0001201");
    diversionPlatform.put("00042", "A32597203");
    ...

    // 机构编码 → 导流业务编码（部分场景回调时需要反查）
    diversionPartnerCodeMap.put("A32597203", "00042");
    ...
}

// RequestPartnerRouteFactory 中专门针对导流产品路由// 导流请求：productNo = MT_AIDLOAN_PPD（拍拍贷）// → 对应的 RequestPartnerRouteService 实现做编码转换// → 实际通信用 diversionPlatform.get("L0001203") 查到的机构编码
```

<strong>导流特有的路由逻辑：</strong>

```java
// CommonRequestPartnerRouteServiceImpl.java（通用路由）@Overridepublic Boolean isFit(ProductNo productNo, PartnerCode partnerCode, TradeCode tradeCode) {
    // 导流产品（PPD、MSXF）走专属路由，不走通用路由
    return !ProductNo.MT_AIDLOAN_MSXF.equals(productNo)
        && !ProductNo.MT_AIDLOAN_PPD.equals(productNo);
}

```

<strong>【可能追问】</strong>

- <strong>双向映射为什么用静态 Map 而不是存数据库？</strong>
导流平台编码映射是相对稳定的配置（不像机构密钥需要轮换），且启动时就需要，用静态 Map 更直接，启动后即可用，无 DB 查询开销。缺点是新增导流平台需要改代码，这是可以接受的权衡（相对低频的操作）。

## 十八、Maven 多模块与 Client 独立发布类

### Q22. 你们项目是怎么分模块的？Client 包独立发布有什么好处？

<strong>【标准回答】</strong>

项目分为五个 Maven 模块，分层清晰：

```text
xy-ccp-gateway（父 POM，统一依赖版本）
├── xy-ccp-gateway-core        # 领域模型：枚举、领域对象（无 Spring 依赖）
├── xy-ccp-gateway-client      # 对外 API：Thrift 接口定义、DTO、异常（独立发布）
├── xy-ccp-gateway-thrift-api  # 内部 Thrift API（供 web 层使用）
├── xy-ccp-gateway-service     # 业务逻辑（加解密、报文、路由等核心实现）
├── xy-ccp-gateway-repository  # 数据访问层（MyBatis Mapper）
└── xy-ccp-gateway-web         # 入口层（Thrift Server、Controller、Crane Job）
```

`xy-ccp-gateway-client`<strong> 独立发布的意义：</strong>

```text
上游信贷核心系统只需引入：
<dependency>
    <groupId>com.sankuai.meituan.ccp</groupId>
    <artifactId>xy-ccp-gateway-client</artifactId>
    <version>1.x.x</version></dependency>

Client 包包含：
- TPartnerInvokerRpcService（主调接口）
- TGatewayConfigRpcService（配置管理接口）
- TBusinessRequest / TBusinessResponse（DTO）
- TGateWayRuntimeException / TPartnerBusinessException / TIllegalArgumentException（异常定义）
- 回调接口定义（供下游系统实现，接收机构回调通知）

```

<strong>好处：</strong>

1. **接口与实现解耦**：上游只知道接口，不知道实现细节，网关内部重构不影响调用方
2. **版本管理清晰**：Client 包有独立版本号，接口变更可以发新版本，调用方按需升级
3. **编译安全**：调用方引入 Client 包后，编译期就能发现接口不匹配问题，而不是运行时才报错
4. **最小依赖**：Client 包不引入业务实现依赖，调用方不会因为网关的实现依赖产生冲突

<strong>【可能追问】</strong>

- <strong>Client 包升级了但上游没有及时升级，会出现问题吗？</strong>
Thrift 接口有一定的向后兼容性（新增字段不影响老版本解析）。但如果删除了字段或修改了方法签名，老版本 Client 调用新版本网关时会报序列化错误。所以我们遵守「只新增，不删除，不修改已有字段」的接口演进原则。
- <strong>Maven 多模块中父 POM 有什么作用？</strong>
统一管理所有子模块的依赖版本（`<dependencyManagement>`），避免子模块之间版本不一致导致的 ClassNotFound 或方法签名冲突。同时可以统一编译插件、打包配置等。

## 十九、配置管理 API 开放类

### Q23. 你们网关有没有对外暴露运维管理类接口？怎么设计的？

<strong>【标准回答】</strong>

有，通过 `TGatewayConfigRpcService` 将网关的配置管理能力对外暴露给运营平台。

<strong>背景：</strong>
早期网关的机构配置（URL、密钥、专线、TradeCode 映射等）只能由网关 RD 直接操作数据库修改，每次新机构接入或配置变更都需要网关 RD 介入，效率低。

<strong>设计：</strong>

```java
// TGatewayConfigRpcService（Thrift 接口）interface TGatewayConfigRpcService {
    // 新增/修改机构 URL 配置
    TResult updateGatewayURL(TGatewayURLRequest request);

    // 新增/修改机构密钥配置
    TResult updatePartnerSecret(TPartnerSecretRequest request);

    // 查询专线状态
    List<TPrivateNet> queryPrivateNet(String partnerCode);

    // 手动触发专线状态切换
    TResult switchPrivateNetStatus(String netCode, String status);

    // 触发配置热更新
    TResult refreshConfig();
}

```

<strong>权限控制：</strong>
配置管理接口只对内部运营平台（有独立的 AppKey）开放，通过美团内部的 API 鉴权体系（Token 验证）保证只有授权系统才能调用，不对外暴露。

<strong>好处：</strong>

- 运营配置变更不再需要网关 RD 手动操库，降低操作风险
- 所有配置变更有操作日志，可追溯
- 结合热更新接口，配置变更可以实时生效

<strong>【可能追问】</strong>

- <strong>配置管理 API 和业务 API 放在同一个 Thrift Server 上吗？</strong>
分开的。配置管理接口走独立端口，和业务接口隔离，避免配置操作的流量影响业务接口的响应时间。这也是端口隔离的常见实践。

## 二十、Crane 分布式任务调度类

### Q24. 你们用 Crane 做了哪些事？和 XXL-Job 有什么区别？

<strong>【标准回答】</strong>

Crane 是美团内部的分布式任务调度平台，我们在以下三个场景用到了它：

**场景一：配置热更新**

```java
// 在运营配置变更后，通过 Crane 任务触发内存配置重新加载// Crane 控制台手动触发，或由运营平台 API 触发@Crane("refresh-partner-config")public void refreshConfig() {
    initialConfig.initPartnerConfig(); // 重新从数据库加载所有机构密钥、TradeCode
}

```

**场景二：专线可用性定时维护**

```java
// OrganizationAvailabilityMaintenanceJob.java@Crane("private-net-availability-maintenance")public void privateNetAvailabilityMaintenance(@CraneParam String paramsStr) {
    // 接收 JSON 数组参数：要探活的专线编号列表
    List<String> containsNetCodeList = JsonUtils.toList(paramsStr, String.class);
    netSwitchService.mainPrivateNetAvailability(containsNetCodeList);
}

```

运维人员可以通过 Crane 控制台传入专线编号列表，触发一次性探活，不需要重启服务。

**场景三：专线搬迁工单**

机房迁移、运营商更换等专线搬迁操作，通过 Crane 定时任务按步骤自动执行，替代人工按步骤操作，降低操作失误风险。

<strong>Crane 和 XXL-Job 的对比：</strong>


|  | Crane（美团内部） | XXL-Job（开源） |
|---|---|---|
| 调度方式 | 支持手动触发、定时、事件驱动 | 支持定时（cron）、手动触发 |
| 参数传递 | `@CraneParam` 注解，支持动态参数 | 支持 JobParam |
| 执行策略 | 支持广播（所有实例执行）、分片 | 支持路由策略（第一个、轮询、随机等） |
| 与内部系统集成 | 深度集成美团监控、权限体系 | 需要单独部署调度中心 |

<strong>【可能追问】</strong>

- <strong>Crane 任务执行失败了怎么办？</strong>
Crane 平台有失败告警和重试机制，执行结果可在控制台查看。对于关键的配置更新任务，失败后会报告警，需要人工处理或手动重试。
- <strong>多实例部署时，Crane 任务会在所有实例上都执行吗？</strong>
取决于任务的执行策略。`@Crane` 注解默认是广播模式（所有实例都执行），热更新配置时需要每个实例都更新自己内存中的配置，所以广播是合理的。如果不需要广播，可以配置只在一个实例上执行。

## 二十一、CAT 监控体系类

### Q25. 你们是怎么用 CAT 做监控的？监控体系是怎么设计的？

<strong>【标准回答】</strong>

CAT 是美团开源的 APM（应用性能监控）平台，我们在项目里用了三层监控：

<strong>第一层：Transaction（链路耗时）</strong>

每次外部机构调用，同时开启两个 Transaction：

```java
// MonitorService.java// 业务维度（产品线 × 机构）：看哪个产品的哪个机构耗时长Transaction businessTransaction = Cat.newTransaction(productNo.getNo(), partnerCode.getPartnerCode());

// 机构维度（机构 × 交易类型）：看某个机构的哪种业务耗时长Transaction partnerTransaction = Cat.newTransaction(partnerCode.getPartnerCode(), tradeCode.getTradeCode());

```

两个维度的 Transaction 让我们既能从业务视角（某个产品出问题了）也能从机构视角（某个银行接口变慢了）定位问题。

<strong>第二层：BatchEvent（错误统计）</strong>

四种异常类型独立打点：

```java
// 网络连接异常、解密异常、业务异常、内部异常 各自对应独立的 CatEvent// 这样 CAT 上能看到每种异常类型的发生频率，快速判断是哪个维度的问题
Cat.logBatchEvent(catEvent.getName(), partnerCode + "_" + tradeCode, totalCount, errorCount);

```

<strong>第三层：MetricHelper（流量大盘）</strong>

```java
// 三维标签（产品 × 业务 × 机构），支持多维度下钻分析
MetricHelper.build().name("产品-资方-业务流量大盘")
    .tag("产品号", productNo.getName())
    .tag("业务类型", tradeCode.getTradeName())
    .tag("外部机构", partnerCode.getPartnerName())
    .count(1);

// 新老接口流量占比监控（用于迁移进度跟踪）
MetricHelper.build().name("资方-业务-接口流量大盘")
    .tag("接口类型", isNew ? "newAPI" : "oldAPI")
    .count(1);

```

<strong>排查问题的路径：</strong>

```text
问题现象（用户投诉/告警）
  ↓
CAT Transaction 看整体成功率 → 确定哪个产品/机构出问题
  ↓
看对应的 BatchEvent → 判断是网络层/密钥层/业务层
  ↓
看 CAT 详细日志 → 找具体请求的报文内容（已脱敏）
  ↓
如果是网络问题 → 去 Falcon 看专线监控
  ↓
如果是密钥问题 → 去 KMS 看密钥状态

```

<strong>【可能追问】</strong>

- <strong>CAT Transaction 和 SkyWalking Span 有什么区别？</strong>
本质上都是链路追踪的基本单元，记录一段操作的开始/结束时间和状态。区别是：CAT 的 Transaction 更侧重统计（aggregation，汇总成成功率、平均耗时等指标）；SkyWalking 的 Span 更侧重单次请求的完整链路还原（可以看一次请求里调用了哪些下游服务，每个步骤耗时多少）。
- <strong>为什么要两个维度的 Transaction 而不是一个？</strong>
业务维度的 Transaction（productNo × partnerCode）帮助产品/运营发现"哪个产品出问题了"；机构维度的 Transaction（partnerCode × tradeCode）帮助 RD 发现"哪个银行的哪个接口变慢了"。两个维度的观察角度不同，互补。

## 二十二、日志脱敏与合规类

### Q26. 金融场景下，日志怎么处理用户隐私数据？你们是怎么实现的？

<strong>【标准回答】</strong>

金融场景下，用户数据（手机号、身份证、银行卡号）受《个人信息保护法》和金融监管规定保护，**不允许以明文形式出现在日志里**，否则一旦日志文件泄露，就是严重的数据安全事故。

**我们的方案：强制脱敏工具 **`DesenseUtil`

```java
// PartnerInvokeRpcServiceImpl.java - 接口入口强制脱敏
LOG.info("收到业务请求, 请求参数为:tBusinessRequest={}",
    DesenseUtil.desensitizeJson(JsonUtils.toJsonString(tBusinessRequest)));
// desensitizeJson 会把 JSON 中的敏感字段自动 mask：// {"mobile": "13812345678"} → {"mobile": "138****5678"}// {"idCard": "310101199001011234"} → {"idCard": "310101****1234"}

// PartnerAppService.java - 报文截断（防止超长报文撑爆日志）
LOG.info("报文为:{}", LogPrintUtil.getPrintLogContent(requestMessage));
// getPrintLogContent 对超过阈值长度的报文进行截断，避免影像件等大报文把日志文件写满
```

<strong>规则层面的保障：</strong>

1. 代码 Review 阶段，含用户数据的日志打印必须经过脱敏处理，否则 Review 不通过
2. 静态代码检查（自定义 CheckStyle 规则）检测 `LOG.info` 里包含用户字段名的模式

<strong>脱敏粒度：</strong>

- **手机号**：保留前 3 位和后 4 位，中间 4 位用 `*` 替代（`138****5678`）
- **身份证**：保留前 6 位（省市区）和后 4 位，中间 8 位用 `*` 替代
- **银行卡**：保留后 4 位，前面全部用 `*` 替代
- **姓名**：保留姓（第一个字），后面用 `*` 替代

<strong>【可能追问】</strong>

- <strong>脱敏后如果出了问题，还能查到完整数据吗？</strong>
可以，但需要走审批流程。完整的用户数据存在业务数据库里（加密存储），日志只用来辅助排查流程问题，不用来查业务数据。如果需要查完整数据，通过数据权限申请流程获取。
- <strong>报文截断会不会导致排查问题时看不到完整报文？</strong>
是的，这是一个权衡。我们的策略是对常规报文不截断，只对超过一定长度（如影像件 base64）的报文才截断，并在截断处加注明"[已截断 xxx bytes]"。对于需要查完整报文的情况，可以通过机构侧的接口日志配合查。

## 二十三、系统设计延伸

### Q27. 如果让你重新设计这个网关，你会做哪些改进？

<strong>【标准回答】</strong>

基于我在项目里的实际经验，我会重点改进以下几个方面：

<strong>改进一：本地配置缓存（高优先级）</strong>

当前 `partnerMessageConfigRepository.getByProductNoAndPartnerCode()`、`partnerTradeConfigRepository.query()` 等在每次请求时都查数据库，机构配置几乎不变，是明显的优化点：

```java
// 用 Guava Cache 做本地缓存，过期时间 30 分钟// 配置变更时触发 Crane 任务让所有实例清除缓存
Cache<String, PartnerMessageConfig> configCache = CacheBuilder.newBuilder()
    .expireAfterWrite(30, TimeUnit.MINUTES)
    .maximumSize(1000)
    .recordStats() // 开启 hit/miss 统计，接入 CAT 监控
    .build();

```

<strong>改进二：主备专线自动 Failover（高优先级）</strong>

当前主线故障时会报异常，需要运维手动处理。更好的设计是主线不可用时自动降级到备线：

```java
// 改进后的 NetLineService.selectPrivateNet()if (!masterNets.isEmpty()) {
    return randomSelect(masterNets); // 优先主线
}
// 主线全部不可用时，自动降级到备线（而不是报异常）if (!slaveNets.isEmpty()) {
    LOG.warn("主线全部不可用，自动降级到备线，请及时检查主线状态");
    sendDXAlert("主线故障，已自动切换到备线"); // 同时告警
    return randomSelect(slaveNets);
}
throw new NoAvailablePrivateNetException("主备线路均不可用");

```

**改进三：HTTP 连接池统一管理**

把所有机构的 HTTP 连接池配置化，支持按机构/接口类型设置不同的超时和最大连接数，避免一个慢机构影响其他机构的连接池资源。

**改进四：异步化非核心链路**

大象 IM 通知、CAT 埋点的部分操作、日志写入等，改为异步线程池执行，减少对主链路的阻塞。

<strong>改进五：</strong>`factory 选择器`** O(1) 优化**

把所有 Factory 的 List 匹配改为 Map 查找，代码改动很小，在机构量大时有收益：

```java
// 初始化时构建 Map
Map<EncryptionType, EncryptionService> encryptionServiceMap = encryptionServiceList.stream()
    .collect(Collectors.toMap(EncryptionService::getEncryptionType, Function.identity()));
// 查找 O(1)return encryptionServiceMap.get(encryptionType);

```

<strong>【可能追问】</strong>

- <strong>你说的这些改进，有没有评估过优先级？</strong>
有。本地配置缓存和主备自动 Failover 对可用性影响最大，是 P0 优化；连接池管理和异步化是性能优化，是 P1；Factory O(1) 改进是锦上添花，是 P2。实际工作中会先把 P0 做了，P1 和 P2 放在迭代里逐步推进。

## 二十四、踩坑与线上问题类

### Q28. 你在项目里遇到过什么印象深刻的线上问题？怎么排查的？

<strong>【标准回答（选 1-2 个真实案例讲）】</strong>

<strong>案例一：光大银行回调报文乱码（GBK 编码问题）</strong>

**现象**：新接口上线后，光大银行的回调请求处理失败，日志显示 JSON 解析报错，回调报文内容是乱码。

**排查过程**：

1. 先确认是不是代码逻辑问题 → 在 UAT 测试正常，怀疑是数据问题
2. 打印原始 `callBackMessage` → 发现中文字段是乱码，说明编码在接收阶段就出错了
3. 怀疑编码问题，检查光大银行文档 → 确认光大接口使用 GBK 编码
4. 检查 Spring 的 `@RequestBody` 处理 → 默认按 ISO-8859-1 解析 HTTP Body
5. 定位根因：光大发来 GBK 编码的 HTTP Body，Spring 按 ISO-8859-1 接收，字节被错误解读了

**解决方案**：

```java
// 字节级别重新解释：用 ISO-8859-1 拿回原始字节，再按 UTF-8 重新构造字符串// （因为光大 GBK 和 UTF-8 都兼容 ASCII，中文字节序列是一样的，只是编码解读方式不同）
callBackMessage = new String(
    StringEscapeUtils.unescapeHtml(callBackMessage).getBytes("iso8859-1"),
    StandardCharsets.UTF_8);

```

**教训**：编码问题一定要用生产级别的含中文的真实报文测试，UAT 的测试数据不够完整。

---

**案例二：Mafka 消息积压导致专线被误切回故障状态**

**现象**：某天专线恢复正常后约 20 分钟，用户反馈某银行请求失败，查看专线状态发现又被切回了 OFF。

**排查过程**：

1. 查看 `ZhuanXianSwitchService` 的处理日志 → 发现确实又消费到了一条"专线故障"消息
2. 查看消息的 `time` 字段 → 是 20 分钟前告警的时间，消息积压了 20 分钟才被消费到
3. 根本原因：消费者实例在告警期间因为某种原因处理慢，消息积压，积压的故障消息在专线恢复后才被消费到，触发了二次切换

**解决方案**：加入时间窗口校验

```java
// 消费时校验：告警时间 vs 当前时间 > 5 分钟则丢弃Long diffTime = System.currentTimeMillis() - alertTime.getTime();
if (diffTime > ZhuanXianContant.getFaultDelayTime()) {
    LOG.error("过期消息，不执行切换，故障时间：{}，当前时间：{}", alertTime, new Date());
    return;
}

```

**教训**：使用消息队列处理状态变更类的操作时，要考虑消息的"有效期"，过期的消息可能反而会造成误操作。
