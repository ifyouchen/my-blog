# 项目介绍

## <strong>项目全景介绍：xy-ccp-qualification-evaluate（资质评估服务）</strong>

### **一、项目定位**

<strong>这是一个金融资质评估平台，属于美团金融CCP（消费信贷）体系中的核心服务。它的核心职责是：作为美团与外部金融机构（资方/机构）之间的"桥梁"，将美团的客户资质评估请求转化为各机构要求的报文格式，发送到机构网关，接收机构回调结果，并同步给美团内部系统（机构风险、小贷等）。</strong>

---

### <strong>三、核心业务（四大评估 + 衍生）</strong>

**从 **`EvaluateType`<strong> 枚举可以看出，系统支持 4大核心评估类型 + 2个管理操作：</strong>


`QUALIFICATION_EVALUATE(1, "qualification_evaluate", "客户资质评估"), // 核心：授信`
`PURCHASE_EVALUATE(2, "purchase_evaluate", "购买评估"), // 核心：购买`
`LIMIT_ADJUST(3, "limit_adjust", "额度调整"), // 核心：调额`

`ORG_RELIEVE(6, "org_relieve", "机构清退"), // 管理：机构清退`

#### <strong>3.1 客户资质评估（QUALIFICATION_EVALUATE）</strong>

<strong>最核心的业务。美团客户申请贷款时，需要将客户信息发送给金融机构做授信审批。流程：接收请求 → 组装报文 → 发送机构 → 回调解析 → 更新客户资质状态。</strong>

#### <strong>3.2 购买评估（PURCHASE_EVALUATE）</strong>

<strong>客户实际购买（支用）时，向机构发送购买/支用申请，机构审批通过后才能放款。</strong>

#### <strong>3.3 额度调整（LIMIT_ADJUST）</strong>

<strong>客户已授信后，如果需要调整额度（提额/降额），向机构发起调额申请。</strong>

#### <strong>3.4 客户资质预评估（QUALIFICATION_PRE_EVALUATE）</strong>

<strong>在正式授信之前，先向机构做一次预评估，获取初步授信额度/利率等信息。</strong>

#### **3.5 衍生业务**

- <strong>解除客户资质（</strong>`OrganizationCheckRelieveService`<strong>）：支持按产品号或按机构产品号解除客户资质</strong>
- <strong>放款结果通知（</strong>`LoanNotificationService`** + **`LoanDisbursementMessageFactory`<strong>）：放款成功/失败后通知机构</strong>
- <strong>抵押贷（</strong>`MortgageLoanService`<strong>）：车抵贷的匹配客户、匹配车辆、提交抵押等</strong>
- <strong>资质补登（</strong>`customerQualificationSupplement`<strong>）：融合产品场景下，一个机构产品对应多个C端产品，需要补登授信记录</strong>
- <strong>客群审批维护（</strong>`CustomerGroupAuditDomainService`<strong>）：不同客群精细化管理</strong>

<strong>事件处理工厂（</strong>`EvaluateHandleFactory`<strong>）根据 </strong>`EvaluateType`** + **`EventType`<strong> 双维度路由到具体处理器：</strong>

```text

```

![](assets/image-002.png)[EvaluateHandleFactory.java](http://EvaluateHandleFactory.java)

Apply

`public void evaluateEventHandle(EventDriveModel eventDriveModel) {`
`switch (eventDriveModel.getEventType()) {`
`case EVALUATE:`
`evaluateHandlerMap.get(eventDriveModel.getEvaluateType()).organizationEvaluate(eventDriveModel);`
`return;`
`case COMPENSATE:`
`compensateEvaluateHandlerMap.get(eventDriveModel.getEvaluateType()).compensateEvaluate(eventDriveModel);`
`return;`
`case CALLBACK:`
`callbackHandlerMap.get(eventDriveModel.getEvaluateType()).evaluateCallBack(eventDriveModel);`

**展开**

<strong>模板方法模式体现在三个抽象基类中：</strong>


| **基类** | **职责** | **核心流程** |
|---|---|---|
| `**AbstractEvaluateHandler**` | **发起评估** | **checkContext → parseRequest → 限流 → executeEvaluate → completeEvaluateRequest** |
| `**AbstractCallbackHandler**` | **处理回调** | **parseEvaluateResult → checkContext → processEvaluateResult → completeEvaluate** |
| `**AbstractCompensateEvaluateHandler**` | **补偿处理** | **recordOverTime → checkOrganizationRecord → reEvaluateHandle / callbackHandle** |

<strong>每种评估类型（授信/购买/调额/预评估）都有对应的 Handler 子类实现。</strong>

#### <strong>4.2 策略模式 + 工厂模式（机构路由）</strong>

**不同金融机构（平安、光大、天津银行、中银等）的报文格式和接口协议各不相同。系统通过 **`OrganizationHandlerFactory`<strong> 实现机构级别的策略路由：</strong>

```text

```

![](assets/image-002.png)[OrganizationHandlerFactory.java](http://OrganizationHandlerFactory.java)

`private StringValuedEnum getOrganizationRouteConfig(EvaluateType evaluateType, OrganizationConfig organizationConfig) {`
`switch (evaluateType) {`
`case LIMIT_ADJUST:`
`return organizationConfig.getLimitAdjustEvaluateRoute();`
`case QUALIFICATION_EVALUATE:`
`return organizationConfig.getCustomerQualificationEvaluateRoute();`
`case PURCHASE_EVALUATE:`
`return organizationConfig.getPurchaseQualificationEvaluateRoute();`
`case QUALIFICATION_PRE_EVALUATE:`
`return organizationConfig.getCustomerQualificationPreEvaluateRoute();`
`}`
`}`

**展开**

**每个机构+评估类型组合对应一个具体的 **`OrganizationHandler`<strong> 实现，如：</strong>

- `**PingAnAidLoanQualificationEvaluateOrganizationHandler`（平安助贷授信）**
- `**GUANGDAUnionLoanPurchaseEvaluateHandler`（光大联合贷购买）**
- `**TJBankPurchaseEvaluateHandler`（天津银行购买）**
- `**ZhongYinBankLimitAdjustEvaluateHandler`（中银调额）**

#### <strong>4.3 报文模板引擎（可配置化报文）</strong>

**这是项目最精巧的设计之一。通过 **`RequestBuildFactory`** + **`VariableImplement`** + **`VariableConfig`<strong>，实现了报文模板的数据库可配置化：</strong>

1. **从数据库加载 **`RequestConfig`<strong>（报文模板 JSON + 变量配置列表）</strong>
2. <strong>按优先级解析变量（</strong>`VariableImplement`** 接口有 80+ 实现，如 **`CustomerBaseVariable`<strong>、</strong>`CreditLimitVariable`<strong> 等）</strong>
3. **通过 **`CustomExpresionHelper`** 做占位符替换，生成最终报文**

<strong>这意味着新增机构或修改报文格式时，不需要改代码，只需在数据库中配置新的模板和变量即可。</strong>

#### <strong>4.4 放款报文构建（策略+工厂）</strong>

`**LoanDisbursementMessageFactory` 根据放款消息类型路由到不同的报文构建器：**

- `**DefaultLoanDisbursementMessageServiceB/C`（默认报文）**
- `**Special2LoanDisbursementMessageServiceB`（机构产品链路报文）**
- `**Special5LoanDisbursementMessageServiceC`（费用类特殊报文）**
- `**SimplifiedLoanDisbursementMessageServiceB/C`（精简报文）**

## <strong>三、面试怎么讲（沟通技巧）</strong>

### **核心原则**

1. <strong>先说What（是什么），再说How（怎么做），最后说Why（为什么）</strong>
2. **用类比代替术语**：不要说"我们用了策略模式"，说"每家银行的报文格式不一样，我们让每家银行各写一个处理器，系统根据配置自动选择用哪个"
3. **说数据不说感觉**：不要说"性能提升了很多"，说"P99从30s降到200ms"
4. **说决策不说实现**：面试官不关心代码怎么写的，关心你为什么这么做、还有没有更好的方案

### **常见面试问题及回答模板**

---

**Q：简单介绍一下你这个项目**

> 这个系统是美团金融对接合作银行的统一评估平台。用户在美团借钱时，背后会调银行做授信审批——用户能不能借、给多少额度、利率多少——这些审批请求都是通过我们系统转发到银行的。
> 我们对接了平安、光大、中银等10多家银行，每天处理几十万笔评估请求。技术上最大的特点是：第一，银行接口又慢又不稳定，所以我们用了MQ做异步，请求来了先接住、异步处理，通过补偿机制保证不丢；第二，每家银行的报文格式都不一样，所以我们设计了一套可配置的策略路由框架，新增银行基本不需要改核心代码。

---

<strong>Q：为什么用异步架构？同步不行吗？</strong>

> 核心原因是银行审批太慢了。快的几百毫秒，慢的要几十秒，而且不稳定。如果同步等待，高峰期线程池会被耗尽，用户也等不了这么久。
> 所以我们设计成：请求进来先存库、发MQ、立刻返回成功。后台消费者拿到消息再去调银行。银行审批完会回调我们，我们更新结果再通知下游。
> 当然异步就有丢失和失败的风险，所以我们做了三层保障：一是MQ消费幂等，通过唯一索引保证不重复处理；二是补偿定时任务，扫描超时未完成的记录主动查询银行；三是乐观锁，防止并发回调导致数据不一致。

---

<strong>Q：10多家银行怎么管理的？新增一家银行要多久？</strong>

> 我们用策略模式，每种评估类型都有一套Handler继承体系。系统启动时自动注册所有Handler，运行时根据"机构编码+机构产品号+评估类型"这三个维度查配置，自动匹配到对应的Handler。
> 大部分银行走默认逻辑，只需要在数据库配一下报文模板和变量就行，不用改代码。只有报文格式特别特殊的银行才需要写定制化Handler，一般1-2天能搞定。

---

<strong>Q：数据量大了怎么处理？</strong>

> 我们用Zebra中间件做分库分表，两个分片键：产品号和客户号。产品号做主分片，把月付和微贷两个业务线的数据物理隔离；客户号做辅助分片，同一客户的数据落在同一张表上，查询效率高。
> 补偿任务要做全表扫描，我们通过ForceShardReader遍历所有物理表，用滑动窗口分批次处理，避免一次性加载太多数据。

---

<strong>Q：你觉得这个项目有什么可以改进的？</strong>

> （说一个真实的小问题，不要说太大的，显得你在思考但不是在否定团队）
> 一个我觉得可以优化的是补偿任务的Handler匹配。现在是用线性遍历List去匹配，机构多了之后效率不高，可以在初始化时预构建一个Map索引，用组合键直接O(1)查找。
> 另一个是分片参数清理的风险。Zebra的ThreadLocal需要手动清理，如果某个异常分支漏了clear，线程池复用时会导致分片参数串用。可以考虑封装一个模板方法，在finally里自动清理。

---

## **四、面试官可能追问的深水区**

这些问题是面试官用来判断你是"真做过"还是"背的"：

1. <strong>“补偿任务会不会和正常回调冲突？”</strong> → 乐观锁，只有一个能成功
2. <strong>“MQ消息发出去但消费前服务重启了怎么办？”</strong> → Mafka有消费位点持久化，重启后继续消费
3. <strong>“如果银行一直不回调也不返回结果怎么办？”</strong> → 超时机制，默认12小时后标记失败，不再补偿
4. <strong>“月付和微贷为什么要隔离？”</strong> → 两个业务线的流量特征、数据量、SLA要求不同，隔离后互不影响
5. <strong>“额度校验的逻辑是什么？为什么授信有而支用没有？”</strong> → 授信通过后要展示额度给用户，所以必须校验；支用是实际放款，额度在授信阶段已经确定了
6. <strong>“你们怎么测试的？怎么模拟银行回调？”</strong> → 有TestController，可以手动发补偿请求；也有测试环境的Mock网关

## **一、项目全景与架构理解**

**Q1：简单介绍一下你这个项目**

<strong>面试官问这句话的时候，他其实是在问三件事：这是个什么系统、解决什么问题、你在里面干什么。你用一分钟说清楚就行。</strong>

> <strong>这个系统是美团金融对接合作银行的统一评估平台。</strong>
> <strong>具体来说，用户在美团借钱的时候，背后需要银行来审批——这个用户能不能借、给多少额度、利率多少——这些审批请求都是通过我们系统转发给银行的，银行批完之后把结果回传给我们，我们再更新用户的额度状态，通知下游业务系统放款。</strong>
> <strong>目前我们对接了平安、光大、中银、江苏银行等10多家机构，每天处理几十万笔评估请求。我在里面主要负责授信和支用两条核心链路的开发和维护。</strong>

---

<strong>Q2：整体架构是怎样的？模块怎么划分的？</strong>

<strong>面试官想听的是你的分层思路，不是你列出所有包名。</strong>

> <strong>整个系统是标准的Maven多模块分层架构，一共5个模块，自底向上依赖：</strong>
> <strong>最底层是 thrift-api，纯接口定义，定义了对外的RPC服务契约，独立版本发布给调用方依赖。</strong>
> <strong>第二层是 core，就是领域模型层，所有的业务实体、枚举、Repository接口定义都在这。你可以理解为整个系统的"通用语言"，比如客户资质、评估记录、机构配置这些核心概念都在这里定义。</strong>
> <strong>第三层是 server，最核心的业务逻辑都在这，包括领域服务、应用服务、请求报文构建引擎、MQ的生产和消费、补偿机制。</strong>
> <strong>第四层是 repository，数据访问层，通过MyBatis实现，配合Zebra中间件做分库分表。</strong>
> <strong>最上层是 adaptor，对外暴露层，包含Thrift服务实现、HTTP接口、远程调用实现、定时任务、限流配置。Spring Boot启动入口也在这。</strong>
> <strong>这样分层的好处是：core层非常稳定，几乎不变；新增业务逻辑主要改server层；对接新的外部系统只改adaptor层。各层职责清晰，互相不影响。</strong>

---

<strong>Q3：授信和支用分别是什么意思？有什么区别？</strong>

<strong>这个问题如果答不清楚，后面所有技术问题都没意义。先用大白话解释概念。</strong>

> <strong>授信就是银行审批"这个用户能不能借"，支用就是用户实际发起借款时银行审批"这笔款放不放"。</strong>
> <strong>打个比方：授信就像银行给你办了一张信用卡，告诉你额度5万；支用就是你拿着这张卡去刷了一笔3千的消费。</strong>
> <strong>在我们系统里，两者的技术差异主要有三点：</strong>
> <strong>第一，结果处理不同。授信通过后要创建客户资质记录，里面包含额度、利率、有效期，前端靠这个展示"可借额度"；支用通过后只更新评估记录状态，通知下游放款。</strong>
> <strong>第二，额度校验不同。授信返回的额度和利率要做校验，比如银行给的额度和我们申请的额度差异不能超过1000块，利率差异要在合理范围内；支用不处理额度和利率。</strong>
> <strong>第三，业务频率不同。一个用户可能只授信一次，但可以多次支用。</strong>
> <strong>不过从技术架构上看，它们共享同一套框架——MQ异步消费、策略模式选机构Handler、补偿定时任务、回调处理模板，只是具体的Handler实现和结果处理器不同。</strong>

## **二、核心业务流程**

<strong>Q4：授信的完整流程是怎样的？</strong>

<strong>面试官想听你把一个请求从头讲到尾，不是让你背代码，而是看你能不能把复杂流程讲清楚。</strong>

> <strong>整个流程分三个阶段：</strong>
> <strong>第一阶段：请求受理（同步，毫秒级返回）</strong>
> <strong>外部系统调我们的Thrift接口发起授信请求。我们做参数校验（客户号、流水号、机构、产品号这些必填项），校验通过后在同一个事务里做两件事：一是创建评估记录，状态是"初始化"；二是创建客户资质记录，状态是"评估中"。然后构建一个事件消息发到MQ，立刻返回成功给调用方。</strong>
> <strong>为什么不等银行结果再返回？因为银行审批太慢了，快的几百毫秒，慢的要几十秒，同步等不起。</strong>
> <strong>第二阶段：机构评估（异步消费MQ）</strong>
> <strong>后台消费者拿到消息后，走一个标准的模板流程：</strong>
> - **先检查评估记录状态，确保是"初始化"或"处理中"才能继续**
> - **然后构建发往银行的交易报文——这里有个我后面会详细讲的请求构建引擎**
> - **做限流检查，按机构+产品维度限流，防止打满银行接口**
> - **根据机构配置选择对应的Handler，通过网关调用银行**
> - **调完之后把评估记录状态改成"机构处理中"，等银行回调**
> <strong>第三阶段：回调处理（银行主动回调）</strong>
> <strong>银行审批完会回调我们。我们解析银行返回的结果，校验记录状态是"机构处理中"才能处理，然后根据结果走不同分支：</strong>
> - **通过：校验额度和利率，更新客户资质（状态改为成功），存入额度和利率**
> - **拒绝：更新客户资质（状态改为失败），记录拒绝原因**
> - **处理中：什么都不做，等下次回调或补偿查询**
> <strong>处理完之后发MQ通知下游系统，如果是小贷的请求还会额外通知小贷系统。</strong>

---

<strong>Q5：请求报文是怎么构建的？每家银行格式都不一样吧？</strong>

<strong>这是这个系统最有技术含量的部分之一，要讲清楚"可配置化"这个点。</strong>

> <strong>是的，每家银行的报文格式都不一样。如果每接一家银行就硬编码一套报文构建逻辑，那代码量会爆炸，也不好维护。</strong>
> <strong>所以我们设计了一套可配置的请求构建引擎。核心思路是：把报文的构建规则放到数据库里配置，代码只负责执行。</strong>
> <strong>具体来说，数据库里有两类配置：</strong>
> - **变量配置：定义报文里每个字段从哪取值。比如"客户姓名"从客户服务查，"身份证号"从OCR服务查，"申请金额"从请求参数取**
> - **请求模板：一个JSON字符串，用 **`${变量名}`** 占位，渲染后就是最终的交易报文**
> <strong>变量解析支持三种类型：基础变量直接查服务取值；路由变量用Aviator表达式引擎计算（比如做一些字符串拼接、类型转换）；自定义变量走专门的Java实现（89个变量实现类）。</strong>
> <strong>变量还有优先级，高优先级的变量会覆盖低优先级的。还支持两种容错模式：强依赖的变量取不到就报错，弱依赖的取不到就用默认值。</strong>
> <strong>好处是什么呢？大部分银行接入只需要在数据库配一下变量和模板就行了，不用改代码、不用发版。只有极少数报文特别复杂的银行才需要写定制化的Handler。</strong>

---

## **三、设计模式与架构决策**

<strong>Q6：系统用了哪些设计模式？为什么这么设计？</strong>

<strong>不要列课本上的定义，要结合项目场景讲。</strong>

> <strong>最核心的是策略模式加工厂模式，用来解决多机构差异化处理的问题。</strong>
> <strong>问题是这样的：10多家银行，每家的报文格式、结果解析、状态码映射都不一样。如果全写if-else，代码根本没法维护。</strong>
> <strong>我们的方案是：每种评估类型（授信、支用、调额、预评估）都定义一套Handler继承体系，每家银行实现自己的Handler。系统启动时Spring自动注入所有Handler到一个List里，运行时工厂根据"评估类型+机构配置的路由标识"匹配对应的Handler。</strong>
> <strong>新增银行接入，只需要写一个Handler实现类，配一下路由标识，核心流程代码一行不改。这就是开闭原则的实际应用。</strong>
> <strong>第二个是模板方法模式。评估、回调、补偿这三个流程都有固定的步骤，比如评估就是"检查状态→构建报文→调机构→更新状态"。我们把这个流程定义在抽象基类里，子类只实现具体步骤。这样保证所有机构走的流程是一致的，不会出现有的银行漏了状态更新这种问题。</strong>
> <strong>第三个是工厂模式，用在消息生产者选择上。月付和微贷的消息要发到不同的MQ Topic，通过工厂根据产品号自动选择对应的Producer，调用方不需要关心消息发到哪里。</strong>

---

<strong>Q7：为什么要用异步架构？同步调用有什么问题？</strong>

<strong>面试官在考察你对架构取舍的思考。</strong>

> <strong>核心原因有三个：</strong>
> <strong>第一是外部依赖不可控。 银行的接口响应时间从几百毫秒到几十秒不等，高峰期还可能超时。如果同步等待，线程池很容易被打满，一个慢请求会拖垮整个服务。</strong>
> <strong>第二是业务场景允许异步。 授信审批本身就是一个异步过程，用户不需要站在页面等银行批完。我们只需要告诉调用方"请求已受理"，银行批完再通过回调通知下游就行。</strong>
> <strong>第三是可以天然削峰。 如果某个时刻进来大量授信请求，MQ可以缓冲住，后台消费者按自己的节奏处理，不会把银行接口打爆。</strong>
> <strong>当然异步也有代价，就是复杂度上升了——你得处理消息丢失、重复消费、超时补偿这些问题。所以我们又做了三层保障来兜底，后面可以细讲。</strong>

---

## **四、可靠性与一致性**

<strong>Q8：MQ消息消费怎么保证幂等的？</strong>

<strong>这个问题金融系统必问，要用最简单的话讲清楚。</strong>

> <strong>用的是最经典也最可靠的方案：数据库唯一索引。</strong>
> <strong>每条MQ消息消费之前，我们先用"业务流水号+客户号+机构+产品+评估类型"这几个字段拼成一个唯一键，往MqConsumeRecord表里插一条记录。这个表的唯一键字段建了唯一索引。</strong>
> <strong>如果是重复消息，唯一索引会冲突，抛出DuplicateKeyException。我们捕获这个异常，判定为重复消费，直接返回成功，不做任何业务处理。</strong>
> <strong>为什么不用Redis做幂等？因为这条记录本身就有用——它记录了消息的消费状态（初始化、成功、失败），后面的补偿任务要靠这张表来判断哪些消息需要重新处理。</strong>

---

<strong>Q9：如果MQ消费失败了，或者银行一直没有回调，怎么办？</strong>

> <strong>靠补偿机制兜底。我们有两层补偿：</strong>
> <strong>第一层是消息级补偿。 有一个定时任务专门扫描MqConsumeRecord表里状态为"初始化"或"失败"的记录，重新投递到处理流程中。</strong>
> <strong>第二层是业务级补偿。 这个更关键。比如评估请求已经成功发给银行了，MQ消费也成功了，但银行就是一直不回调。这种情况下消息级补偿是没用的，因为记录状态已经是"机构处理中"了。</strong>
> <strong>所以我们有一个业务补偿定时任务，它的逻辑是：</strong>
> - <strong>用滑动窗口扫描最近24小时内的所有评估记录（要遍历所有分库分表的物理表）</strong>
> - **对每条"机构处理中"的记录，主动调银行的查询接口去问结果**
> - **如果银行说还在处理中，就跳过下次再查**
> - **如果银行说没找到这条记录，就重新发起一次评估**
> - **如果银行返回了终态结果（通过或拒绝），就走正常的回调流程处理**
> <strong>还有个超时保护：默认12小时还没出结果就标记失败不再补偿了。这个超时时间支持按"机构+产品+评估类型"配置不同的值，因为不同银行的审批效率确实不一样。</strong>

---

<strong>Q10：补偿查询和银行回调会不会同时到达，产生并发问题？</strong>

<strong>这个问题能区分你是真做过还是背八股文的。</strong>

> <strong>确实有可能同时到达。我们用了两层保障：</strong>
> <strong>第一层是状态机校验。回调处理前会检查评估记录的状态必须是"机构处理中"，如果是终态（已完成）就直接跳过。这个检查可以挡住大部分并发场景。</strong>
> <strong>第二层是乐观锁。更新评估记录的SQL里带了WHERE条件——</strong>`WHERE evaluate_status = 当前状态`<strong>。如果补偿查询先更新成功了，回调再更新时状态已经变了，影响行数为0，会抛出RecordNotFoundException。这个异常被捕获后不会往外抛，等下一次补偿来兜底。</strong>
> <strong>所以即使并发到达，也只有一个能成功，另一个会失败后由补偿机制兜底，最终结果是一致的。</strong>

---

## **五、分库分表与数据隔离**

<strong>Q11：数据库是怎么分库分表的？</strong>

> <strong>用的是美团的Zebra中间件，双分片键策略：</strong>
> <strong>主分片键是产品号，把月付和微贷两个业务线的数据打到不同的物理库上。这两个业务线的流量特征、数据量、SLA要求都不一样，物理隔离后互不影响。</strong>
> <strong>辅助分片键是客户号，同一产品下按客户分表。好处是同一客户的所有数据都在同一张表上，查询的时候不需要跨表。</strong>
> <strong>代码层面通过Zebra的ThreadLocal API设置分片参数，执行完查询后必须在finally里清理，不然线程池复用的时候分片参数会串。补偿任务要做全表扫描，就用ForceShardReader强制指定物理表，一个一个遍历。</strong>

---

## **六、可扩展性与机构接入**

<strong>Q12：新接入一家银行需要做什么？大概要多久？</strong>

<strong>这个问题面试官想听的是你的系统的扩展性有多好。</strong>

> <strong>分两种情况：</strong>
> <strong>大部分银行，只需要配数据库，不用写代码。 在机构配置表里插一条记录，配上报文模板和变量，指定一下走哪个路由标识（大部分走默认的DEFAULT就行），上线就完事了。快的话半天能搞定。</strong>
> <strong>少数银行，需要写定制化代码。 如果这家银行的报文格式特别复杂，或者结果解析逻辑跟别人不一样，就要写一个专属的Handler。比如平安银行的状态码用的是"SUCCESS/FAILURE/PROCESSING"，而其他银行用的是"A/R/U/W"，这种就需要定制解析逻辑。不过因为有模板方法兜底，我们只需要重写差异化的部分（比如parseEvaluateResult），其他流程框架已经帮我们处理好了。一般1到2天。</strong>
> <strong>整体来说，得益于策略路由+可配置报文引擎这两层设计，机构接入的成本已经压得很低了。</strong>

---

## **七、限流与稳定性**

<strong>Q13：系统怎么做限流的？</strong>

> <strong>用的是美团的Rhino限流框架。</strong>
> <strong>限流维度是机构编码+机构产品号。思路是：每个银行的接口承受能力不同，不能无限制地往一家银行发请求。</strong>
> <strong>限流放在评估流程的"构建报文"之后、“调银行"之前。如果拿到令牌就继续调银行；拿不到令牌就直接返回，不调了。这时候评估记录的状态还是"处理中”，等后面补偿任务再来重试。</strong>
> <strong>授信评估做了限流，其他评估类型默认不限。还有一个细节是限流之后我们会重新检查一次记录状态，因为从限流等待到真正调银行之间可能过了一段时间，状态可能已经被别人改了。</strong>

---

## **八、融合与产品标准化**

<strong>Q14：什么是"融合"？三元组是什么意思？</strong>

> <strong>这个是美团金融内部的产品架构演进。</strong>
> <strong>融合之前，每条产品线（生活费、生意贷、月付）各自维护一套产品号，机构侧也要按产品号分别配置。产品一多管理就很混乱。</strong>
> <strong>融合之后，用三元组统一标识一个产品概念：贷款类型、额度类型、系统运作模式。你可以理解为这是产品的"身份证"，不管你是生活费还是生意贷，只要这三个维度一样，就是同一个产品概念。</strong>
> <strong>对我们系统的影响是：查询接口要支持双查——既要查融合前按产品号的数据，也要查融合后按三元组的数据，取时间最新的返回给调用方。这样保证了新老数据的兼容。</strong>

---

## **九、放款结果通知**

<strong>Q15：放款结果通知是怎么实现的？</strong>

> <strong>放款成功后需要通知银行，告诉他们"这笔钱已经到用户账上了"。</strong>
> <strong>流程是：下游系统调我们的Thrift接口，我们根据放款结果（成功/失败）和贷款类型（生活费/生意贷）选择对应的报文构建器，生成交易报文后通过网关发给银行。</strong>
> <strong>生活费和生意贷的报文格式不一样，通过工厂模式自动选择。通知失败的话会把通知记录存到数据库，有一个专门的补偿定时任务来重试。</strong>

## **十、系统改进思考**

<strong>Q16：你觉得这个系统有什么可以改进的地方？</strong>

<strong>这种问题不要说大方向上的否定，要说具体的、可操作的小优化，显得你有思考但不是在挑刺。</strong>

> <strong>一个是Handler匹配效率。现在工厂是遍历List去匹配Handler，机构多了之后是O(n)的。可以在启动时预构建一个Map索引，用"评估类型+路由标识"做key，查找变成O(1)。</strong>
> <strong>另一个是分片参数清理的风险。Zebra的ThreadLocal必须手动clear，代码里散落着很多调用点，如果某个异常分支漏了clear，线程池复用的时候分片参数就会串。可以考虑封装一个模板方法，在finally里自动清理，减少人为遗漏的可能。</strong>
> <strong>还有一个是补偿任务可以更智能。现在的超时时间是写死的默认12小时，但不同银行的审批效率差异很大，快的几分钟，慢的要几个小时。可以加一个动态调整机制，根据历史的平均审批时间自动调整超时阈值，避免对快的银行等太久、对慢的银行等不够。</strong>

## **十一、高频追问**

<strong>Q17：月付和微贷为什么要隔离？不隔离会怎样？</strong>

> <strong>两个原因。第一是数据安全，两个业务线的客户数据和额度数据不应该混在一起，物理隔离更安全。第二是故障隔离，如果月付出问题需要排查或者做数据迁移，不能影响微贷业务。MQ层面也做了Topic隔离，消费速度、消费逻辑都可以独立控制。</strong>

---

<strong>Q18：资质解除是怎么做的？什么场景会触发？</strong>

> <strong>主要有两个场景：一是客户主动关闭账户，二是风控系统判定客户有风险需要清退。</strong>
> <strong>流程是：查询该客户在所有资方的资质记录，逐个检查机构配置里是否支持解除（不是所有银行都支持），支持的就把资质状态改成"已解除"，然后通知下游关户。</strong>
> <strong>有个细节是：不是所有机构都支持关户解除，机构配置里有两个开关——</strong>`supportRelieve`**和**`supportCloseOrgLimit`<strong>，只有两个都打开的才会执行解除操作。</strong>

---

<strong>Q19：预评估和正式评估有什么区别？</strong>

> <strong>预评估是正式评估之前的一个轻量级校验。你可以理解为"预先问一下银行这个客户大概行不行"。</strong>
> <strong>区别在于：预评估的结果有有效期，过期就失效了；预评估不会创建正式的客户资质记录，只是一个临时的预审结果。正式评估通过后会创建CustomerQualification，里面包含额度、利率，前端靠这个展示可借额度。</strong>
> <strong>技术上它们走的框架是一样的，只是评估类型不同，对应不同的Handler和结果处理器。</strong>

## **第二部分：面试官连环问深度探析**

<strong>面试官的追问一般沿着 “架构选型 → 细节实现 → 边界场景 → 深度原理” 这条线逐步深入。</strong>

---

### :link: **第一条连环问：事件驱动架构**

#### <strong>Q1：为什么用 MQ 自产自消，不用 Spring Event？</strong>

**考察点：分布式消息 vs 进程内事件的理解深度**

<strong>回答话术：</strong>

> <strong>主要有三个原因。第一是可靠性，Spring Event 是 JVM 内存事件，服务重启或宕机后未处理的事件全部丢失，但一笔评估请求代表的是向金融机构发起的授信交易，丢了就是资损。MQ 消息持久化在 Broker，即使服务全挂也不会丢。</strong>
> <strong>第二是分布式支持，我们是多实例部署，Spring Event 只在当前 JVM 内传播，而 MQ 天然支持跨实例消费和负载均衡。</strong>
> <strong>第三是补偿重放，我们的补偿机制需要"重新构造一个事件投递到 MQ"，Spring Event 没有持久化和重放能力，除非自己造一套，那就是重新发明 MQ 了。</strong>

#### <strong>Q2（追问）：MQ 消费失败了怎么办？会不会消息丢了？</strong>

**考察点：消息可靠性保障方案**

<strong>回答话术：</strong>

> <strong>不会丢。我们有三层保障。</strong>
> **第一层是消费端幂等控制。每次消费前先往 **`mq_consume_record`** 表插入一条记录，用 **`eventUniqueKey`<strong> 做唯一索引，如果插入失败说明是重复消费，直接返回成功。</strong>
> <strong>第二层是发送端重试。</strong>`AbstractMafkaProducer`<strong> 内置了 guava-retryer，发送失败会重试 3 次，间隔 10 毫秒。</strong>
> <strong>第三层是定时补偿任务。</strong>`CustomerQualificationEvaluateTask`<strong> 会扫描处理中超时的评估记录，重新构造 EVALUATE 事件投递到 MQ，或者主动查询机构结果走回调流程。</strong>

#### <strong>Q3（追问）：补偿任务扫描超时记录，怎么定义"超时"？不同机构的处理速度不同怎么办？</strong>

**考察点：灵活的超时配置设计**

<strong>回答话术：</strong>

> <strong>我们设计了一个六级优先级的超时配置，通过 Lion 配置中心动态管理：</strong>
> **最高优先级：机构_产品_评估类型（如：平安_A01_授信 = 300s） 次高优先级：机构_产品 第三优先级：机构 第四优先级：产品 第五优先级：评估类型 兜底默认：12 小时**
> <strong>查询时从高到低匹配，第一个命中的就用。这样快的机构可以配短超时快速失败，慢的机构可以配长超时避免误判。</strong>

---

### :link: **第二条连环问：报文模板引擎**

#### <strong>Q4：报文模板引擎是怎么工作的？给我举个具体例子。</strong>

**考察点：对实际代码实现的掌握程度**

<strong>回答话术：</strong>

> <strong>整个流程分三步。</strong>
> **第一步，从数据库加载 **`RequestConfig`<strong>，它包含一个 JSON 格式的报文模板（requestModel）和一组变量配置（variables）。变量配置定义了变量名、变量类型、优先级、解析模式等。</strong>
> <strong>第二步，</strong>`RequestBuildFactory`** 按优先级从高到低分组变量，然后按变量类型（VariableType）路由到不同的解析器。比如 **`ROUTE`** 类型走 **`RouteVariableParseHandler`<strong> 从数据库查，</strong>`CUSTOM`** 类型走 **`VariableImplementFactory`<strong> 获取具体实现类解析。</strong>
> <strong>第三步，所有变量解析完后，</strong>`CustomExpresionHelper`** 用类似 **`${CREDIT_LIMIT}`<strong> 的占位符替换，生成最终报文 JSON。</strong>
> **举个具体例子，平安助贷的授信报文里有一个 **`${CREDIT_LIMIT}`** 字段，它对应的变量实现类是 **`CreditLimitVariable`<strong>，会根据客户类型、风险等级等因素计算出授信额度，然后替换到模板中。</strong>

#### <strong>Q5（追问）：变量之间有依赖关系怎么办？比如变量 B 的值依赖变量 A 先解析完？</strong>

**考察点：变量解析顺序设计**

<strong>回答话术：</strong>

> **通过 **`priority`** 字段解决。每个 **`VariableConfig`<strong> 有一个优先级（priority），数值越高越先解析。</strong>`RequestBuildFactory`<strong> 用 TreeMap 按优先级倒序排列，先解析高优先级变量，低优先级变量可以引用已解析的结果。</strong>
> **另外 **`RequestConfig`** 还有一个 **`dependencyVariable`<strong> 字段，声明哪些变量是最终报文依赖的。解析完后只把这些依赖变量的值注入到表达式上下文中做占位符替换，避免无关变量污染报文。</strong>

#### <strong>Q6（追问）：新增一个机构，要改代码吗？</strong>

**考察点：可扩展性设计**

<strong>回答话术：</strong>

> <strong>大部分情况不需要改代码。只需要两步操作：</strong>
> 1. **在数据库的 **`organization_config`** 表配置机构编码、机构产品号、以及对应的评估路由枚举**
> 2. **在 **`request_config`** 和 **`variable_config`** 表配置该机构的报文模板和变量**
> **只有在机构有定制化的回调解析逻辑或评估结果处理逻辑时，才需要新增一个 Handler 实现类继承 **`AbstractOrganizationHandler`<strong>。但报文组装和发送流程完全复用。</strong>

---

### :link: **第三条连环问：回调路由与数据一致性**

#### <strong>Q7：机构回调时，如果缺少关键字段怎么定位到正确的评估记录？</strong>

**考察点：复杂边界场景的处理能力**

<strong>回答话术：</strong>

> <strong>这是实际生产中经常遇到的问题。我们设计了两级回退策略。</strong>
> <strong>第一级：正常路径。通过 serialNo + customerNo + fundPartyCode + productNo 精确匹配评估记录。</strong>
> **第二级：关键字段缺失时的回退路径。当 fundPartyCode 或 customerNo 为空时，我们通过 **`biz_serial_no_relation_record`<strong> 表查出该流水号关联的所有评估记录。如果只有一条，直接匹配。如果有多条，说明是融合场景下多个 C 端产品的记录，我们会主动调用机构的查询接口获取最新结果，再走回调流程。</strong>
> <strong>最后还有一个兜底逻辑：如果本地完全找不到记录，会判断机构风险侧是否有非终态数据，如果有就转发给机构风险处理，没有则抛异常拒绝。</strong>

#### <strong>Q8（追问）：你刚才提到融合场景下一个流水号对应多条记录，为什么不直接用唯一索引保证一对一？</strong>

**考察点：业务背景的理解**

<strong>回答话术：</strong>

> <strong>因为融合前一个机构产品只对应一个 C 端产品号，融合后一个机构产品对应多个 C 端产品号。在融合过渡期，我们做了一个补登机制——融合后新产生的 C 端产品号的评估记录会补写到数据库。这就导致同一笔业务流水在同一个机构产品下可能存在两条记录，但它们的评估结果是一致的。所以我们不能简单用唯一索引，而是用关联表 + 多级查询来处理。</strong>

#### <strong>Q9（追问）：本系统和机构风险系统的数据不一致怎么办？</strong>

**考察点：多数据源一致性问题**

<strong>回答话术：</strong>

> **我们设计了一个多级回退查询策略。查询授信结果时，先查本系统的 **`customer_qualification_evaluation_record`** 表，如果查到终态结果直接返回。如果没查到（可能是评估记录还没同步过来），就兜底调机构风险系统的接口查询。这个接口也分了两套——月付走旧的 **`organizationRiskQueryService`<strong>，微贷走新的 </strong>`orgRiskQueryService`<strong>，用三元组查询而不是产品号查询。</strong>

### :link: **第四条连环问：并发与幂等**

#### <strong>Q10：如果同一笔请求被重复发起（比如网络超时客户端重试），怎么保证不重复评估？</strong>

**考察点：幂等设计**

<strong>回答话术：</strong>

> <strong>我们在两个层面做了幂等。</strong>
> <strong>入口层：</strong>`CustomerQualificationEvaluateLaunchService`** 创建评估记录时会用 Leaf 分布式 ID 生成主键，并且用业务流水号 + 客户号 + 机构号做唯一性校验，如果记录已存在则抛 **`IdempotentException`<strong>。</strong>
> **消费层：MQ 消费时 **`AbstractAsyncEvaluateMessageConsumer`** 会先往 **`mq_consume_record`** 表插入消费记录，用 **`eventUniqueKey`<strong>（由 eventId + eventType + evaluateType 组成）做唯一索引，重复消费直接跳过。</strong>
> <strong>回调层：</strong>`sendCallBackMsgToMq`<strong> 方法会先检查评估记录的状态，如果已经是终态（</strong>`isComplete()`<strong>）则直接返回，不再处理。</strong>

#### <strong>Q11（追问）：幂等校验和业务操作不在同一个事务里，有没有并发问题？</strong>

**考察点：分布式并发场景**

<strong>回答话术：</strong>

> <strong>确实存在理论上的并发窗口。但实际影响很小，原因有两个：</strong>
> <strong>第一，入口层的幂等检查是在数据库层面用唯一索引做的，即使并发请求同时到达，数据库也只会有一条插入成功，其他会抛唯一索引异常。</strong>
> <strong>第二，回调层的状态检查虽然不是强事务，但即使有重复处理，最终结果也是幂等的——因为更新客户资质状态时，会校验当前状态是否允许转换（比如 PROCESSING 才能变成 SUCCESS），非法状态转换会直接拒绝。</strong>

---

### :link: **第五条连环问：性能与运维**

#### <strong>Q12：限流是怎么做的？为什么要限流？</strong>

**考察点：限流设计和下游保护意识**

<strong>回答话术：</strong>

> <strong>限流在两个地方做了。</strong>
> <strong>第一处在 MQ 消费端（</strong>`AbstractEvaluateHandler`<strong>），用 Rhino 的 OneLimiter 按「评估类型 + 机构编码 + 机构产品号」维度限流。这是为了保护下游机构接口不被打满，因为有些机构对请求频率有限制，超了会封禁我们。</strong>
> <strong>第二处在 HTTP Controller 层（</strong>`LimiterService`<strong>），也是用 Rhino 按「自定义限流器 key + 机构编码 + 机构产品号」维度限流，主要用于查询接口。</strong>
> <strong>限流命中的请求会直接跳过本次处理，等待下次补偿任务重新发起。</strong>

#### <strong>Q13（追问）：月付和微贷为什么要用不同的 Topic 隔离？</strong>

**考察点：流量隔离的理解**

<strong>回答话术：</strong>

> <strong>因为月付和微贷的业务特征不同。月付的机构处理速度普遍较慢，如果用同一个 Topic，月付消息积压会导致微贷消息也被阻塞，微贷用户体验下降。</strong>
> **通过 **`QualificationEvaluateDomainEventProducerFactory`** 按产品号路由到不同的 Topic（微贷走 **`domain`<strong>，月付走 </strong>`domain.billpay`<strong>），由各自的 Consumer 独立消费。这样月付积压不会影响微贷，而且可以独立监控各自的消费延迟，分别扩缩容。</strong>

## <strong>第三部分：面试终极杀招题（25K 必问）</strong>

#### <strong>终极 Q14：如果让你重新设计这个系统，你会做哪些改进？</strong>

**考察点：技术视野和反思能力**

<strong>回答话术：</strong>

> <strong>我认为当前架构有三个可以优化的点。</strong>
> <strong>第一，报文模板引擎可以更通用化。现在的变量解析和表达式替换逻辑耦合比较重，如果引入类似 Apache Velocity 或 FreeMarker 的成熟模板引擎，模板的配置会更加直观，也不需要自己维护表达式解析逻辑。</strong>
> <strong>第二，补偿机制可以更精准。现在的定时补偿是全表扫描超时记录，在数据量大的时候效率不高。可以考虑引入延迟队列（如 RocketMQ 的定时消息），为每条评估记录精确设定延迟时间，到时间自动触发补偿，避免全表扫描。</strong>
> <strong>第三，回调匹配可以更简洁。现在融合场景下的多记录匹配逻辑比较复杂（轮询关联表 + 交叉验证），本质是因为融合过渡期的历史包袱。长期来看应该统一数据模型，一个业务流水只保留一条评估记录，用字段关联多个 C 端产品，从根本上消除多记录问题。</strong>

## **第一条：领域事件驱动架构**

### **连环问**

<strong>Q1：为什么用 MQ 自产自消，不用 Spring Event？</strong>

> <strong>三个原因。第一可靠性，Spring Event 是 JVM 内存事件，服务重启后未处理的事件全丢，金融交易不能接受。MQ 消息持久化在 Broker，即使服务全挂也不丢。第二分布式支持，我们是多实例部署，Spring Event 只在当前 JVM 内传播，MQ 天然跨实例负载均衡。第三补偿重放，定时补偿需要"重新构造事件投递到 MQ"，Spring Event 没有重放能力，除非自己造轮子。</strong>

<strong>Q2：MQ 消费失败了怎么办？消息丢了怎么办？</strong>

> <strong>不会丢，三层保障。第一层消费端幂等——</strong>`AbstractAsyncEvaluateMessageConsumer`** 消费前先往 **`mq_consume_record`** 表插入一条记录，用 **`eventUniqueKey`<strong>（eventId + eventType + evaluateType）做唯一索引，重复消费直接跳过。第二层发送端重试——</strong>`AbstractMafkaProducer`<strong> 内置 guava-retryer，失败重试 3 次间隔 10ms。第三层定时补偿——</strong>`CustomerQualificationEvaluateTask`<strong> 扫描超时记录重新发 MQ。</strong>

<strong>Q3：消费端一直消费失败，消息会不会堆积爆掉？</strong>

> **会的。但我们有兜底。MQ 消费无论成功失败都返回 **`ConsumeStatus.CONSUME_SUCCESS`<strong>，不会触发 Mafka 的重投递（避免死信队列堆积）。失败的记录由定时任务来补偿处理。另外我们在消费端做了 Rhino 限流，本身就是控制消费速率，防止打满下游机构接口。</strong>

<strong>Q4（场景题）：某天凌晨 2 点，线上突然大量授信评估请求超时，你怎么排查？</strong>

> <strong>我会按这条链路排查：</strong>
> 1. **先看 Cat 监控，看是哪个阶段耗时高——是报文组装慢、限流命中多、还是调网关慢**
> 2. **如果是限流命中多，看 Rhino 配置，是不是某个机构的限流阈值被调低了**
> 3. **如果是调网关慢，看网关侧是否有变更或者机构侧响应变慢**
> 4. **看补偿任务的执行情况，超时的记录是否被正确标记失败或重新发起**
> 5. **看月付 Topic 和微贷 Topic 的消费延迟是否互相影响**

## **第二条：报文模板引擎**

### **连环问**

**Q5：报文模板引擎是怎么工作的？举个具体例子**

> **分三步。第一步从 DB 加载 **`RequestConfig`<strong>，包含报文模板 JSON 和一组 </strong>`VariableConfig`<strong>。第二步 </strong>`RequestBuildFactory`** 按变量优先级（priority 字段）从高到低分组，然后按变量类型路由到不同解析器——ROUTE 类型从 DB 查，CUSTOM 类型走 **`VariableImplementFactory`** 获取具体实现类。第三步所有变量解析完后，用 **`CustomExpresionHelper`** 做 **`${VARIABLE_NAME}`<strong> 占位符替换，生成最终 JSON 报文。</strong>
> **比如平安助贷的报文有 **`${CREDIT_LIMIT}`<strong>，对应 </strong>`CreditLimitVariable`<strong> 实现类，会根据客户风险等级等计算授信额度，然后替换进去。</strong>

**Q6：变量之间有依赖关系怎么办？变量 B 依赖变量 A 的值**

> **通过 **`priority`** 解决。每个 VariableConfig 有优先级，数值越高越先解析。用 TreeMap 倒序排列，先解析高优先级变量放入上下文 Map，低优先级变量可以直接从 Map 中引用已解析的值。另外 **`dependencyVariable`<strong> 字段声明了最终报文依赖哪些变量，替换时只注入这些变量，防止无关变量污染报文。</strong>

<strong>Q7（场景题）：现在新增一个银行，报文格式和现有机构完全不同，同时回调解析逻辑也有定制化需求，你怎么做？</strong>

> **第一步，在 **`organization_config`** 表配置新银行的机构编码、机构产品号、对应四种评估类型的路由枚举。第二步，在 **`request_config`** 和 **`variable_config`** 表配置报文模板和变量。如果报文中有新字段，新增一个 **`VariableImplement`** 实现类。第三步，如果回调解析逻辑有定制化需求（比如返回字段名不同、状态码映射不同），新增一个 Handler 继承 **`AbstractQualificationEvaluateHandler`<strong>，实现 </strong>`parseEvaluateResult`** 和 **`parseOrganizationResultStatus`<strong> 方法。报文组装和发送流程完全复用，不需要改。</strong>

## **第三条：三级补偿与幂等**

### **连环问**

**Q8：补偿任务的超时时间怎么配置？不同机构处理速度不同**

> <strong>设计了六级优先级，通过 Lion 配置中心动态管理，从高到低匹配第一个命中的：</strong>
> 1. <strong>机构_产品_评估类型（最精确）</strong>
> 2. **机构_产品**
> 3. **机构**
> 4. **产品**
> 5. **评估类型**
> 6. <strong>默认 12 小时（兜底）</strong>
> <strong>快的机构配短超时快速失败重试，慢的机构配长超时避免误判。配置变更实时生效不需要重启。</strong>

<strong>Q9：补偿任务重新发起评估，会不会和上一次的评估冲突？</strong>

> **不会。重新发起评估是构造一个新的 EVALUATE 事件投递到 MQ，消费时 **`checkContext`<strong> 会重新查数据库检查评估记录状态。如果已经是终态就直接跳过，如果是处理中状态才会重新走报文组装和网关调用的流程。而且网关侧本身也有幂等——同一笔业务流水机构不会重复处理。</strong>

<strong>Q10（场景题）：一笔授信评估，调网关返回成功，但 MQ 发送回调事件时服务挂了，怎么办？</strong>

> **服务重启后，定时补偿任务会扫描到这笔"处理中超时"的记录。然后 **`checkOrganizationRecord`<strong> 会主动调用机构查询接口（</strong>`queryEvaluateResult`<strong>），查到终态结果后走 </strong>`callbackHandle`<strong>，构造 CALLBACK 事件投递到 MQ，走正常的回调处理流程。这笔评估最终会被正确处理。</strong>

## **第四条：回调路由与数据一致性**

### **连环问**

<strong>Q11：机构回调时缺少 customerNo 或 fundPartyCode，怎么定位到正确的评估记录？</strong>

> **两级回退策略。第一级正常路径：serialNo + customerNo + fundPartyCode + productNo 精确匹配。第二级回退路径：关键字段缺失时，通过 **`biz_serial_no_relation_record`<strong> 关联表查出该流水号的所有评估记录。如果只有一条直接匹配；如果有多条（融合场景），主动调机构查询接口获取最新结果，拿到终态结果后走回调流程。</strong>

<strong>Q12：融合场景下同一流水号为什么会有多条记录？</strong>

> <strong>融合前一个机构产品对应一个 C 端产品号。融合后一个机构产品对应多个 C 端产品号（三元组）。融合过渡期我们做了补登机制——新 C 端产品的评估记录会补写到数据库。这就导致同一笔业务流水在同一个机构产品下可能存在两条记录（产品号不同，其他字段相同），但评估结果一致。</strong>

<strong>Q13：本系统查不到评估记录，但机构风险系统有数据，怎么办？</strong>

> **多级回退查询。先查本系统的 **`customer_qualification_evaluation_record`** 表。查到终态直接返回。查不到就兜底调机构风险系统的接口。这个接口也分了两套——月付走旧的 **`organizationRiskQueryService.queryCustomerQualification(customerNo, orgCode, productNo)`<strong>，微贷走新的 </strong>`orgRiskQueryService.queryCustomerQualification(customerNo, orgCode, loanType, creditLimitType, systemOperationMode)`<strong>，用三元组查询而不是产品号查询。</strong>

<strong>Q14（场景题）：机构回调报文中，fundPartyCode 传的是一个我们不认识的值，既匹配不到本系统的评估记录，也匹配不到机构风险的数据，你怎么处理？</strong>

> <strong>代码中实际就是这样处理的。</strong>`customerQualificationEvaluateCallBack`** 方法判断 **`fundPartyCode`** 或 **`customerNo`** 为空时，会走 **`qualificationCallBackProcessPolling`<strong>——先查 bizSerialNo 关联表，关联表也查不到就直接返回，不会抛异常。同时 </strong>`checkOrgRiskProcess`** 会判断机构风险侧是否有非终态数据，如果有就返回 **`transmit=true`** 让网关继续尝试回调。如果两边都找不到，抛 **`IllegalRequestException`<strong> 拒绝。</strong>

## **第五条：运维保障体系**

### **连环问**

<strong>Q15：Rhino 限流是怎么做的？限流命中了怎么办？</strong>

> <strong>两处做了限流。第一处 MQ 消费端（</strong>`CustomerQualificationEvaluateHandler.isRateLimit`<strong>），用 Rhino 的 OneLimiter 按「评估类型 + 机构编码 + 机构产品号」维度限流。限流命中直接跳过本次处理，</strong>`organizationEvaluate`<strong> 返回 null，等下次补偿任务重新发起。第二处 HTTP Controller 层（</strong>`LimiterService`<strong>），按「限流器 key + 机构编码 + 机构产品号」维度限流。</strong>
> <strong>限流的目的是保护下游机构接口——有些机构对请求频率有硬性限制，超了会封禁我们整个合作方身份。</strong>

<strong>Q16：Cat 监控埋了哪些指标？</strong>

> **通过 **`EvaluateMetricLogRouter`<strong> 做了统一的指标路由，在以下关键节点埋点：</strong>
> - `**RECEIVE_REQUEST`：请求接收**
> - `**REQUEST_ORGANIZATION`：调网关发机构**
> - `**EVALUATE_CALLBACK_FLOW`：回调处理完成**
> - `**ORGANIZATION_OVERTIME`：补偿超时标记失败**
> - `**QUALIFICATION_EVALUATE_CALLBACK`：授信回调（按机构上报）**
> - `**ORG_LOAN_RESULT_NOTICE`：放款结果通知**
> <strong>每个指标都带有 MetricModel（orgCode + orgProductNo + result），可以按机构维度分析。</strong>

<strong>Q17：月付和微贷 Topic 隔离具体怎么实现的？</strong>

> `**QualificationEvaluateDomainEventProducerFactory.getQualificationEvaluateDomainEventProducer(routeProductNo)` 根据产品号路由。如果产品号是月付（`BILL_PAY`），返回 `BillPayQualificationEvaluateDomainEventProducer`，走 Topic `domain.billpay`，由 `BillPayAsyncEvaluateMessageConsumer` 消费。其他产品走默认 Topic `domain`，由 `AsyncEvaluateMessageConsumer` 消费。两边完全独立，互不影响。**

<strong>Q18（场景题）：线上月付 Topic 消费延迟突然变高，从 1 秒涨到 30 秒，你怎么处理？</strong>

> <strong>分三步：</strong>
> 1. **先看延迟是因为消费慢还是消息量突增。看 Cat 的消费 TPS 和消息堆积量**
> 2. **如果是消费慢，看是不是某个机构回调变慢导致事务阻塞（回调处理在 **`@Transactional`** 里）。通过 Cat 的接口级监控定位到具体的慢机构**
> 3. **如果是消息量突增，看是不是上游有活动放量。月付和微贷已经隔离了，不会互相影响，所以只需要关注月付自身的消费能力，必要时临时扩容月付消费实例**

## <strong>终极场景题合集（25K 必问级别）</strong>

### **场景题 1：幂等边界**

<strong>问：消费者拿到消息后，插入幂等记录成功了，但在处理业务逻辑时抛了异常（比如数据库超时），这条消息怎么办？</strong>

> **消费者返回的是 **`ConsumeStatus.CONSUME_SUCCESS`<strong>，所以 Mafka 不会重新投递。但幂等记录的状态会被更新为 </strong>`FAILURE`<strong>。后续定时补偿任务 </strong>`AsyncMessageCompensateTask`** 会扫描 **`mq_consume_record`<strong> 中状态为 FAILURE 的记录，重新构造消息投递到 MQ 触发重新消费。</strong>

### **场景题 2：并发回调**

<strong>问：同一笔评估的机构回调，由于网络原因机构重发了 3 次，3 个请求几乎同时到达，怎么保证只处理一次？</strong>

> `**sendCallBackMsgToMq` 方法第一行就检查评估记录状态：`if (customerQualificationEvaluationRecord.getEvaluateStatus().isComplete()) return;`。第一次处理会把状态更新为终态（在 `@Transactional` 中），后续两次请求查到终态直接跳过。即使两个请求同时通过状态检查进入 MQ 消费阶段，消费端的幂等表（eventUniqueKey 唯一索引）也会保证只处理一次。**

### **场景题 3：数据不一致修复**

<strong>问：线上发现一笔授信评估，本系统状态是 PROCESSING，但机构风险侧已经是 SUCCESS，客户来查却查不到授信额度，你怎么修复？</strong>

> <strong>这是一个需要补偿处理的典型案例。定时补偿任务扫描到这笔 PROCESSING 超时的记录后，</strong>`checkOrganizationRecord`** 会主动调机构查询接口查到 SUCCESS 结果，然后走 **`callbackHandle`** 构造 CALLBACK 事件投递到 MQ。消费端回调 Handler 会更新本系统评估记录为终态，同时更新客户资质状态，客户下次查询就能看到授信额度了。如果这是紧急情况需要人工介入，也可以通过 TestController 的 **`/compensate`<strong> 接口手动触发补偿。</strong>

### **场景题 4：新机构接入**

<strong>问：产品说下周要新接入一家宁波银行，本周要完成开发，你怎么安排？</strong>

> <strong>首先确认宁波银行和现有哪些机构报文格式类似，如果有类似的可以直接复用报文模板，只改变量值。然后分三步：</strong>
> 1. <strong>DB 配置：</strong>`organization_config`<strong> 表配机构编码和路由枚举，</strong>`request_config`** 和 **`variable_config`** 表配报文模板和变量**
> 2. **如有新字段：新增 **`VariableImplement`** 实现类**
> 3. **如有定制回调：新增 **`OrganizationHandler`** 子类**
> <strong>代码改动量很小，大部分时间花在配置和联调上。</strong>

### **场景题 5：系统重构**

<strong>问：如果让你重构报文模板引擎，你怎么改？</strong>

> **三个方向。第一，把 **`CustomExpresionHelper`<strong> 替换为成熟的模板引擎如 Velocity 或 FreeMarker，模板配置会更直观，不需要自己维护表达式解析。第二，变量解析引入策略模式替代现在的 VariableType 路由，让每个 Variable 自己决定能不能处理，而不是靠类型枚举分发。第三，报文模板加版本管理，支持灰度发布——新模板先在测试机构验证，没问题再全量切换。</strong>

# **Thrift RPC与dubbo有何区别？Mafka与kafka呢？Zebra与Sharding呢？Rhino与sentinel呢？请从写出区别和使用场景**

## **一、Thrift RPC vs Dubbo**


| **维度** | <strong>Thrift RPC（美团 OCTO-Thrift）</strong> | <strong>Dubbo（Apache）</strong> |
|---|---|---|
| **协议** | **Thrift 二进制协议，紧凑高效，自带 IDL 定义语言** | **支持多协议（dubbo/hessian/rest 等），默认 dubbo 协议** |
| **IDL** | `**.thrift` 文件定义接口和结构体，编译生成多语言代码** | **Java 接口定义，需配合 API Jar 包依赖** |
| **序列化** | **Thrift Binary/TCompact，体积小、速度快** | **Hessian2/Kryo/Protobuf 等，需额外配置** |
| **服务治理** | <strong>美团 OCTO 平台统一治理（注册发现、限流、熔断、灰度）</strong> | **自身提供注册中心（ZooKeeper/Nacos）+治理能力** |
| **跨语言** | <strong>天然跨语言（C++/Java/Python/Go/Node.js）</strong> | **主要面向 Java 生态** |
| **参数类型** | **基本类型 + Struct/List/Map，不支持方法重载** | **支持任意 Java 对象，支持方法重载** |
| **使用场景** | **微服务间高频 RPC 调用，跨语言服务对接** | **Java 微服务生态，内部服务间调用** |

<strong>面试话术：</strong>

> <strong>我们项目用的是美团 OCTO-Thrift。Thrift 的优势是序列化效率高、跨语言支持好——网关那层就是 C++ 写的，我们用 Thrift 天然就能对接。缺点是 IDL 定义比较受限，不支持方法重载，自定义异常也不如 Java 原生异常灵活。Dubbo 在 Java 生态里更方便，注解化开发体验好，但跨语言需要额外适配。</strong>

---

## **二、Mafka（美团 Mafka）vs Apache Kafka**


| **维度** | <strong>Mafka（美团）</strong> | **Apache Kafka** |
|---|---|---|
| **本质** | **基于 Apache Kafka 深度定制的封装** | **开源消息队列** |
| **差异** | **统一了配置方式、支持权限管理、与 OCTO 平台集成** | **原生 Kafka 配置管理较原始** |
| **生产 API** | **MafkaProducer / IProducerProcessor 封装** | **KafkaProducer 原生 API** |
| **消费 API** | **IMessageListener 接口 + 注解配置** | **Consumer / KafkaConsumer 原生 API** |
| **Topic 管理** | **通过平台统一创建和权限分配** | **需要手动创建和管理** |
| **监控** | **集成 CAT 监控，自动上报消费延迟等指标** | <strong>需要自建监控体系（Kafka Manager/Metrics）</strong> |
| **可靠性** | <strong>继承 Kafka 的高可靠性（副本、ACK 机制）</strong> | **同 Kafka** |
| **使用场景** | **事件驱动、异步解耦、日志收集** | **同上** |

<strong>面试话术：</strong>

> **我们项目用的是美团 Mafka，本质上就是 Apache Kafka 的封装。它帮我们省去了 Topic 管理、权限控制、监控集成等运维工作。我们的消息发送用 **`MafkaProducer`<strong>，消费实现 </strong>`IMessageListener<MafkaMessageRecord>`<strong> 接口。Mafka 本身的可靠性机制（副本 + ACK）保证了消息不丢，配合我们自己的幂等表实现了 Exactly-Once 语义。性能上和原生 Kafka 没有本质差异。</strong>

---

## **三、Zebra（美团）vs ShardingSphere**


| **维度** | <strong>Zebra（美团 Zebra）</strong> | <strong>ShardingSphere（Apache）</strong> |
|---|---|---|
| **定位** | **美团数据库中间件，统一数据源代理** | **开源分布式数据库中间件** |
| **分库分表** | **通过 **`ruleName`** 配置分片规则，对业务代码透明** | **支持 Groovy/inline 等多种分片策略配置** |
| **数据源路由** | **通过 **`jdbcRef`** 配置多数据源，自动路由** | **支持读写分离 + 分库分表路由** |
| **集成方式** | **配置 **`application.yml`** 中的 zebra 段即可** | **需要引入 starter + 配置分片规则** |
| **监控** | **集成 CAT，自动上报 SQL 执行指标** | **需要自建监控** |
| **强制读从** | **支持 **`@ForceShardRead`** 注解指定读从库** | **支持读写分离 Hint** |
| **使用场景** | **美团内部统一的数据访问层** | **开源社区广泛使用** |

<strong>面试话术：</strong>

> **我们项目用 Zebra 做分库分表，按 **`customerNo`** 分片。Zebra 是美团统一的数据访问层，对业务代码完全透明，SQL 写法和单库没有区别。配置上只需要在 **`application.yml`** 指定 **`ruleName`** 和 **`jdbcRef`<strong>。另外我们还用了 Zebra 的读写分离能力，通过 </strong>`@ForceShardRead`** 注解在特定场景下强制走从库查询，比如 **`ForceShardRouter`<strong> 就是用来处理强制读从路由的 AOP 切面。</strong>

---

## <strong>四、Rhino（美团）vs Sentinel（阿里）</strong>


| **维度** | <strong>Rhino（美团）</strong> | <strong>Sentinel（阿里）</strong> |
|---|---|---|
| **定位** | **美团统一限流熔断平台** | **开源流量防护组件** |
| **限流模式** | **OneLimiter：单机令牌桶，适合高频轻量限流** | **滑动窗口/令牌桶/漏桶多种模式** |
| **配置** | **通过管理后台配置限流规则，按 key + params 维度** | **支持代码配置、Nacos 动态规则、控制台配置** |
| **熔断** | **支持，与限流统一管理** | <strong>独立的熔断降级机制（状态机）</strong> |
| **粒度** | <strong>按 key + 自定义参数（如 organization_code + org_product_no）</strong> | **按资源名 + 限流规则** |
| **集成** | `**Rhino.newOneLimiter("name")` 一行代码创建** | `**@SentinelResource` 注解或 SphU.entry()** |
| **返回值** | `**isPass()` 返回布尔值，简单直接** | **返回 FlowSlotResult，包含剩余配额等信息** |
| **使用场景** | **接口级高频限流，保护下游外部系统** | <strong>通用流量防护（限流+熔断+降级+热点参数）</strong> |

<strong>面试话术：</strong>

> <strong>我们项目用 Rhino 做接口级限流，保护下游机构接口不被打满。用法很简单——</strong>`Rhino.newOneLimiter("organization-request")`** 创建限流器，调用 ****[oneLimiter.run](http://oneLimiter.run)**`(entrance, params, 1).isPass()`<strong> 判断是否放行。我们在两个地方做了限流：MQ 消费端按「评估类型 + 机构编码 + 产品号」维度限流，HTTP Controller 层按自定义 key + 机构 + 产品号维度限流。限流命中的请求直接跳过，等补偿任务重试。</strong>
> <strong>和 Sentinel 相比，Rhino 更轻量，API 设计更简洁（直接返回 boolean），适合我们的场景——只需要判断放不放行，不需要 Sentinel 那么多的熔断降级策略。</strong>

## **六、面试加分回答**

<strong>如果面试官问 “为什么不用开源的，非要用美团内部的”，你可以这样回答：</strong>

> <strong>这不是我们选择的，是美团基础架构层面的统一技术栈。所有美团 Java 服务都用 OCTO-Thrift 做 RPC、Mafka 做消息、Zebra 做数据访问、Rhino 做限流、Cat 做监控。好处是全链路工具统一打通——比如 Cat 的 traceId 可以从 Thrift 调用透传到 Mafka 消费再到 Zebra SQL 执行，形成完整的调用链追踪，用开源组件很难做到这种深度集成。</strong>
