# 二、流程编排总结

分析一下类图核心类

Process 接口定义了一套标准的子流程执行过程：参数是WorkFlow 定义了是购买流程还是赎回流程或者其他。

check 方法校验当前子流程是不是需要执行：参数是WorkFlow 定义了是购买流程还是赎回流程或者其他。

excute 用来执行子流程：传入一个 Node，记录当前子流程的状态，也就是打的状态机。

componsate 补偿子流程

receiveCallBack 处理子流程回调

最后一个场景接口 getProcess 给一个子流程类型。



每一个子流程只需要实现Process并且加上自定义注解@SubProcessConfig(code = "PURCHASE_RESULT_NOTIFY", desc = "购买结果领域事件")

就可以定义子流程类型和执行逻辑。



“静态双向链表配置 + 动态快照驱动 + 递归前置检查”

引擎不会记住流程走到哪了，而是每次都去查 快照表 (`++OrganizationOrderProcessSnapshot++`)。

在 `AbstractWorkFlowEngine` 中，流转的核心逻辑如下：

1. 判断当前节点是否结束： 只有当前节点的状态是 `SUCCESS` 或 `FAIL`（终态），才允许流转。
2. 指针移动 (Next Step)： 找到下一个节点，更新快照的 currentSubProcess 字段，并将状态重置为 INIT。
3. 条件跳过：递归检查 (The Filter)
  这是流转中比较智能的一部分。链表是死的（A->B->C），但业务是活的（有时候不需要跑 B）。

假设配置了流程：A -> B -> C -> D

1. 初始状态：快照指向 A，状态 `INIT`。
2. 执行 A：引擎执行 A 的 `execute()`，A 完成后更新快照状态为 `SUCCESS`。
3. 流转触发：引擎调用 `driveToNextSubProcess`。
  - 看 A 的 next 是 B。
  - 调用 B 的 `check()`。假设 B 返回 `true`。
  - 更新快照：指向 B，状态 `INIT`。
4. 执行 B：B 完成，状态 `SUCCESS`。
5. 流转触发：
  - 看 B 的 next 是 C。
  - 调用 C 的 `check()`。假设 C 返回 `false`（不需要执行）。
  - 递归：看 C 的 next 是 D。
  - 调用 D 的 `check()`。假设 D 返回 `true`。
  - 更新快照：指向 D (直接跳过了 C)，状态 `INIT`。

# 三、redis作用。

### **1、分布式锁 (Distributed Lock) —— “看门人”**

这是代码中体现最明显的一点。在涉及资金、额度、征信等敏感操作时，必须防止同一个单据被并发处理。

当多个线程（或多台服务器）同时尝试处理同一个 WorkFlow（例如同一个 processOrderId）时，系统会利用 Redis 的 SETNX 或 Redisson 框架加锁。

- 场景：防止用户手抖连点两次“申请”，或者补偿任务和正常流程同时触发。
- 效果：只有一个线程能拿到锁执行 creditApply，其他的会抛出 ConcurrencyControlLockException 并放弃或重试。

### 2、**配置与灰度缓存 (Cache) —— “加速器”**

流程编排引擎极其依赖配置数据（OrganizationSubProcess），如果每走一步都查数据库，数据库压力会巨大。

- **作用：**
  - **流程配置缓存：`OrganizationSubProcess` 表的数据是静态的（配置好后很少改）。Redis 会缓存这些配置对象（Map结构：`id -> ConfigJson`）。**
  - **灰度开关缓存：`subProcessGrayAbilityService` 需要判断当前是否走灰度逻辑，这种高频的开关读取通常都在 Redis 或本地缓存（Guava）中。**

### **3. 幂等性校验 (Idempotency)**

**虽然代码中大量使用了 `createIfNotExist`（基于数据库唯一索引的幂等），但在高并发入口处，通常会先查 Redis。**

- **逻辑推断： 在 `AssetRedeemOrderService` 或 `AssetPurchaseOrderService` 的入口处，为了保护数据库，通常会以 `BizSerialNo`（业务流水号）为 Key 在 Redis 设一个短期标记。**
  - **如果 Redis 里有这个 Key，说明该请求正在处理，直接返回，避免请求打到数据库层引发 `DuplicateKeyException`。**

### **4. 澄清：它在这个系统中不是消息队列**

**值得注意的是，虽然代码里出现了 `ResizeableLinkedBlockingQueue` 和 `AsyncProcessingBillExecutor`，但这部分并没有使用 Redis。**

- **内存队列：`ResizeableLinkedBlockingQueue` 是 JVM 内存里的队列。如果机器重启，队列里的任务会丢失。**
- **数据库兜底：为了防止丢失，系统采用了“数据库 + 异步线程”的模式。**
  - **先落库（`processingBillService.addProcessingFlow`）。**
  - **再放入内存线程池执行。**
  - **如果内存队列丢了，会有定时任务（如 `OrganizationCustomerOcrEvaluateService` 中的补偿逻辑）从数据库捞取 `INIT` 状态的记录重新执行。**



# 四、MQ 作用

**基于代码库中的 `ProcessSubject` 使用模式以及金融交易系统的通用架构，MQ（消息队列，如美团内部的 Mafka/Swallow）在该系统中主要承担 “领域事件广播”、“异步解耦” 和 “最终一致性保障” 三大核心作用。**

**以下是结合代码的详细分析：**

### **1. 领域事件广播 (Domain Event Broadcasting)**

**这是代码中出现频率最高的使用场景。**

- **代码证据： 几乎所有的流程节点服务（`AbstractContractCreationProcessService`, `AbstractRedeemAssetDeductProcessService` 等）在 `execute` 方法的最后一步，都会调用：**

```java
// 4、领域事件
processSubject.notify(buildExecutionEvent(record));
```

- **作用： 当一个子流程节点（如“合同签署”或“资金扣减”）完成时，系统通过 MQ 发出一个“事件消息”。**
  - **解耦：当前流程引擎不需要知道谁关心这个结果。**
  - **下游订阅：**
    - **账务系统：订阅“扣减成功”消息，记流水。**
    - **风控系统：订阅“评估完成”消息，更新额度。**
    - **数据仓库：订阅所有节点消息，生成报表。**

### **2. 异步处理与削峰 (Asynchronous Processing)**

对于耗时较长且非核心链路的逻辑，通过 MQ 进行异步化，避免阻塞主流程线程。

**代码证据**： 在 http://OrganizationCustomerOcrEvaluateService.java 中出现了异常抛出：

throw new CompensateTaskException("补偿消费消息出现异常", e);

- **场景：OCR（图片文字识别）、生成 PDF 合同、上传影像件等操作非常耗时（可能几秒到几十秒）。**
- **机制：主流程只需发一条“需要OCR”的消息到 MQ，然后立即返回或进入等待状态。专门的 Consumer 服务从 MQ 拉取消息慢慢处理，处理完再回调主流程。这能有效防止流量高峰冲垮这些重资源服务。**

### **3. 结果通知 (Result Notification)**

**当整个工作流（WorkFlow）终态（成功或失败）时，需要通知上游业务方（如 C 端 APP 后端）。**

- **代码证据： `PurchaseResultNotifyService`（虽然未展示全部代码，但从命名和 `buildResultNotifySubProcess` 可推断）。**
- **作用：**
  - **可靠投递：如果直接调 HTTP 通知上游，上游挂了怎么办？用 MQ 可以利用其 重试机制。如果通知失败，MQ 会在 1s, 5s, 1min... 后自动重试，直到成功，确保“不丢单”。**

### **4. 分布式事务的最终一致性 (Eventual Consistency)**

**在微服务架构下，无法使用本地数据库事务来保证跨服务的操作原子性。MQ 是实现最终一致性的关键。**

- **机制分析： 在 `execute` 方法中：**
  1. **本地事务：更新数据库状态为 `SUCCESS`。**
  2. **发送消息：`processSubject.notify(...)` 发送 MQ 消息。**
  ***注意：为了保证这两步的原子性，通常会配合“本地消息表”或“事务型消息”（如 RocketMQ 的 Transaction Message）。***
  **一旦消息发出，下游系统（如额度系统）一定会收到消息并执行操作。如果下游处理失败，依靠 MQ 的重试或死信队列（DLQ）人工介入，从而保证“主流程成功”=“下游最终也成功”。**

## 六、其他值得关注的技术点：

### **1. 亲源性线程池 (Key Affinity Executor)**

# **这是一个非常有特色的技术点，用于解决 “并发处理与顺序性” 的冲突。**

- **代码证据： AsyncProcessingBillExecutor.java 中使用了 `KeyAffinityExecutor`。**

- **作用：**
  - **顺序性：对于同一个订单（FlowId），它保证了任务是串行执行的。这避免了多线程同时操作同一个订单导致的数据库死锁或状态覆盖问题。**
  - **并发性：对于不同的订单，它们会被分发到不同的线程中并行执行，保证了系统的整体吞吐量。**
  - **比全局锁更轻量：不需要去 Redis 抢锁，直接通过哈希算法在内存层面就分配好了“跑道”。**

**4. 乐观锁 (Optimistic Locking)**

**在数据库更新层面，为了防止并发覆盖。**

- **代码证据： `orderProcessSnapshotService.updateByStatus` 方法的参数暗示了这一点。**
- **SQL 层面**

```sql
UPDATE snapshot_table 
SET status = 'PROCESSING', version = version + 1 
WHERE id = 123 
  AND status = 'INIT'  -- 关键：带上旧状态作为条件
  AND current_subprocess = 10;
```

- **作用：**
  - **如果两个线程同时想驱动同一个订单，第一个更新成功后，状态变了；第二个线程的 SQL `WHERE` 条件就不满足了，更新行数为 0，从而避免了状态错乱。**
