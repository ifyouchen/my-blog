<strong>依赖关系：</strong>`adaptor → service → repository → core`<strong>，</strong>`client`<strong> 独立对外发布。</strong>

**这个项目本质上是一个流程编排引擎，管理着8大类工作流**


| **流** | **引擎类** | **业务含义** | **通俗解释** |
|---|---|---|---|
| <strong>PURCHASE（购买流程）</strong> | `**PurchaseProcessEngine**` | **银行购买客户的借款资产** | **银行出钱，把用户的借条买过来** |
| <strong>ORDER_REDEEM（赎回单流程）</strong> | `**RedeemOrderProcessEngine**` | **按赎回单赎回资产** | **按单笔赎回，把钱还给银行** |
| <strong>ASSET_REDEEM（资产赎回流程）</strong> | `**AssetRedeemProcessEngine**` | **按资产维度赎回** | **用户还款后，对应资产赎回** |
| <strong>CUSTOMER_REPAYMENT（客户还款流程）</strong> | `**CustomerRepaymentProcessEngine**` | **客户主动还款处理** | **用户还钱后的处理流程** |
| <strong>PRE_EVALUATE（预评估流程）</strong> | `**PreEvaluateProcessEngine**` | **放款前的初步评估** | **先看看用户够不够格借钱** |
| <strong>QUALIFICATION_EVALUATION（资质评估流程）</strong> | `**QualsEvaluateProcessEngine**` | **正式的资质评估** | **详细评估用户的信用资质** |
| <strong>QUALIFICATION_REFRESH（资质刷新流程）</strong> | `**QualificationRefreshProcessEngine**` | **刷新已有资质** | **资质过期了，重新评估一下** |
| <strong>ORGANIZATION_CUSTOMER_CLEAR（客户清退流程）</strong> | `**OrganizationCustomerClearProcessEngine**` | **清退某个银行下的客户** | **银行不要这个客** |

### **3.2 购买流程（最核心的流程）详细拆解**

<strong>购买流程是整个系统最核心、最复杂的流程，它的子流程链如下：</strong>

`购买申请 → 资质评估 → 购买评估 → 影像传输 → 合同创建 → 额度冻结 → 支付 → `
`购买结果通知 → 机构审批 → 额度扣减 → 提现 → 额度创建/提额 → 完成`

**每个子流程对应一个 **`@SubProcessConfig`<strong> 注解的实现类，例如：</strong>

- `**PURCHASE_PAYMENT` → `AifCommonPaymentProcessService`（通联金融通道支付）**
- `**PURCHASE_RESULT_NOTIFY` → `PurchaseResultNotifyService`（购买结果通知）**
- `**ORGANIZATION_LIMIT_FREEZE` → `OrganizationLimitFreezeService`（额度冻结）**
- `**ORGANIZATION_LIMIT_DEDUCT` → `OrganizationLimitDeductService`（额度扣减）**

## <strong>面试时如何描述"我理解了这个项目"</strong>

<strong>你可以这样组织回答（以购买流程为例）：</strong>

> **问：介绍一下你负责的核心业务**
> <strong>我负责的是金融机构资产交换平台的购买流程。简单说就是美团把用户的借款资产"卖"给银行。</strong>
> **整个流程基于一套自研的流程编排引擎驱动。创建购买单后发MQ异步消息，消费者触发 **`PurchaseProcessEngine`<strong>，引擎读取 </strong>`organization_order_process_snapshot`<strong> 表（流程快照），根据快照中记录的当前子流程状态，决定是执行、补偿、还是流转到下一个子流程。</strong>
> <strong>购买单有两个维度的状态：</strong>`processStatus`<strong>（流程层面）和 </strong>`purchaseStatus`<strong>（业务层面），分别用乐观锁（CAS）保证并发安全。</strong>
> <strong>子流程之间通过数据库链表（</strong>`prevSubProcessConfigId`** / **`nextSubProcessConfigId`<strong>）串联，引擎用 </strong>`@SubProcessConfig`** 注解 + **`ProcessImplMatcher`<strong> 服务定位器模式匹配具体实现类。</strong>
> **每个子流程执行时都遵循幂等模式（先查再建），状态更新都带前置状态校验（乐观锁）。异步子流程（如支付、风控评估）通过 Mafka 消息回调，回调后更新记录状态，再通过 Spring **`@EventListener`<strong> 异步驱动引擎流转到下一子流程。</strong>
> **对于失败场景，项目用定时补偿 Job 定期扫描 PROCESSING 状态超时的记录，重新触发执行，保证最终一致性**

## <strong>为什么要"自产自消"？</strong>

<strong>这是一种常见的异步化设计模式，核心目的是：</strong>


| **原因** | **解释** |
|---|---|
| **RPC快速返回** | **上游系统调用 **`purchase()`<strong> 时，只需要等 DB INSERT + MQ 发送完成，不用等整个流程执行完。购买流程有十多个子流程，可能耗时几秒甚至几十秒。</strong> |
| **解耦** | **RPC 线程和流程引擎线程池隔离，避免一个慢子流程把 RPC 线程池打满** |
| **天然重试** | <strong>如果 MQ 消费失败，Mafka 会自动重投（</strong>`RECONSUME_LATER`<strong>），不需要额外实现重试逻辑</strong> |
| **幂等保护** | **消费前会查 `mq_consumer_record` 表判断是否重复消费，唯一键 = `customerNo + productNo + referenceNo + topic + bizScene + uniqueKey**` |

# 补偿

## 补偿机制

```java
当前时间: NOW
│
│◄── 15分钟 ──►│◄──────────────── 24小时 ────────────────►│
│              │                                     │
│    不补偿      │          补偿扫描范围                │  不补偿
│  (可能还在处理)│                                     │ (太久远了)
│              │                                     │
         endTime                               beginTime
         = NOW - 15min                        = NOW - 24h15min


分批查询（每1小时一批）:

  beginTime                  endTime
     │                          │
     ├───── 1小时 ──────┤────── ───── 1小时 ──────┤────── ...
     │                  │                        │
     批次1              批次2                   批次N

```

<strong>为什么分批？ 防止单次查询数据量过大。24小时的窗口如果一次查出来，可能有几万条记录。</strong>

```java
补偿 Job 执行流程：

1. Crane 定时触发（约每5-10分钟）
      ↓
2. 计算时间窗口 [NOW-24h15min, NOW-15min]
      ↓
3. 按每小时分批扫描 processing_flow / processing_record 表
      ↓
4. 反查业务表，过滤出 PROCESSING 状态的记录
      ↓
5. 逐条执行补偿逻辑：
   - 流程级：重新触发 PurchaseProcessEngine.execute()
   - 子流程级：调用 Process.compensate() → 主动查询外部结果
      ↓
6. 更新状态 → 发 Spring Event → 引擎流转到下一子流程

```

**面试话术：补偿采用滑动窗口 + 分批查询的策略，扫描 [15分钟前, 24小时15分钟前] 的记录，按1小时间隔分批，避免全表扫描。流程级补偿触发引擎重新执行，子流程级补偿主动查询外部结果。还通过 **`processing_flow`** / **`processing_record`<strong> 辅助表做索引，不直接 scan 业务大表。</strong>

## 超时关单机制？

```java
订单卡住（超时/外部异常）
    │
    ├── 15分钟后 → 补偿Job扫描到 → 重新执行 PurchaseProcessEngine（反复重试）
    │   └── 持续补偿直到 24.5 小时
    │
    ├── 30分钟后 → 监控Job扫描到 → 发送 DX 告警通知运营/开发
    │   └── 持续监控直到 48 小时
    │
    └── 24.5小时后 → 运营人工确认 → 触发强制失败接口
        └── 校验通过 → 更新购买单 + 快照 + 各子流程记录为 FAIL

```

<strong>系统不会自动关闭超时订单。卡住的订单会被不断补偿重试，同时告警通知相关人员，最终由人工判断并执行强制失败操作。这是一个典型的"宁可重试也不误关"的设计，因为涉及资金交易，自动关单可能导致资金风险。</strong>

# **Spring Event 和 Mafka 消息在这个项目中是两套完全不同的机制**

## <strong>先看本质：它们底层都是谁？</strong>


|  | `**processSubject.notify()**` | `**exchangeAsyncMessageProducer.sendMsgToMq()**` |
|---|---|---|
| **底层实现** | `**ApplicationEventPublisher.publishEvent()**` | `**MafkaProducer.sendMessage()**` |
| **本质** | **Spring 本地事件，JVM 进程内同步/异步调用** | <strong>Mafka 远程消息，跨进程（虽然自产自消）</strong> |
| **触发方** | **子流程内部的 **`process.execute()`** 方法** | **RPC 入口 **`purchase()`** 方法** |
| **监听方** | `**NodeEventListener`（`@Async @EventListener`）** | `**ExchangeAsyncEventConsumer`（Mafka Consumer）** |

## **关键区别：看 **`isExecutionFinished`** 的判断**

`**ProcessSubject` 的代码非常关键：**

```text
// private void notifyExecutionFinishedEvent(ProcessExecutionEvent<Node> event) {
    if (event.isExecutionFinished()) {   // ← 只有子流程执行完成时才发布
        applicationEventPublisher.publishEvent(event);
    }
}

```

**而 **`NodeEventListener`<strong> 收到事件后的处理：</strong>

```text
// private void handlePurchaseProcessFinishEvent(ProcessExecutionEvent<Node> event) {
    if (!WorkFlowType.PURCHASE.equals(event.getWorkFlowType()) || !event.isExecutionFinished()) {
        return;  // ← 只关心"执行完成"的事件
    }
    // 触发流程引擎继续执行下一个子流程
    purchaseProcessEngine.execute(order, new EngineExecuteContext(...));
}


```

## **两条链路的对比图**

`═══════════════════════════════════════════════════════════════════`
`链路A：Mafka 消息（触发整个流程引擎，仅执行一次）`
`═══════════════════════════════════════════════════════════════════`

`时机：RPC入口处`
`触发：purchase() 创建完购买单后`
`目的：把"同步RPC"转成"异步执行流程"`

`RPC → 创建购买单(DB INSERT) → 发Mafka → RPC立即返回`
`↓`
`MQ消费者线程(另一个线程)`
`↓`
`PurchaseProcessEngine.execute()`
`↓`
`beginPurchaseProcess() ← 第1次：初始化流程`
`执行第1个子流程`
`↓`
`(子流程完成后发Spring Event)`
`↓`
`PurchaseProcessEngine.execute() ← 第2次：执行第2个子流程`
`↓`
`... 依次类推 ...`

`═══════════════════════════════════════════════════════════════════`
`链路B：Spring Event（子流程间的级联流转，每次子流程完成都会触发）`
`═══════════════════════════════════════════════════════════════════`

`时机：每个子流程的 process.execute() 内部最后一步`
`触发：子流程执行完，processSubject.notify(event)`
`目的：驱动引擎流转到下一个子流程`

`子流程1.execute()`
`→ createIfNotExist() // 创建记录`
`→ applyXxx() // 远程调用`
`→ processSyncResult() // 处理结果`
`→ processSubject.notify() // ★ 发Spring Event ★`
`↓`
`Spring ApplicationEventPublisher`
`↓ (异步，eventExecutor线程池)`
`NodeEventListener.handleNodeEvent()`
`↓`
`PurchaseProcessEngine.execute() ← 触发下一个子流程`

## <strong>为什么不能把链路A（Mafka）换成 Spring Event？</strong>

<strong>答案是可以换，但这个项目选择用 Mafka 有三个工程层面的原因：</strong>

### <strong>原因1：跨进程可靠性（最重要）</strong>

<strong>虽然现在是"自产自消"，但 Mafka 消息是持久化到 broker 磁盘上的。如果 JVM 在发完消息后、消费前崩溃了：</strong>


| **方案** | **JVM崩溃后** | **机器重启后** |
|---|---|---|
| **Mafka** | **消息已经在broker上，不会丢** | **自动重新消费，流程继续执行** |
| **Spring Event** | **事件在内存中，直接丢失** | **购买单永远卡在 INIT 状态，需要人工处理** |

<strong>Mafka 还自带重试机制（</strong>`RECONSUME_LATER`<strong>），消费失败会自动重投。Spring Event 如果异步线程抛异常，事件就丢了。</strong>

### **原因2：消息消费有幂等表保护**

<strong>Mafka 消费链路多了一层保护：</strong>

```text
// AbstractMafkaListener 中protected boolean checkRepeatMessage(mapping) {
    MqConsumeRecord oldRecord = mqConsumerRecordService
        .queryByUniqueKey(customerNo, productNo, referenceNo, topic, bizScene, uniqueKey);
    return oldRecord == null;  // 已消费过 → 拦截
}

```

**先查 **`mq_consumer_record`<strong> 表，如果已存在说明重复消息，直接跳过。Spring Event 没有这层保护，如果因为某种原因触发了两次，就会导致流程重复执行。</strong>

### **原因3：未来可扩展为跨服务消费**

**虽然现在 **`purchase()`** 和 **`ExchangeAsyncEventConsumer`<strong> 在同一个 JVM 里，但这个架构天然支持拆分。比如将来购买申请由服务A处理，购买流程引擎由服务B处理，只需要把 MQ 消费者挪到服务B就行。Spring Event 永远只能在同一个 JVM 里。</strong>

## <strong>为什么子流程间流转又用 Spring Event 而不用 Mafka？</strong>

<strong>因为子流程间流转的可靠性要求不同：</strong>


| **维度** | <strong>链路A（入口 → 引擎）</strong> | <strong>链路B（子流程 → 子流程）</strong> |
|---|---|---|
| **丢失后果** | **整个购买流程不会启动，严重** | **子流程卡在 PROCESSING，补偿 Job 会修复** |
| **频率** | **每个购买单只有1次** | <strong>每个子流程完成都触发（十几次）</strong> |
| **延迟要求** | **不敏感，几秒内即可** | **越快越好，减少整体耗时** |
| **持久化成本** | <strong>值得（1条消息保证1个流程）</strong> | <strong>不值得（十几次MQ开销太大）</strong> |
| **失败处理** | **没有 Job 兜底** | **有 **`WorkFlowCompensateJob`** 兜底** |

<strong>简单说：入口用 Mafka 是为了不丢，子流程间用 Spring Event 是为了快。</strong>

## **一、项目全景与架构理解**

### **Q1：请你画一下整体架构图，解释各模块的职责和依赖关系**

**考察点：模块化设计能力、分层架构理解**

<strong>回答：</strong>

<strong>这个项目是一个典型的多模块 Maven 分层架构的金融交易系统，一共6个模块：</strong>

```text
xy-fin-orgexchange（父POM）
├── xy-fin-orgexchange-core ← 领域核心层：领域模型、枚举、工具类，零外部依赖
├── xy-fin-orgexchange-client ← API契约层：纯 Thrift DTO + 接口定义，供消费方引用
├── xy-fin-orgexchange-repository ← 持久层：MyBatis Mapper + Zebra数据源
├── xy-fin-orgexchange-threadpool ← 基础设施层：自定义动态线程池封装
```

- **core 只放纯粹的领域概念，不依赖任何框架，保证可测试性和稳定性**
- <strong>client 独立发布到私服，消费方只需要引用 client 就能调用接口，实现了API契约与实现的解耦，版本号独立维护（当前 1.0.54）</strong>
- **service 是最重的模块，承载所有业务逻辑，通过内部 DDD 分层（application/facade/service/domain/persist）组织**
- <strong>adaptor 作为薄层，只做协议转换（Thrift→Service）和外部触发（MQ Consumer、Job）</strong>
- <strong>如果让你重新设计模块拆分，你会怎么调整？</strong>

**追问 - 如果重新设计： 我会考虑把 **`service`** 中职责过重的部分进一步拆分。当前 service 模块承载了额度管理、支付、赎回、合同、评估、监控、灰度等几乎所有业务，模块已经有 200+ 类。可以考虑按业务域拆分为独立 service 模块（如 **`orglimit-service`<strong>、</strong>`purchase-service`<strong>），通过 Thrift 接口互相调用，但这样会增加运维复杂度，需要权衡。</strong>

### <strong>Q2：系统是做什么的？核心业务流程？</strong>

**考察点**：业务理解深度

<strong>回答：</strong>

<strong>这个系统叫 orgexchange（机构交易系统），是连接美团金融和外部合作金融机构（银行、消费金融公司）的核心桥接平台。</strong>

<strong>一句话概括：美团金融的"机构对接网关"，把用户的借款、还款、赎回等金融操作路由到不同的合作机构去执行。</strong>

<strong>核心业务流程包括：</strong>


| **流程** | **说明** | **典型场景** |
|---|---|---|
| <strong>Purchase（购买/放款）</strong> | **用户申请借款 → 资质评估 → 额度冻结/扣减 → 支付 → 通知** | **消费贷/经营贷放款** |
| <strong>Redeem（赎回）</strong> | **用户提前结清或资产赎回 → 额度恢复 → 结算** | **提前还款、资产转让** |
| <strong>Repayment（还款）</strong> | **用户正常还款 → 机构确认 → 账务更新** | **月供还款** |
| <strong>Evaluate（评估）</strong> | **用户资质预评估 / 资质评估更新** | **准入判断、授信评估** |
| <strong>Organization Customer Clear（清退）</strong> | **机构客户清退 → 资质清退 → 结果通知** | **合作终止** |

**追问**：Purchase（购买/放款）、Redeem（赎回）、Repayment（还款）、Evaluate（评估）这四个流程之间有什么关联？

<strong>四个流程之间的关联：</strong>

- **Purchase 流程中会嵌套调用 Evaluate（资质评估），评估通过后才会进入放款环节**
- **Purchase 成功后产生了额度扣减，后续的 Redeem/Repayment 会触发额度恢复**
- **所有流程都通过统一的工作流引擎驱动，共享快照恢复和补偿机制**

<strong>回答：</strong>

<strong>当前多租户方案分两个层面：</strong>

<strong>1. 构建层面 - Maven Profile：</strong>

**通过 Maven Profile 区分 **`mt`<strong>（美团内部）和 </strong>`bank`<strong>（银行外部）两套部署包，银行下面还细分 </strong>`yinshu_js`<strong>（银树江苏）、</strong>`tonglian_gd`<strong>（通联广达）、</strong>`tencent`<strong> 等。每个 Profile 有独立的配置文件目录（</strong>`src/main/resources/{env}/{tenant}/`<strong>）。</strong>

<strong>2. 配置层面 - 环境隔离：</strong>

<strong>美团环境用 MCC 配置中心，银行环境用 Nacos，数据源配置、Thrift 发布、MQ topic 都不同。</strong>

```text

```

<strong>3. 运行层面 - 机构隔离：</strong>

`**BusinessThreadResourceManager` 中对特定机构（如广达银行 `M03030000`）做了线程池隔离：**

**追问**：新增一家银行机构接入，需要改动哪些地方？当前方案有什么局限性？

<strong>局限性：</strong>

- **当前是同一套代码通过 Profile 编译出不同部署包，不是运行时动态切换。新增银行机构需要加 Profile + 新增配置文件，改动成本中等**
- **线程池隔离是硬编码机构编号，不够灵活。更好的做法是通过配置中心动态配置哪些机构需要独立线程池**
- **业务逻辑上的差异化处理散落在代码中（如 **`TjBankLoanNoticeService`** 是天津银行专用通知），缺乏统一的机构适配层抽象**

<strong>更好的方案： 可以引入机构适配器注册机制，每个机构实现统一的适配器接口，通过机构编码动态路由到对应的适配器实现，新增机构只需要新增一个适配器类 + 配置注册即可。</strong>

## **二、工作流引擎设计**

### <strong>Q4：为什么自研工作流引擎，不用 Activiti/Flowable？</strong>

**考察点**：技术选型能力、Trade-off 思考

<strong>回答：</strong>

<strong>这个选型决策我是认可的，原因有几点：</strong>

**1. 我们的流程是"线性子流程编排"，不需要复杂的分支/并行网关**

**看实际代码，不管是 Purchase、Redeem 还是 Evaluate，流程都是线性链表结构——每个子流程有 **`prevSubProcessConfigId`** 和 **`nextSubProcessConfigId`<strong>，只能从前到后顺序执行。Activiti/Flowable 支持的并行网关、排他网关、事件网关等复杂能力，在我们这里完全用不到。</strong>

**2. 我们需要深度集成业务上下文**

<strong>Activiti 的流程定义（BPMN XML）是独立于业务代码的，但我们的子流程执行需要依赖大量业务对象（</strong>`WorkFlow`<strong>、</strong>`OrganizationOrderProcessSnapshot`<strong> 等），需要和灰度机制、策略模式、监控体系深度耦合。自研引擎可以直接操作这些对象。</strong>

**3. 灰度发布要求流程可差异化配置**

**同一笔 Purchase 订单，根据灰度配置可能走不同的子流程实现（灰度实现 vs 稳定实现）。这在 Activiti 中很难表达——Activiti 的流程定义是静态的，不像我们有 **`OrganizationSubProcessAttribute.implementCode`<strong> 可以动态指定实现。</strong>

**4. 轻量化，避免引入重依赖**

**Activiti 引入后会增加数据库表（28+张）、流程设计器、流程引擎内核等重量级组件，运维和学习成本高。我们自研引擎核心代码就一个 **`AbstractWorkFlowEngine`<strong>（234行），足够清晰可控。</strong>

<strong>Trade-off： 自研的代价是不支持可视化流程编排——当前流程模板是 Java 代码硬编码的（</strong>`LoanFlowTemplate`** 中用 **`static {}`<strong> 块初始化），调整流程需要发版。如果后续需要产品/运营自主配置流程，可能还是需要引入轻量级的流程编排能力。</strong>

---

### <strong>Q5：工作流的执行模型是怎样的？一个 Purchase 流程会经历哪些子步骤？</strong>

**考察点**：对流程编排的理解

<strong>回答：</strong>

<strong>执行模型是典型的"状态机 + 链表驱动"模式：</strong>

<strong>核心有三个状态转换：</strong>

```text
INIT（初始化）→ check()通过 → execute() → SUCCESS/FAIL（终态）→ 流转到下一个子流程
INIT（初始化）→ check()不通过 → 跳过 → 流转到下一个子流程
PROCESSING（处理中）→ compensate() 补偿 → 重新执行或人工介入

```

<strong>Purchase（放款）流程的完整子步骤（以消费类助贷为例）：</strong>

```text
阶段1 - 资信准入（1000系列，executeOrder）
① 合同创建（资质评估前） QUAL_EVAL_CONTRACT_CREATE_COMMON 1002
② 影像传输（资质评估前） QUALIFICATION_EVALUATE_IMAGE_TRANSFER 1003
③ 资质评估 ORGPRODUCT_CASH_QUALS_EVAL_COMM 1004
④ 额度创建 ORGPRODUCT_LIMIT_CREATION 1005

阶段2 - 贷中提额（2000系列）
⑤ 贷中提额申请 ORGPRODUCT_LIMIT_RAISE_PURCHASING 2001

阶段3 - 放款执行（3000系列）
⑥ 额度冻结 COMMON_ORGLIMIT_FREEZE 3001
⑦ 影像传输（购买评估前） PURCHASE_EVALUATE_IMAGE_TRANSFER 3002
⑧ 购买评估 ORGPRODUCT_CASH_PURC_EVAL_COMM 3003
⑨ 放款审批（助贷特有，联合贷无此步骤） PURCHASE_APPROVAL_ORG 3004
⑩ 支付 PURCHASE_PAYMENT_COMMON 3005
⑪ 额度解冻 COMMON_ORGLIMIT_UN_FREEZE 3006
⑫ 额度扣减 COMMON_ORGLIMIT_DEDUCT 3007

阶段4 - 结果通知（4000系列）
⑬ 购买结果通知 PURCHASE_RESULT_NOTIFY 4001
⑭ 机构放款通知（联合贷特有） SELF_BUSS_COMM_LOAN_NOTICE 4002

阶段5 - 贷后提额（5000系列）
⑮ 贷后提额 ORGPRODUCT_LIMIT_RAISE_PURCHASED 5001

```

<strong>值得注意的细节：</strong>

- **助贷比联合贷多了一步放款审批（步骤⑨），因为助贷模式下资金方需要审批才能放款**
- **联合贷比助贷多了一步机构放款通知（步骤⑭），因为联合贷是双方共同放款**
- **executeOrder 使用 千位分段（1000/2000/3000/4000/5000），这样中间插入新步骤不需要修改已有步骤的顺序号，这是个很实用的设计**

### <strong>Q6：工作流执行中断后怎么恢复？快照机制是怎么设计的？</strong>

**考察点**：幂等性、断点续执、状态机设计

<strong>回答：</strong>

**快照是工作流恢复的核心。 每次工作流推进到一个新子流程，都会在数据库中记录一条 **`OrganizationOrderProcessSnapshot`<strong>，保存：</strong>

- `**processOrderId` - 工作流单号**
- `**currentSubProcess` - 当前子流程配置ID**
- `**status` - 当前状态（INIT/SUCCESS/FAIL/PROCESSING）**
- `**isGray` - 是否灰度流程**
- `**customerNo` / `tradeProductNo` - 业务标识**

<strong>恢复流程（</strong>`executeBusiness`<strong> 方法）：</strong>

```java
if (getWorkFlowProcessStatus(workFlow).isInit()) {
// 首次执行：初始化快照 → 匹配子流程 → 执行
beginWorkFlow(workFlow);
snapshot = orderProcessSnapshotService.selectByOrderId(...);
executeWorkFlowInit(workFlow, snapshot);

} else if (getWorkFlowProcessStatus(workFlow).isProcessing()) {
// 恢复执行：根据快照状态决定动作
snapshot = orderProcessSnapshotService.selectByOrderId(...);
snapshotExecute(workFlow, snapshot);
}


```

`**snapshotExecute` 的四种状态处理：**


| **快照状态** | **处理方式** | **场景** |
|---|---|---|
| `**SUCCESS` / `FAIL**` | **当前子流程已终态，驱动流转到下一个子流程** | **MQ回调触发流程继续推进** |
| `**INIT**` | **当前子流程尚未执行，执行check→execute** | **首次进入或跳过后的第一个有效子流程** |
| `**PROCESSING**` | **当前子流程卡在处理中，执行compensate补偿** | **超时检测或人工触发重试** |

<strong>幂等保证：</strong>

- <strong>快照状态更新使用 CAS（Compare And Set）：</strong>`updateByStatus(snapshot, lastSubProcessId, lastSubProcessStatus)`** 只在旧状态匹配时才更新，防止并发重复推进**
- `**findNextExecuteSubProcess` 递归查找时会校验每个子流程的 `check()` 条件，不满足就跳过**

**实际场景举例： 用户放款卡在步骤⑥「额度冻结」（状态为 PROCESSING），可能是外部银行接口超时。系统通过定时任务扫描 PROCESSING 状态的快照，触发 **`compensate()`<strong> 进行补偿处理（如重新冻结或回滚），然后根据补偿结果推进到下一步。</strong>

### **Q7：每个子流程都有 **`compensate()`<strong> 方法，补偿机制是怎么触发的？如果补偿本身也失败了呢？</strong>

**考察点**：分布式事务、补偿事务模式、异常兜底策略

<strong>回答：</strong>

**补偿触发条件： 当 MQ 消费或定时任务驱动引擎执行时，发现当前子流程快照状态为 **`PROCESSING`<strong>，说明这个子流程之前的执行卡住了（可能是外部接口超时、系统宕机等），此时会调用 </strong>`matchProcess(workFlow).compensate(getCurrentNode(snapshot))`<strong>。</strong>

<strong>补偿的设计思路是"正向补偿"而非"逆向回滚"：</strong>

- `**compensate()` 接收一个 `Node` 对象（子流程执行的实体记录），根据当前节点的状态决定是重试还是标记失败**
- **补偿方法内部通常会调用外部接口查询真实状态，而不是假设失败了去执行回滚。比如支付补偿会先查询支付网关的真实支付状态**

<strong>补偿失败的处理策略：</strong>

<strong>我们的设计采用了分层兜底机制：</strong>

1. <strong>第一层 - 子流程自身补偿：</strong>`compensate()`** 方法内部有重试逻辑**
2. <strong>第二层 - 监控告警兜底：</strong>`UnknownFinalStatusOrderMonitor`** 和 **`Withdraw24hUnknownFinalStatusMonitor`** 会扫描长时间处于未知状态的订单，通过大象（DX）发送告警**
3. **第三层 - 人工介入：告警后由运营/开发人工排查，必要时通过后台工具手动推进快照状态**

**追问**：有没有遇到过补偿链路过长的情况？如何避免补偿风暴？

<strong>关于补偿风暴： 我们通过快照状态 CAS 更新来防止并发补偿。即使多个线程同时触发补偿，只有一个能成功推进状态。另外，</strong>`BusinessThreadResourceManager`<strong> 通过线程池隔离限制了单机构的并发量，也降低了风暴概率。</strong>

<strong>更好的见解（补充）： 当前补偿是被动触发的（定时任务扫描），可以增加主动超时机制——在每个子流程开始执行时记录开始时间，超过阈值（如30分钟）自动触发补偿，不用等定时任务扫到。另外可以引入死信队列，补偿超过N次失败的消息转入死信队列，由专门的处理程序分析处理。</strong>

### <strong>Q8：</strong>`OrganizationOrderProcessSnapshot`<strong> 记录了流程快照，快照数据存在哪里？会不会有数据量过大的问题？</strong>

**考察点**：数据存储设计、数据生命周期管理

<strong>回答：</strong>

<strong>快照存储在 MySQL 数据库中（通过 MyBatis + Zebra 中间件），每条工作流订单对应一条快照记录。快照只记录当前子流程的状态，而不是所有子流程的历史记录——这是一个很关键的设计：快照是"指针"而非"日志"。</strong>

<strong>数据量分析：</strong>

- <strong>每笔放款订单只有 1 条快照记录（currentSubProcess 指向当前步骤）</strong>
- **假设日均放款 10 万笔，一个月就是 300 万条，一年 3600 万条**
- **单条记录字段不多（ID、订单号、子流程ID、状态、时间等），约 200 字节**

<strong>数据量可控的原因：</strong>

1. <strong>快照不是追加写入，而是更新同一条记录（</strong>`updateByStatus`<strong>）</strong>
2. **工作流完成后，快照记录可以归档或清理**

<strong>潜在风险和优化方案：</strong>

- **如果需要审计追溯"每一步经历了什么"，当前单条快照是不够的。代码中有 **`Node`<strong> 实体（</strong>`nodeEntityAppService`<strong>）记录每个子流程的执行详情，这部分才是完整的执行日志</strong>
- **对于历史数据，可以做冷热分离：3个月内的快照在 MySQL，超过3个月的归档到 Hive 或 TiDB**
- **当前缺少快照数据的 TTL（生命周期管理），建议增加定期清理策略**

## **三、额度管理体系**

### <strong>Q9：额度管理涉及创建、提额、降额、冻结、解冻、扣减、关闭等操作，这些操作的并发安全怎么保证？</strong>

**考察点**：并发控制、分布式锁、数据库行锁

<strong>回答：</strong>

<strong>这是金融系统最关键的问题之一。我们的并发控制分三层：</strong>

<strong>第一层 - 数据库行锁（最核心）：</strong>

**额度扣减/冻结操作使用数据库的 **`UPDATE ... SET amount = amount - ? WHERE id = ? AND amount >= ?`<strong> 语句，通过 WHERE 条件实现乐观锁。Zebra 数据源中间件会将同一行的 UPDATE 序列化执行，天然避免了并发超扣。</strong>

<strong>第二层 - 业务层状态机：</strong>

<strong>额度的状态流转有严格的约束：</strong>`可用 → 冻结 → 扣减`** 或 **`可用 → 冻结 → 解冻回可用`<strong>。每个状态转换都会更新快照并使用 CAS 保证。比如冻结操作会在快照中记录冻结金额，解冻时会校验冻结金额是否匹配。</strong>

<strong>第三层 - 线程池隔离：</strong>

`**BusinessThreadResourceManager` 对特定机构做了线程池隔离，限制了同一机构同时执行的额度操作并发数，从源头降低了竞争概率。**

<strong>追问：如果两个请求同时扣减同一笔额度，你们的方案是什么？</strong>

<strong>两个请求同时扣减同一笔额度的场景：</strong>

- **请求A 和 请求B 同时执行 `UPDATE SET amount = amount - 100 WHERE id = 123 AND amount >= 100**`
- **Zebra/MySQL 会串行化执行，先到的成功，后到的因为 **`amount >= 100`** 条件不满足而影响行数为0**
- **业务层检测到影响行数为0，抛出 **`OrgLimitRuleException`<strong>，触发业务异常处理</strong>

<strong>更好的见解（补充）： 可以考虑引入 Redis 分布式锁 作为第一层防护（</strong>`SETNX orderId + 过期时间`<strong>），在数据库操作前先获取锁，减少数据库层面的竞争压力。但对于当前业务量级，数据库行锁已经够用，Redis 锁反而增加了复杂度和故障点。</strong>

### <strong>Q10：额度创建使用了策略模式（</strong>`ProductLimitCreateStrategy`<strong>），区分了 NormalProduct、FuseProduct、SwitchProduct，这三种策略的差异是什么？</strong>

**考察点**：策略模式实战、业务抽象能力

<strong>回答：</strong>

**三种策略对应三种产品类型，通过 **`ProductLimitCreateStrategyFactory`<strong> 注册和管理：</strong>

<strong>1. NormalProduct（普通产品）：</strong>

- **最简单的场景，直接为客户创建一条新的额度记录**
- **额度金额由 **`OrgProductLimitConfig`** 配置决定**
- <strong>实现：</strong>`NormalProductLimitCreateStrategy`** → 调用 **`OrganizationLimitAccountV2Service`** 创建额度账户**

<strong>2. FuseProduct（融合产品）：</strong>

- **融合产品是指一个产品同时对接多家资金方**
- **创建额度时需要关联多个产品/机构信息**
- **额度不是独立创建的，而是继承或关联已有产品的额度**
- <strong>实现：</strong>`FuseProductLimitCreateStrategy`** → 使用关联产品/机构的额度信息进行创建**

<strong>3. SwitchProduct（切换产品）：</strong>

- **切换产品是指用户从一家资金方切换到另一家**
- <strong>创建额度前需要先清空原有关联额度（旧的额度记录置为失效）</strong>
- **确保同一时间只有一个有效额度**
- <strong>实现：</strong>`SwitchProductLimitCreateStrategy`** → 先清空关联额度，再创建新额度**

<strong>策略模式的使用方式：</strong>

```text
// 策略注册（通常在 Spring @PostConstruct 中）
ProductLimitCreateStrategyFactory.register(ProductLimitType.NORMAL, normalStrategy);
ProductLimitCreateStrategyFactory.register(ProductLimitType.FUSE, fuseStrategy);
ProductLimitCreateStrategyFactory.register(ProductLimitType.SWITCH, switchStrategy);

// 策略获取和使用ProductLimitCreateStrategy strategy = ProductLimitCreateStrategyFactory.getStrategy(limitType);
strategy.createLimit(request);


```

**每种策略都有对应的抽象基类（如 **`AbstractProductLimitCreateStrategy`<strong>）封装通用逻辑，具体策略只实现差异部分，这是典型的 模板方法 + 策略模式 组合。</strong>

### <strong>Q11：提额流程涉及异步审批，</strong>`RaiseLimitRequestProcessStrategy`** 和 **`RaiseLimitResultProcessStrategy`<strong> 分别处理什么？两者如何协作？</strong>

**考察点**：异步流程编排、回调机制

<strong>回答：</strong>

<strong>提额流程是"异步请求-回调"模式：</strong>

```text
用户触发提额
↓
RaiseLimitRequestProcessStrategy（提额请求策略）
↓ 判断是否需要风险审批
├── 不需要 → 直接调整额度
└── 需要 → 发送提额请求到风控系统（通过MQ或RPC）
↓
风控系统异步审批
↓
MQ回调 / RPC回调
↓
RaiseLimitResultProcessStrategy（提额结果策略）
├── 审批通过 → 执行额度调整
└── 审批拒绝 → 记录拒绝原因，通知用户

```

<strong>两个策略的职责划分：</strong>


| **策略** | **职责** | **触发时机** |
|---|---|---|
| `**RaiseLimitRequestProcessStrategy**` | **判断是否需要风控审批、构造审批请求、发送到外部系统** | **工作流执行到提额子流程时** |
| `**RaiseLimitResultProcessStrategy**` | **接收审批结果、执行额度变更、记录审计日志** | **MQ消费审批结果消息时** |

<strong>风险链路压单（关键细节）：</strong>

`**AbstractProductLimitRaiseRequestProcessStrategy` 中有 “风险链路压单” 的判断逻辑：**

- **如果用户在风控审批过程中又有新的放款请求，需要判断提额是否已经完成**
- **如果提额还在审批中（PROCESSING），放款请求可能会被压住等待，等提额结果出来后再决定放款额度**
- **这是通过检查 **`RaiseLimitMaintenanceRecord`<strong>（提额维护记录）的状态来实现的</strong>

`**RaiseLimitHoldStrategyManageService` 负责管理这个压单策略。**

## <strong>四、消息队列与异步处理(考察中间件使用）</strong>

### **Q12：Mafka 消费者使用了统一的 **`MqConsumer<T>`<strong> 接口设计，</strong>`fitForTopic`<strong>、</strong>`fitForBizScene`<strong> 等方法的作用是什么？</strong>

**考察点：消息路由设计、消费者抽象**

<strong>回答：</strong>

`**MqConsumer<T>` 是一个模板化的消费者接口，目的是让不同业务场景的消费者复用同一套消费框架。每个方法都有明确的设计意图：**

```text
public interface MqConsumer<T> {
// 1. 反序列化：把MQ消息字符串转换为业务对象
T constructBusinessModel(String message);

// 2. Topic路由：判断这个消费者是否处理该Topic的消息boolean fitForTopic(String topic);

// 3. 场景路由：同一个Topic下可能有多种业务场景，进一步过滤boolean fitForBizScene(OrganizationExchangeMQMessageBizScene scene);

// 4. 业务过滤：消息反序列化后，根据业务逻辑判断是否需要处理boolean isNeedConsume(T businessModel);

// 5. 核心消费逻辑
MQMessageStatus consume(T businessModel);

// 6. 幂等记录：创建消费记录，重复消费时抛出 RepeatConsumeException
MqConsumeRecord createConsumerRecord(String message, T businessModel);
}


```

<strong>为什么需要这么多层过滤？</strong>

**因为我们使用了共享消费者组的架构模式——多个消费者监听同一个 Topic，通过 **`fitForTopic`** + **`fitForBizScene`** + **`isNeedConsume`<strong> 三层过滤来实现消息路由：</strong>

- `**fitForTopic`：物理层路由——消息刚进来，判断这个Topic是不是我该处理的**
- `**fitForBizScene`：业务场景路由——同一个 Topic 的消息可能包含支付结果、风控结果等多种场景**
- `**isNeedConsume`：数据层路由——消息内容反序列化后，根据具体字段判断（比如只处理特定机构的消息）**

<strong>好处：</strong>

1. <strong>新增消费者只需要实现接口 + 注册，不需要改消费框架代码（开闭原则）</strong>
2. **三层过滤职责清晰，每层只关注自己的判断逻辑**
3. <strong>统一的消费记录管理（</strong>`createConsumerRecord`<strong>）保证幂等</strong>

### <strong>Q13：消息发送使用了 guava-retrying 做重试，重试策略是怎么配置的？重试期间消息顺序怎么保证？</strong>

**考察点：重试机制、消息有序性**

<strong>回答：</strong>

<strong>重试策略：</strong>

`**AbstractMafkaProducer` 使用了 guava-retrying 框架：**

```text
public void sendMsgToMq(T object) {
try {
(() -> sendMsgToMafka(object, getProducerProcessor()));
} catch (Exception e) {
throw new MessageSendFailureException("消息发送失败", e);
}
}


```

<strong>guava-retrying 默认配置通常是：</strong>

- **重试次数：3 次**
- <strong>重试间隔：指数退避（如 1s → 2s → 4s）</strong>
- <strong>可配置重试异常类型（如只对网络异常重试，不对业务异常重试）</strong>

<strong>消息顺序保证：</strong>

<strong>坦白说，我们的架构没有严格保证消息的全局顺序性，但在业务层面做了以下处理：</strong>

1. **Mafka 本身保证 Partition 内有序——同一 Key 的消息会路由到同一 Partition，保证消费顺序**
2. **业务层通过状态机保证——即使消息乱序到达，工作流引擎通过快照状态（INIT/SUCCESS/FAIL/PROCESSING）来判断当前应该执行什么操作。比如先收到了"支付成功"的回调，但额度还在冻结状态，引擎会正确处理**
3. <strong>幂等设计——</strong>`createConsumerRecord`** 保证重复消费不会产生副作用**

<strong>补充更好的见解： 如果需要严格的全局有序（如同一笔订单的多个事件必须按顺序处理），可以引入 分区有序 策略——以订单号为 Key 发送消息，保证同一订单的所有事件在同一个 Partition 内有序消费。同时消费端使用单线程消费同一 Partition，避免并发消费导致的乱序。</strong>

### <strong>Q14：</strong>`createConsumerRecord`** 方法会抛出 **`RepeatConsumeException`<strong>，重复消费是怎么检测的？幂等方案是什么？</strong>

**考察点：消息幂等性设计**

<strong>回答：</strong>

<strong>幂等检测方案：</strong>

`**MqConsumeRecord` 是幂等的核心。每次消费前会调用 `createConsumerRecord()`，这个方法内部会检查数据库中是否已经存在该消息的消费记录：**

```text
// 伪代码public MqConsumeRecord createConsumerRecord(String message, T businessModel) throws RepeatConsumeException {
String messageId = extractMessageId(message); // 从消息中提取唯一IDMqConsumeRecord record = mqConsumeRecordMapper.selectByMessageId(messageId);
if (record != null) {
if (record.getStatus() == SUCCESS) {
throw new RepeatConsumeException("消息已消费"); // 已成功消费过，跳过
}
// 如果是 FAILED 状态，允许重新消费（可能是之前的消费失败了）
}
// 插入新的消费记录（INSERT，利用唯一索引保证并发安全）return insertRecord(messageId, businessModel);
}

```

<strong>幂等的多层保障：</strong>


| **层级** | **机制** | **说明** |
|---|---|---|
| **MQ层** | **消息唯一ID** | **消息体中包含 messageId** |
| **消费层** | **消费记录表** | **数据库唯一索引防并发插入** |
| **业务层** | **状态机** | **工作流快照 CAS 更新** |
| **数据层** | **乐观锁** | **额度操作的 WHERE 条件检查** |

**消费记录的状态流转： **`PROCESSING → SUCCESS / FAILED`<strong>，只有 FAILED 状态的消息才允许被重新消费，SUCCESS 的直接抛 </strong>`RepeatConsumeException`<strong> 跳过。</strong>

## <strong>五、灰度发布机制（考察技术亮点）</strong>

### <strong>Q15：灰度分流使用了 CRC32 算法，为什么选择这个算法？和取模、一致性哈希相比有什么优劣？</strong>

**考察点**：灰度策略选型

<strong>回答：</strong>

<strong>选型对比：</strong>


| **方案** | **优点** | **缺点** | **适用场景** |
|---|---|---|---|
| **简单取模 `hash(key) % N**` | **实现极简** | <strong>key 微小变化导致结果剧变（雪崩效应）</strong> | **不推荐** |
| **CRC32** | **分布均匀、计算快速、相同输入始终相同输出** | **不支持节点动态增减** | :check_mark_button: **我们的场景** |
| **一致性哈希** | **支持节点增减时最小化迁移** | **实现复杂，适合分片场景** | **分布式缓存、分库分表** |

<strong>选择 CRC32 的原因：</strong>

1. **我们的灰度是"固定比例分流"，不是"分片"——我们只需要把 key 映射到 [0, 10000) 的空间，然后和灰度比例比较，不需要一致性哈希的虚拟节点机制**
2. **确定性——同一个 grayKey（如订单号）每次计算结果一致，保证同一笔订单始终走同一条流程**
3. **分布均匀——CRC32 的散列均匀性很好，不会出现"扎堆"现象**
4. <strong>性能——</strong>`CRC32.update()`** 计算极快，纳秒级别**

```text
public boolean isShutGray(String grayKey) {
CRC32 crc32 = new CRC32();
crc32.update(grayKey.getBytes());
long crc32Value = crc32.getValue();
BigDecimal keyValue = new BigDecimal(crc32Value % 10000)
.divide(new BigDecimal(10000), 4, RoundingMode.HALF_UP);
return keyValue.compareTo(this.grayRatio) < 0; // grayRatio = 0.1 表示10%
}


```

**补充更好的见解： CRC32 的一个潜在问题是 grayKey 的选择很关键——如果用时间戳作为 key，灰度效果会很好（均匀分布），但如果用递增的订单号，CRC32 在订单号连续递增时可能产生局部聚集。更好的做法是用 **`订单号 + 机构编码 + 产品编码`<strong> 的组合作为 grayKey，保证多维度的均匀分布。</strong>

### <strong>Q17：为什么引入 Aviator 表达式引擎？解决了什么业务问题？</strong>

**考察点：规则引擎选型、业务配置化**

<strong>回答：</strong>

<strong>解决的业务问题：</strong>

<strong>在评估流程中，我们需要根据各种条件动态判断用户的准入规则。比如：</strong>

<strong>为什么选 Aviator 而不是其他方案：</strong>


| **方案** | **评估** |
|---|---|
| **Aviator** | :check_mark_button: **轻量级、纯 Java、性能好、支持自定义函数、美团内部广泛使用** |
| <strong>QLExpress（阿里）</strong> | **功能类似，但依赖较重** |
| **Groovy** | <strong>太重，安全性差（可执行任意代码）</strong> |

```text
public class AviatorUtils {
public static Object parse(String expression, Map<String, Object> context) {
	return AviatorEvaluator.execute(expression, context, true);
	}
}

```

**核心使用方式：把业务变量组装成 **`Map<String, Object>`<strong>，传入 Aviator 表达式执行。</strong>

### <strong>Aviator 表达式是什么？</strong>

<strong>Aviator 是一个轻量级的 Java 表达式求值引擎，可以让你把"表达式字符串"当成代码来执行。简单说就是：把一段字符串变成可执行的逻辑。</strong>

#### **核心用法**

```java
import com.googlecode.aviator.AviatorEvaluator;

// 1. 基本运算
AviatorEvaluator.execute("1 + 2  3");            // 返回 7

// 2. 带变量的表达式
Map<String, Object> env = new HashMap<>();
env.put("amount", 5000);
env.put("rate", 0.05);
AviatorEvaluator.execute("amount  rate", env);    // 返回 250.0

// 3. 字符串拼接
env.put("name", "张三");
AviatorEvaluator.execute("'您好，' + name + '先生'", env);  // 返回 "您好，张三先生"

// 4. 三元表达式
env.put("age", 25);
AviatorEvaluator.execute("age > 18 ? '成年' : '未成年'", env);  // 返回 "成年"

// 5. 调用方法
AviatorEvaluator.execute("string.contains('hello world', 'hello')");  // 返回 true
```

**第三个参数 **`true`** 表示开启缓存，相同表达式不会重复编译，性能更好（项目中 **`AviatorUtils`<strong> 就用了这个）：</strong>

```text
return AviatorEvaluator.execute(expresion, context, true);

```

#### **在本项目中的场景**

<strong>在你们的系统中，Aviator 用于合同模板的动态变量填充。核心逻辑是：</strong>

1. <strong>数据库存储了合同变量配置（</strong>`VariableConfig`<strong> 表），其中有字段：</strong>

- `**variableName`：变量名（如 `合同金额`）**
- `**expresion`：Aviator 表达式（如 `loanAmount + feeAmount`）**
- `**priority`：优先级（高优先级的变量可以引用低优先级的计算结果）**
- `**variableType`：变量类型（`BASE` 静态值 / `ROUTE` 表达式计算 / `CUSTOMER` 自定义代码）**

2. <strong>执行流程：先解析基础变量（</strong>`BASE`<strong>）和自定义变量（</strong>`CUSTOMER`<strong>），再把它们作为上下文传入 Aviator，执行 </strong>`ROUTE`<strong> 类型变量的表达式，实现变量间的依赖计算。</strong>

<strong>举个实际例子：假设合同需要"应收总额"字段，数据库配置的表达式可以是：</strong>


| **优先级** | **变量名** | **类型** | **表达式** |
|---|---|---|---|
| **1** | **贷款金额** | **BASE** | `**50000**` |
| **1** | **服务费** | **CUSTOMER** | <strong>(代码查数据库)</strong> |
| **2** | **应收总额** | **ROUTE** | `**loanAmount + serviceFee**` |

`**ROUTE` 类型的 `应收总额` 会通过 Aviator 表达式 `loanAmount + serviceFee` 引用前面已解析好的变量值，动态计算出结果。**

#### <strong>为什么要用它？</strong>


| **优势** | **说明** |
|---|---|
| **动态可配** | **表达式存数据库，改逻辑不用发版** |
| **变量引用** | **表达式可以引用其他已解析的变量** |
| **轻量高性能** | **比 Groovy/Jexl 等轻量，支持编译缓存** |
| **语法简单** | **类 Java 语法，学习成本低** |

<strong>简单总结：Aviator 就是让你们能在数据库里配置一段"小公式"，运行时动态计算结果，避免每次改合同计算逻辑都要改代码发版。</strong>

### <strong>Q18：Variable 体系（CustomerBaseVariable、OcrBaseVariable、LocationBaseVariable 等）是怎么和表达式引擎配合的？如何新增一个变量？</strong>

**考察点：扩展性设计、策略模式的灵活运用**

<strong>回答：</strong>

<strong>整体架构是一个三层结构：</strong>

```text
业务变量数据（VariableImplement实现类）
↓ 采集
VariableParseHandler（变量解析处理器）
↓ 组装上下文
VariableContext（变量上下文 Map）
↓ 传入表达式
AviatorUtils.parse(expression, context)
↓ 输出
true/false 或具体值


```

<strong>第一层 - VariableImplement（变量实现）：</strong>

<strong>每个实现类负责从不同数据源获取变量值：</strong>

- `**CustomerBaseVariable` → 从客户服务获取客户基本信息（姓名、年龄、性别等）**
- `**OcrBaseVariable` → 从 OCR 服务获取证件识别结果**
- `**BankCardBaseVariable` → 从银行卡服务获取绑卡信息**
- `**LocationBaseVariable` → 从位置服务获取用户地理位置**
- `**UserMobileVariable` → 获取用户手机号**
- `**OrganizationNameVariable` → 获取机构名称**

<strong>第二层 - VariableParseHandler（变量解析）：</strong>

<strong>按优先级分三种类型：</strong>

- `**BASE`（基础变量）→ 直接返回默认值或从缓存获取**
- `**ROUTE`（路由变量）→ 通过 Aviator 表达式计算决定路由目标**
- `**CUSTOMER`（自定义变量）→ 调用对应的 VariableImplement 实现类获取**

<strong>第三层 - Aviator 表达式执行：</strong>

**最终把所有变量组装成 **`VariableContext`<strong>（本质是 </strong>`Map<String, Object>`<strong>），传入 Aviator 执行表达式。</strong>

<strong>新增一个变量的步骤：</strong>

1. **创建一个类实现 **`VariableImplement`** 接口，实现变量获取逻辑**
2. **在 **`VariableImplementFactory`** 中注册**
3. **在数据库中配置变量名称和对应的实现类编码**
4. **在评估规则表达式中使用新变量**

<strong>整个过程不需要改核心框架代码，符合开闭原则。</strong>

- <strong>区分可降级操作（通知类，失败可异步重试）和不可降级操作（支付类，必须等待成功）</strong>

## <strong>九、数据库与持久化（考察底层基础）</strong>

### <strong>Q23：项目用了 Zebra（美团数据库中间件）+ MyBatis，有没有遇到过分库分表的场景？</strong>

**考察点：数据库中间件理解、分片策略**

<strong>回答：</strong>

<strong>Zebra 是美团自研的数据库中间件，类似于 ShardingSphere，提供分库分表、读写分离、动态数据源等能力。</strong>

<strong>当前项目的数据库使用场景：</strong>

1. **业务数据表：快照表、子流程配置表、额度表等，通过 Zebra 管理数据源**
2. **配置数据表：灰度配置表、流程配置表等，变更频率低**
3. **记录表：消费记录表、调整记录表等，数据量增长快**

**分库分表策略： 项目中使用了 **`LeafIdFactory`<strong>（美团 Leaf 分布式ID）生成全局唯一 ID，这是分库分表的前提条件。</strong>`LeafIdFactory`<strong> 生成的 ID 中包含时间戳和机器标识，可以直接用于按 ID 取模分表。</strong>

<strong>使用 Zebra 的好处：</strong>

- <strong>业务代码不需要关心分库分表逻辑（SQL 写法和单库一样）</strong>
- <strong>支持动态扩容（新增分片只需要在 Zebra 配置中调整）</strong>
- **读写分离对业务透明**

### **Q24：分布式 ID 用了 **`LeafIdFactory`<strong>，为什么不用数据库自增 ID？</strong>

> **考察点：分布式ID选型、性能考量**

<strong>回答：</strong>

<strong>数据库自增 ID 的局限：</strong>

1. **单点问题：自增 ID 依赖单个 MySQL 实例，分库分表后无法保证全局唯一**
2. <strong>暴露数据量：自增 ID 递增可被推算出业务量（安全风险）</strong>
3. **性能瓶颈：高并发写入时自增锁成为瓶颈**

<strong>Leaf 方案： 美团 Leaf 采用号段模式：</strong>

- **Zebra/DB 中存储号段（如 **`max_id=10000, step=1000`<strong>）</strong>
- **应用启动时从 DB 获取一个号段 [10001, 11000]，缓存在内存中**
- **内存中的 ID 用完后，再去 DB 获取下一个号段**
- **双 buffer 优化：当前号段用到 10% 时，异步加载下一个号段，确保无缝切换**

<strong>优势：</strong>

- <strong>全局唯一、趋势递增（利于 B+ 树索引性能）</strong>
- <strong>高性能（内存分配，不依赖数据库实时查询）</strong>
- <strong>高可用（号段用完后才查 DB，DB 短暂不可用不影响 ID 生成）</strong>

```text
}
}

```

<strong>项目中大量使用枚举，这是很好的实践。枚举的设计原则：</strong>

```text
public enum ProcessStatus {
INIT("init", "初始化") {

public boolean isFinish() { return false; }
},
SUCCESS("success", "成功") {
```

### <strong>Q30-1：亲缘性线程池（KeyAffinityExecutor）</strong>

#### **核心概念**

<strong>亲缘性线程池是一种按 Key 分片路由到固定线程的线程池策略。它的核心思想是：相同 Key 的任务永远由同一个线程执行，从而在不需要锁的情况下实现"同一业务实体的操作串行化"。</strong>

#### **与普通线程池的对比**


| **对比维度** | <strong>普通线程池 (</strong>`ThreadPoolExecutor`<strong>)</strong> | <strong>亲缘性线程池 (</strong>`KeyAffinityExecutor`<strong>)</strong> |
|---|---|---|
| **任务分配** | **任务提交到共享队列，任意空闲线程取走执行** | **按 Key 哈希路由到固定线程的专属队列** |
| **同一实体的执行顺序** | :cross_mark: **不保证，可能被不同线程并行处理** | :check_mark_button: <strong>保证串行执行（同一 Key → 同一线程）</strong> |
| **并发安全** | **需要额外的锁或 CAS 来保护共享数据** | **不需要锁，天然串行，消除竞争** |
| **线程利用率** | **高，所有线程共享所有任务** | <strong>略低，可能出现某些线程忙、某些线程空闲（Key 倾斜）</strong> |

#### **用图来理解**

<strong>普通线程池：</strong>

```text
 线程1 ◄──┐
  线程2 ◄──┼── 共享队列 [任务A, 任务B, 任务C, ...]
  线程3 ◄──┘
  
  问题：订单A的 add 和 delete 可能被线程1和线程2同时执行


```

<strong>亲缘性线程池：</strong>

```text
订单A (flowId%10=3) ──→ 线程3的专属队列 [A_add, A_delete, ...]
订单B (flowId%10=7) ──→ 线程7的专属队列 [B_add, B_delete, ...]
订单C (flowId%10=3) ──→ 线程3的专属队列 [A_add, A_delete, C_add, ...]
                                              ↑
                              同一线程串行执行，天然无竞争


```

#### **项目中的实际使用**

**在 **`AsyncProcessingBillExecutor`<strong> 中，处理中单据的 add/delete 需要保证顺序：</strong>

```java
// 创建 10 个线程，每个线程有容量为 2000 的专属队列private static final Integer DIVIDEND = 10;
private static final KeyAffinityExecutor keyAffinityExecutor =
    KeyAffinityExecutor.newSerializingExecutor(10, 2000, "key-affinity-thread-pool—%d");

// 按订单ID路由到固定线程public void addProcessingFlow(ProcessingFlow processingFlow) {
    keyAffinityExecutor.executeEx(
        Math.toIntExact(processingFlow.getFlowId() % DIVIDEND),  // Key = flowId % 10
        () -> processingBillService.addProcessingFlow(processingFlow)
    );
}

```

<strong>场景：一笔订单的生命周期中可能快速触发多次操作：</strong>

```text
时刻1: 创建订单 → addProcessingFlow(订单A)
时刻2: 流程失败 → deleteProcessingFlow(订单A)  
时刻3: 补偿触发 → addProcessingFlow(订单A)  (重新进入)


```

<strong>如果用普通线程池，时刻1的 add 和时刻2的 delete 可能并发执行 → 处理中记录可能删除后又查不到，或重复插入。用亲缘性线程池，这三个操作都在线程3上串行执行，时序严格保证。</strong>

#### <strong>为什么不用锁？</strong>


| **方案** | **问题** |
|---|---|
| `**synchronized**` | **只能单 JVM 内生效，集群部署无效** |
| **数据库锁** | **每次操作都锁行，性能差，增加数据库压力** |
| **Redis 分布式锁** | **引入外部依赖，网络开销大，锁续期复杂** |
| **亲缘性线程池** | :check_mark_button: **无锁、无外部依赖、物理隔离、性能最优** |

#### **适用场景与局限**

<strong>适合的场景：</strong>

- <strong>同一实体的操作需要串行化（如订单状态变更、账户余额操作）</strong>
- **操作频率高，加锁开销不可接受**
- **单 JVM 内的并发控制即可**

<strong>不适合的场景：</strong>

- **Key 分布不均匀（如大量订单的 flowId 都集中落在同一线程），导致线程倾斜**
- <strong>需要跨 JVM 的串行化（需要配合 Redis 分布式锁）</strong>
- <strong>任务之间完全独立，不需要串行化（用普通线程池更高效）</strong>

#### **一句话总结**

<strong>亲缘性线程池通过"Key → 固定线程"的映射关系，在不加锁的前提下实现了同一业务实体的操作串行化，用空间换安全，是高并发场景下替代分布式锁的轻量级方案。</strong>

- `**createIfNotExist()` 模式保证子流程幂等（先查后建，存在则跳过）**
- <strong>补偿任务用同步执行（</strong>`isSync=true`<strong>），避免异步补偿雪崩</strong>

<strong>解决的问题：</strong>

<strong>同一笔订单在短时间内可能触发多次 add/delete 操作（如创建订单时 add，流程完成时 delete）。如果不做控制，可能出现：</strong>

- **add 还没执行完，delete 已经执行 → 处理中记录丢失**
- **并发 add 导致重复数据**

**原理：创建 10 个线程，按 **`flowId % 10`<strong> 路由到固定线程，保证同一订单的操作在同一线程串行执行，天然避免并发问题。队列容量 2000 提供缓冲。</strong>

<strong>为什么不用锁：加锁会阻塞业务线程，且锁粒度不好控制。亲缘性线程池通过"物理隔离"避免竞争，性能更好。</strong>

### <strong>Q33：【优化题】当前灰度方案基于 CRC32，如果需要按用户 ID 的百分比灰度（如 10% 的用户走新流程），同时保证同一用户多次请求灰度结果一致，你会怎么实现？</strong>

> **考察点：一致性哈希、灰度策略深度理解**

<strong>回答：</strong>

<strong>这个需求恰好就是当前 CRC32 方案能很好解决的场景。</strong>

<strong>实现方案：</strong>

```text
public boolean isGrayByUserId(String userId, BigDecimal grayRatio) {
// 1. 使用 CRC32 计算用户 ID 的哈希值CRC32 crc32 = new CRC32();
crc32.update(userId.getBytes());
long hash = crc32.getValue();

// 2. 映射到 [0, 10000) 的空间int bucket = (int) (hash % 10000);

// 3. 和灰度比例比较（grayRatio = 0.1 表示 10%）int threshold = grayRatio.multiply(new BigDecimal(10000)).intValue();

return bucket < threshold;
}


```

<strong>一致性保证：</strong>

- **CRC32 是确定性算法——同一个 userId 永远映射到同一个 bucket**
- **无论调用多少次、在哪个服务实例上调用，结果都一样**
- **用户不会出现"第一次走灰度、第二次走稳定"的来回切换问题**

<strong>均匀性分析：</strong>

- **CRC32 的输出空间是 [0, 2^32 - 1]，取模 10000 后的分布非常均匀**
- **10000 个用户中，大约 1000 个会命中 10% 灰度**
- <strong>不会出现按用户 ID 顺序导致的聚集（因为 CRC32 打散了顺序性）</strong>

<strong>和一致性哈希的区别： 一致性哈希解决的是"节点增减时最小化数据迁移"的问题，而灰度场景不需要节点增减——灰度比例是固定的。CRC32 完全够用，且实现更简单。</strong>

<strong>更完善的方案（补充）： 如果需要支持多维度组合灰度（如"10% 的北京用户"或"所有VIP用户"），可以引入标签匹配机制：</strong>

- <strong>用户打标签（</strong>`user_tags`<strong> 表）</strong>
- **灰度规则配置为标签表达式（如 **`city=北京 AND vip=true`<strong>）</strong>
- **表达式求值决定是否命中灰度**

## **贡献一：自研工作流引擎**


| **面试官提问点** | **回答要点** |
|---|---|
| <strong>为什么自研？不用 Activiti/Flowable？</strong> | **① 我们的流程是线性链表编排，不需要并行网关等复杂能力；② 需要和灰度、策略模式深度耦合，Activiti 的 BPMN 定义是静态的，无法表达灰度差异化实现；③ 避免引入28+张数据库表等重依赖，核心引擎234行代码足够清晰** |
| <strong>工作流执行模型是怎样的？</strong> | **状态机 + 链表驱动，每个子流程有 **`prevSubProcessConfigId`** 和 **`nextSubProcessConfigId`** 形成链表，三种状态转换：INIT→execute→SUCCESS/FAIL→流转下一步；INIT→check不通过→跳过下一步；PROCESSING→compensate补偿** |
| <strong>断点续执怎么实现的？</strong> | **每次推进到新子流程时写入 **`OrganizationOrderProcessSnapshot`** 快照记录，记录当前子流程ID和状态。引擎执行时先查快照：INIT状态走首次执行逻辑，PROCESSING状态走补偿逻辑，SUCCESS/FAIL状态驱动流转到下一步** |
| <strong>幂等怎么保证？</strong> | **① 快照更新用 **`updateByStatus(snapshot, lastSubProcessId, lastSubProcessStatus)`** CAS 保证，旧状态不匹配则更新失败；② **`findNextExecuteSubProcess`** 递归查找时每个步骤都过 **`check()`** 校验** |
| <strong>补偿机制是什么？</strong> | **当快照状态为 PROCESSING 时调用 **`matchProcess(workFlow).compensate(getCurrentNode(snapshot))`<strong>，补偿方法是"正向查询真实状态"而非"逆向回滚"——先查外部接口真实状态再决定重试还是标记失败</strong> |
| <strong>补偿本身也失败怎么办？</strong> | **三层兜底：① 子流程自身补偿有重试；② **`UnknownFinalStatusOrderMonitor`** 定时扫描未知状态订单告警；③ 人工介入手动推进。通过 CAS 防止并发补偿风暴** |
| <strong>能支持并行子流程吗？</strong> | **当前不支持，全部串行。改造方案：子流程配置增加 **`parallelSubProcessConfigId`** 标记，引擎用 **`CompletableFuture.allOf()`** 并行执行，需要考虑并行步骤失败时的补偿策略和共享状态加锁** |
| <strong>Purchase 流程有哪些步骤？</strong> | **5个阶段15个步骤：资质准入（合同创建→影像传输→资质评估→额度创建）→ 贷中提额 → 放款执行（额度冻结→购买评估→放款审批→支付→额度解冻→额度扣减）→ 结果通知 → 贷后提额。executeOrder 用千位分段设计方便中间插入** |
| <strong>助贷和联合贷的流程差异？</strong> | <strong>助贷多了"放款审批"步骤（资金方需审批），联合贷多了"机构放款通知"步骤（双方共同放款需通知对方）</strong> |
| <strong>流程模板是硬编码的吗？</strong> | <strong>当前是的，</strong>`LoanFlowTemplate`** 用 **`static {}`** 块在类加载时初始化子步骤列表。可以优化的方向是数据库配置 + 管理后台可视化编排** |

## **贡献二：额度管理体系**


| **面试官提问点** | **回答要点** |
|---|---|
| <strong>三种产品策略的差异是什么？</strong> | **① NormalProduct（普通）：直接创建新额度；② FuseProduct（融合）：一个产品对接多家资金方，创建时需关联多产品/机构信息，继承或关联已有额度；③ SwitchProduct（切换）：用户从一家资金方切换到另一家，创建前需先清空原关联额度，确保同一时间只有一个有效额度** |
| <strong>为什么用策略模式？</strong> | **三种产品类型的额度操作逻辑差异大但流程框架一致，策略模式将差异封装在具体策略中，通用逻辑放在 **`AbstractProductLimitCreateStrategy`** 基类（模板方法 + 策略模式组合），新增产品类型只需新增策略类 + 注册** |
| <strong>额度并发安全怎么保证？</strong> | **三层防护：① 数据库层 **`UPDATE SET amount = amount - ? WHERE id = ? AND amount >= ?`** 行锁 + 乐观锁；② 业务层额度状态机流转用 CAS 更新快照；③ 线程池隔离限制单机构并发数从源头降低竞争** |
| <strong>两个请求同时扣减同一额度怎么办？</strong> | **MySQL 串行化 UPDATE 执行，先到成功后到因 WHERE 条件不满足影响行数=0，业务层检测到抛 `OrgLimitRuleException**` |
| <strong>需要分布式锁吗？</strong> | **当前业务量级数据库行锁够用。引入 Redis 锁可以减少 DB 竞争压力，但增加了复杂度和故障点，需要评估** |
| <strong>提额流程是怎样的？</strong> | <strong>异步请求-回调模式：</strong>`RaiseLimitRequestProcessStrategy`** 判断是否需要风控审批 → 需要则发送请求到风控系统 → 风控异步审批 → MQ 回调 → **`RaiseLimitResultProcessStrategy`** 接收结果执行额度变更** |
| <strong>风险链路压单是什么？</strong> | **用户提额审批中（PROCESSING）又有新放款请求，放款会被压住等待提额结果出来后再决定放款额度。通过检查 **`RaiseLimitMaintenanceRecord`<strong> 状态实现，</strong>`RaiseLimitHoldStrategyManageService`** 管理压单策略** |
| <strong>额度状态流转？</strong> | **可用 → 冻结（放款时冻结对应金额）→ 扣减（放款成功扣减冻结金额）→ 解冻回可用（放款失败）；可用 → 冻结 → 解冻（恢复）；赎回/还款时额度恢复** |
| <strong>额度关闭策略的差异？</strong> | **三种策略对应三种产品的关户逻辑：Normal 关闭单条额度；Fuse 需要级联关闭关联产品额度；Switch 需要确保切换链路上的所有额度正确关闭** |

## **贡献三：CRC32 灰度分流**


| **面试官提问点** | **回答要点** |
|---|---|
| <strong>为什么选 CRC32？</strong> | **① 我们是固定比例分流不需要一致性哈希的节点动态增减能力；② CRC32 确定性保证同一 grayKey 始终映射到同一结果；③ 散列均匀性好不会扎堆；④ 计算极快纳秒级别** |
| <strong>和取模/一致性哈希对比？</strong> | **简单取模 key 微小变化导致结果剧变（雪崩效应）不推荐；一致性哈希适合分片场景实现复杂；CRC32 在固定比例分流场景最合适** |
| <strong>灰度精度？</strong> | **CRC32 值 % 10000 映射到 [0, 10000) 空间，和 grayRatio（BigDecimal）比较，精度达到万分之一** |
| <strong>一致性怎么保证？</strong> | **CRC32 是确定性算法，同一个 grayKey 永远映射到同一个 bucket，用户不会出现来回切换** |
| <strong>支持哪些灰度维度？</strong> | **6个：机构编码、产品编码、流程类型、资金模式（助贷/联合贷）、首复贷模式、合作模式。可做到"银树银行消费贷联合贷首贷购买流程按10%灰度"这种精细控制** |
| <strong>灰度异常怎么办？</strong> | <strong>设计了灰度/稳定双流程模板，</strong>`GrayWrapperPurchaseProcessConfig`<strong> 包装灰度配置和稳定配置。</strong>`SubProcessGrayAbilityService.selectById(snapshotId, isGray)`** 查灰度配置，灰度配置不存在时自动 fallback 到稳定配置** |
| <strong>新增灰度维度需要改代码吗？</strong> | <strong>当前需要，</strong>`GrayFlowProcessDivideConfig`** 要加字段 + 修改匹配逻辑。更好的方案是通用标签（Tag）机制——配置存为 **`Map<String, String>`** 键值对，动态比对标签** |
| <strong>grayKey 怎么选的？</strong> | **用订单号 + 机构编码 + 产品编码的组合作为 grayKey，多维度组合保证均匀分布。如果只用递增订单号可能在连续段内局部聚集** |
| <strong>grayRatio 怎么配的？</strong> | <strong>存储在数据库中，</strong>`PurchaseProcessGrayDecisionService`** 加载配置后匹配当前请求维度，根据 **`isShutGray(grayKey)`** 返回值决定是否走灰度流程** |

## **贡献四：统一消息消费框架**


| **面试官提问点** | **回答要点** |
|---|---|
| <strong>为什么设计 MqConsumer 接口？</strong> | **多个业务场景的消费者需要复用同一套消费框架，通过泛型接口统一约束消费流程（反序列化→路由→过滤→消费→幂等），新增消费者实现接口注册即可，符合开闭原则** |
| <strong>三层过滤的作用？</strong> | `**fitForTopic`（物理层：Topic匹配）→ `fitForBizScene`（场景层：同Topic下多种业务场景过滤）→ `isNeedConsume`（数据层：反序列化后根据业务字段判断），三层职责清晰逐级缩小范围** |
| <strong>幂等怎么实现？</strong> | `**createConsumerRecord` 方法从消息提取唯一 messageId → 查消费记录表是否已存在 → 已存在且状态为 SUCCESS 抛 `RepeatConsumeException` 跳过 → 状态为 FAILED 允许重试 → 插入新记录利用数据库唯一索引防并发** |
| <strong>重试策略？</strong> | **生产者用 guava-retrying，通常3次指数退避（1s→2s→4s），可配置重试异常类型** |
| <strong>消息顺序怎么保证？</strong> | <strong>① Mafka 同一 Key 的消息路由到同一 Partition 保证分区内有序；② 业务层通过工作流快照状态机保证，即使消息乱序到达也能正确处理（如先收到支付成功但额度还在冻结状态，引擎会正确推进）</strong> |
| <strong>覆盖哪些场景？</strong> | **PaymentConsumer（支付回调）、LoanRiskResultConsumer（贷中风控）、CreditRiskResultConsumer（授信风控）、RaiseLimitRiskConsumer（提额审批）、SettlementResultConsumer（结算结果）、ContractCreationResultConsumer（合同创建）、ImgTransferResultConsumer（影像传输）等 8+ 个** |
| <strong>消费失败怎么办？</strong> | **① 消费方法返回 **`MQMessageStatus`** 标记成功/失败/重试；② 失败的消息会重试消费；③ 超过重试次数进入死信处理；④ 监控告警** |

## **贡献五：监控告警与资源隔离**


| **面试官提问点** | **回答要点** |
|---|---|
| <strong>监控体系是怎样的？</strong> | <strong>三层：① 工作流级监控（</strong>`AbstractWorkFlowResultMonitor`<strong>）按机构/产品统计成功率、耗时；② 子流程级监控（</strong>`AbstractSubProcessResultMonitorService`<strong>）监控每个子步骤结果和耗时；③ 异常巡检（</strong>`UnknownFinalStatusOrderMonitor`<strong>）扫描长时间未终态订单</strong> |
| <strong>告警通道？</strong> | **CAT 打点 + 大象（DX）告警消息，关键告警触发电话/短信通知** |
| <strong>为什么要机构级线程池隔离？</strong> | **某个机构流量突增或外部接口变慢会导致公共线程池被占满，影响其他机构。隔离后故障影响范围限定在单个机构的线程池内** |
| <strong>线程池隔离怎么实现？</strong> | `**BusinessThreadResourceManager` 中为特定机构（如广达银行 M03030000）分配独立 `ExecutorService`，`getThreadResource(executedObj)` 按机构编码路由，找不到则 fallback 到默认线程池** |
| <strong>为什么要动态调整线程池？</strong> | **金融业务有潮汐特征（午间放款高峰），运行时通过 Lion 配置中心调整核心线程数、最大线程数等参数，不需要重启应用** |
| <strong>MTrace 包装的作用？</strong> | `**ExecutorServiceTraceWrapper` 包装确保异步线程中的调用也能被 MTrace 追踪到完整链路，不会出现链路断裂** |
| <strong>机构隔离是硬编码的吗？</strong> | **当前在 **`afterPropertiesSet()`** 中硬编码 **`map.put("M03030000", ...)`** 。更好的方案是从 Lion 配置中心动态加载需要隔离的机构列表** |
| <strong>生产问题排查流程？</strong> | <strong>① 看监控大盘确认问题范围 → ② 看 CAT 链路追踪定位异常环节 → ③ grep 日志确认异常栈 → ④ 紧急处理（联系银行/Lion关流量）→ ⑤ 定时任务自动补偿 → ⑥ 事后优化（加超时/加熔断）</strong> |
| <strong>流量开关怎么做？</strong> | `**TrafficSwitchBizType` 枚举定义各业务场景的流量开关，通过 Lion 配置一键关闭特定机构/产品的流量入口** |
| <strong>动态线程池的技术细节？</strong> | `**ExecutorWrapper.newThreadPoolExecutorIfNotExist(name)` 基于 Lion 配置创建线程池，参数变更时 Lion 回调动态调整核心线程数等参数** |
