# 私信功能设计 v4

> 本文档为博客平台私信（站内信）功能的设计方案，参考 CSDN 私信模式，以会话为组织单位，消息按时间线排列，支持 SSE 实时推送。

---

## 1. 产品设计

### 1.1 概述

博客平台目前缺乏用户间的直接沟通能力。用户只能通过文章评论、回复进行互动，无法发送私密消息。引入私信功能可以让用户建立更直接的联系，提升社区活跃度和用户粘性。

**核心原则：**
- 两人之间只存在一个会话，自动去重
- 消息实时送达（SSE 推送）
- 软删除，双方独立管理自己的可见性

### 1.2 用户故事

| 用户 | 场景 |
|------|------|
| 读者 A | 在读者 B 的主页看到"发私信"按钮，点击后进入聊天界面发送消息 |
| 作者 B | 收到新消息，导航栏出现未读红点提醒，点击进入私信中心查看 |
| 用户 C | 进入私信中心，看到所有历史会话列表，点击进入某个会话查看聊天记录 |
| 用户 D | 清理会话列表，删除不再需要的会话 |

### 1.3 页面结构

```
私信中心 /messages
  ├── 左侧会话列表
  │   └── 每一项：对方头像、昵称、最后一条消息预览、未读计数、时间
  └── 右侧会话详情 /messages/:conversationId
      ├── 消息时间线（按时间正序排列）
      │   └── 每条消息：头像、昵称、内容、时间
      └── 底部输入框 + 发送按钮

触发入口
  ├── 用户主页 → "发私信"按钮
  ├── 头部导航栏 → 私信图标（带未读红点）→ 下拉快捷跳转
  ├── 文章页用户信息区域 → "发私信"按钮
```

## 2. 功能设计

### 2.1 功能清单

| 模块 | 功能 | 说明 |
|------|------|------|
| 会话 | 获取/创建会话 | 根据对方用户 ID 获取或创建两人之间的会话 |
| 会话 | 会话列表 | 分页，按最后消息时间倒序，含最后一条消息预览 |
| 会话 | 删除会话 | 软删除，仅对操作方不可见 |
| 消息 | 发送消息 | 文本消息 |
| 消息 | 消息历史 | 分页滚动加载，按时间正序 |
| 消息 | 标记已读 | 进入会话时标记该会话所有消息为已读 |
| 消息 | 未读总数 | 所有会话的未读消息总数 |
| 消息 | 删除消息 | 软删除，仅对操作方不可见 |
| 实时 | SSE 推送 | 新消息实时推送 + 未读计数更新 |

### 2.2 数据模型

```sql
-- ============================================================
-- 私信会话表
-- ============================================================
CREATE TABLE `blog_conversation` (
    `id`               bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '会话ID',
    `participant_a_id` bigint unsigned NOT NULL COMMENT '参与者A用户ID（较小者）',
    `participant_b_id` bigint unsigned NOT NULL COMMENT '参与者B用户ID（较大者）',
    `last_message`     varchar(500)    DEFAULT NULL COMMENT '最后一条消息内容预览',
    `last_message_at`  datetime        DEFAULT NULL COMMENT '最后一条消息时间',
    `a_deleted_at`     datetime        DEFAULT NULL COMMENT 'A方删除时间',
    `b_deleted_at`     datetime        DEFAULT NULL COMMENT 'B方删除时间',
    `created_at`       datetime        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at`       datetime        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    `deleted_at`       datetime        DEFAULT NULL,
    `version`          bigint          NOT NULL DEFAULT 0,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_participants` (`participant_a_id`, `participant_b_id`),
    KEY `idx_participant_a` (`participant_a_id`, `a_deleted_at`, `last_message_at` DESC),
    KEY `idx_participant_b` (`participant_b_id`, `b_deleted_at`, `last_message_at` DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='私信会话表';


-- ============================================================
-- 私信消息表
-- ============================================================
CREATE TABLE `blog_message` (
    `id`              bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '消息ID',
    `conversation_id` bigint unsigned NOT NULL COMMENT '所属会话ID',
    `sender_id`       bigint unsigned NOT NULL COMMENT '发送者用户ID',
    `content`         text            NOT NULL COMMENT '消息内容',
    `type`            varchar(20)     NOT NULL DEFAULT 'TEXT' COMMENT '消息类型：TEXT / IMAGE / SYSTEM',
    `read_at`         datetime        DEFAULT NULL COMMENT '已读时间',
    `sender_deleted_at`   datetime    DEFAULT NULL COMMENT '发送方删除时间',
    `receiver_deleted_at` datetime    DEFAULT NULL COMMENT '接收方删除时间',
    `created_at`      datetime        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at`      datetime        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    `deleted_at`      datetime        DEFAULT NULL,
    `version`         bigint          NOT NULL DEFAULT 0,
    PRIMARY KEY (`id`),
    KEY `idx_conversation_time` (`conversation_id`, `created_at` DESC, `id` DESC),
    KEY `idx_sender` (`sender_id`),
    KEY `idx_receiver_unread` (`conversation_id`, `receiver_deleted_at`, `read_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='私信消息表';
```

### 2.3 API 设计

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| POST | `/api/messages/conversations` | 获取或创建会话 | 需要 |
| GET | `/api/messages/conversations` | 获取我的会话列表（分页） | 需要 |
| DELETE | `/api/messages/conversations/{id}` | 删除会话 | 需要 |
| GET | `/api/messages/conversations/{id}/messages` | 获取消息历史（分页） | 需要 |
| POST | `/api/messages/conversations/{id}/messages` | 发送消息 | 需要 |
| POST | `/api/messages/conversations/{id}/read` | 标记会话内所有消息为已读 | 需要 |
| GET | `/api/messages/unread-count` | 获取未读消息总数 | 需要 |
| GET | `/api/messages/stream` | SSE 推送新消息事件 | 需要 |

**详细请求/响应规范：**

```
POST /api/messages/conversations
Request:  { "participantId": 123 }
Response: { "code": 0, "data": { "id": 1, "participant": {...}, "lastMessage": null, "lastMessageAt": null } }

GET /api/messages/conversations?page=1&pageSize=20
Response: { "code": 0, "data": { "items": [{ "id": 1, "participant": {...}, "lastMessage": "...", "lastMessageAt": "...", "unreadCount": 3 }], "total": 5 } }

DELETE /api/messages/conversations/{id}
Response: { "code": 0 }

GET /api/messages/conversations/{id}/messages?page=1&pageSize=50

POST /api/messages/conversations/{id}/messages
Request:  { "content": "你好" }
Response: { "code": 0, "data": { "id": 1, "senderId": 789, "content": "你好", "createdAt": "2026-05-01 12:00:00" } }

POST /api/messages/conversations/{id}/read

```

### 3.3 SSE 集成方案

沿用现有通知系统的 SSE 模式：

- `MessageController` 维护 `ConcurrentHashMap<Long, CopyOnWriteArrayList<SseEmitter>> messageUserEmitters`
- SSE 端点 `GET /api/messages/stream` 使用 JWT token 查询参数鉴权
- 发送消息时，通过 SSE 向接收方推送事件：
  ```
  event: new-message
  data: {"conversationId":1,"senderId":456,"senderName":"张三","content":"你好","createdAt":"..."}
  ```
- 同时推送未读计数更新：
  ```
  event: unread
  data: {"count":5}
  ```
- 前端 `SiteHeader.vue` 在 `startNotificationSync()` 中同时订阅消息 SSE 流
