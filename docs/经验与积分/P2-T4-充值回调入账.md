## 2. 接口清单

### 2.1 充值回调入账

**POST** `/api/open/pay/recharge/callback`
**Auth**：无需登录（需验签）

> 该接口由支付平台异步回调，**不暴露给前端**，需要 IP 白名单或签名验证。

#### 请求示例

```json
{
  "payOrderNo": "PAY-20260515-001",
  "userId": 1001,
  "amountFen": 10000,
  "status": "SUCCESS",
  "sign": "sha256-hmac-signature-string",
  "notifyTime": "2026-05-15T10:30:00",
  "rawPayload": "{...原始支付平台数据...}"
}
```

#### 响应示例 — 入账成功

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "payOrderNo": "PAY-20260515-001",
    "pointsGranted": 100,
    "status": "SUCCESS"
  }
}
```

#### 响应示例 — 重复回调（幂等）

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "payOrderNo": "PAY-20260515-001",
    "status": "SUCCESS",
    "message": "已处理，不重复入账"
  }
}
```

#### 错误码

| code | 说明 |
|------|------|
| 400 | 验签失败 / 参数缺失 |
| 404 | userId 不存在 |
| 500 | 入账处理异常（支付平台会重试） |

## 3. 业务规则实现说明

### 3.1 积分兑换比例

```
默认：1 元（100 分）= 1 积分
即：amountFen / 100 = pointsGranted

比例可配置：point_rule_config.source_type=RECHARGE，point_amount 代表"每元获得的积分数"
```

### 3.2 验签规则

```
1. 取 rawPayload 中约定字段（payOrderNo + userId + amountFen + notifyTime）
2. 拼接为待签字符串
3. HMAC-SHA256(key=配置的签名密钥, data=待签字符串)
4. 与请求中 sign 字段比对（常量时间比较，防时序攻击）
5. 验签失败 → 返回 400，记录日志
```

### 3.3 幂等控制

- `uk_pay_order_no(pay_order_no)` 是核心幂等约束
- 流程：
  1. 查 `recharge_order WHERE pay_order_no = ?`
     - 若状态 = `SUCCESS` → 直接返回成功（不重复入账）
     - 若状态 = `PROCESSING` → 等待或返回处理中
     - 不存在 → 继续处理
  2. INSERT `recharge_order(PENDING)` — 利用唯一索引防并发重入
  3. 积分入账
  4. UPDATE `recharge_order` SET `status=SUCCESS`

### 3.4 事务边界

```
RechargeCallbackAppService.handleCallback()  @Transactional(rollbackFor = Exception.class)
  ├── RechargeSignVerifyService.verify(request) → 验签失败则抛异常
  ├── 查 recharge_order → 已 SUCCESS 直接返回（幂等）
  ├── INSERT recharge_order(PENDING) — 唯一索引兜底
  ├── 计算 pointsGranted = amountFen / 100 × ratio
  ├── PointAppService.addPoints(userId, pointsGranted, "RECHARGE", payOrderNo)
  └── UPDATE recharge_order SET status=SUCCESS, points_granted=?
```

### 3.5 notify_count 监控

每次回调时 `notify_count + 1`，便于监控支付平台的重试行为和定位幂等问题。

## 4. 测试验收点

| 场景 | 预期结果 |
|------|---------|
| 重复回调 20 次同一 payOrderNo | 只入账一次，`notify_count = 20`，balance 精确 |
| 验签失败 | 返回 400，recharge_order 不插入 |
| amountFen = 10000（100元）| pointsGranted = 100 |
| userId 不存在 | 返回 404，不入账 |
| 并发 20 线程同时回调 | 只有一笔入账，唯一索引保证 |

## 5. 风险与限制

- 签名密钥必须通过配置中心管理，不得硬编码在代码里
- 接口需配置 IP 白名单（支付平台 IP），防止伪造回调
- `rawPayload` 字段保留原始报文，便于事后对账和排查

---

## 6. 回滚方案

- **配置回滚**：将签名密钥清空或修改，使所有回调验签失败（停止新入账）
- **代码回滚**：注释 `RechargeCallbackController` 的 `@RestController`
- **数据回滚**：已入账流水不删除；如需冲正，通过管理员调分接口手动扣除，并记录原因
