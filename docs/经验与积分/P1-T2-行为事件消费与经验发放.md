## 2. 业务规则

### 2.1 行为事件契约

| 行为 | event_type | 触发来源 |
|------|-----------|---------|
| 点赞文章 | `LIKE` | `ArticleLikedEvent` |
| 阅读文章 | `READ` | `ArticleViewedEvent` |
| 发表评论 | `COMMENT` | `CommentCreatedEvent` |
| 收藏文章 | `FAVORITE` | `ArticleFavoritedEvent` |
| 分享文章 | `SHARE` | `ArticleSharedEvent`（待定） |
| 关注用户 | `FOLLOW` | `UserFollowedEvent` |

### 2.2 双角色分账规则

每个行为事件触发两笔经验发放：

- **ACTOR**（操作者）：执行行为的用户获得经验
- **AUTHOR**（被操作内容作者）：内容被交互的作者额外获得经验
- 两笔流水独立记账，互不干扰

### 2.3 日上限处理策略

```
当 daily_limit > 0 时：
  已发次数 < daily_limit → 正常发放
  已发次数 >= daily_limit：
    SKIP 策略：本次跳过，不发放
    PARTIAL 策略：计算剩余可发量，发放差值（最小 0）
当 daily_limit = 0 时：无上限限制
```

### 2.4 幂等控制

**幂等键构成规则**：
```
{event_type}:{role}:{actorUserId}:{sourceId}:{eventId}
```

示例：
```
LIKE:ACTOR:1001:2005:evt-abc123
LIKE:AUTHOR:2003:2005:evt-abc123
```

**幂等检查流程**：
1. 先向 `event_consume_record` 插入 `PROCESSING` 状态记录（若已存在则直接返回，幂等处理）
2. 执行发放逻辑
3. 更新 `event_consume_record` 为 `SUCCESS` 或 `FAILED`

## 3. 接口（应用服务方法）

### 3.1 ExpGrantAppService.grantExp()

```
@Transactional(rollbackFor = Exception.class)
void grantExpForBehavior(BehaviorExpEvent event)
```

**入参**：

| 字段 | 类型 | 说明 |
|------|------|------|
| eventId | String | 全局唯一事件 ID（幂等键基础） |
| eventType | String | 行为类型 |
| actorUserId | Long | 操作者用户 ID |
| authorUserId | Long | 内容作者用户 ID（可为 null） |
| sourceId | Long | 来源 ID（文章/评论 ID） |
| occurredAt | LocalDateTime | 事件发生时间 |

## 4. 核心流程图

```
行为事件 (ArticleLikedEvent 等)
        │
        ▼
GrantExpOnBehaviorEventHandler.onEvent()
        │
        ▼
ExpGrantAppService.grantExpForBehavior()  ← @Transactional
        │
        ├──① IdempotentConsumeService.tryLock(eventId, consumerName)
        │      └── INSERT event_consume_record(PROCESSING)
        │          └── 若唯一键冲突 → return（已处理）
        │
        ├──② for role in [ACTOR, AUTHOR]:
        │      └── ExpGrantDomainService.calcGrantAmount(rule, counter)
        │              └── 查 growth_rule_config → 得到 exp_amount / daily_limit
        │              └── 查 daily_reward_counter → 得到已发量
        │              └── 计算本次可发量（SKIP or PARTIAL）
        │
        ├──③ 原子写入 daily_reward_counter（ON DUPLICATE KEY UPDATE）
        │
        ├──④ INSERT user_exp_journal（携带唯一 idempotent_key）
        │
        ├──⑤ UPDATE user_growth_account（CAS 乐观锁）
        │      └── current_exp += delta
        │
        ├──⑥ 发布 LevelUpCheckEvent（由 P1-T3 处理）
        │
        └──⑦ IdempotentConsumeService.markSuccess(eventId, consumerName)
```

## 6. 测试验收点

| 场景 | 预期结果 |
|------|---------|
| 同一 eventId 重复消费 10 次 | `event_consume_record` 仅 1 条 SUCCESS，`user_exp_journal` 仅 1 条 |
| actor 当日已达上限（SKIP） | 不新增 `user_exp_journal`，`daily_reward_counter.exp_granted` 不变 |
| actor 当日已达上限（PARTIAL） | 发放剩余差值，`exp_granted` 精确等于 `daily_limit × exp_amount` |
| author 经验独立记账 | `user_exp_journal` 中 actorUserId ≠ authorUserId 两条独立记录 |
| 并发 20 线程同时发放 | `user_growth_account.current_exp` 精确等于单次发放额，无超发 |

## 7. 风险与限制

- 规则配置变更后需清空 `growth_rule_config` 本地缓存（建议 Caffeine TTL = 60s）
- `user_growth_account` 更新使用乐观锁（CAS version），并发高时需重试，建议最多重试 3 次后降级
- `event_consume_record` 建议定期归档，避免表膨胀（保留 90 天）

---

## 8. 回滚方案

- **配置回滚**：将 `growth_rule_config.enabled` 批量置为 0（关闭所有规则），消费链路自动跳过发放
- **代码回滚**：删除 `GrantExpOnBehaviorEventHandler` 的 `@EventListener` 注解，重启服务
- **数据不回滚**：已发流水为审计记录，不删除；如需冲正，手动插入负数 delta 的流水记录
