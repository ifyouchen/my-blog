### 2.1 创建经验规则

**POST** `/api/admin/growth/rules`

#### 请求示例

```json
{
  "eventType": "LIKE",
  "role": "ACTOR",
  "expAmount": 2,
  "dailyLimit": 20,
  "dailyLimitStrategy": "SKIP",
  "enabled": true,
  "effectiveAt": "2026-06-01T00:00:00",
  "reason": "新增点赞操作者经验规则"
}
```

#### 响应示例

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 1,
    "eventType": "LIKE",
    "role": "ACTOR",
    "expAmount": 2,
    "dailyLimit": 20,
    "dailyLimitStrategy": "SKIP",
    "enabled": true,
    "effectiveAt": "2026-06-01T00:00:00",
    "operator": "admin",
    "reason": "新增点赞操作者经验规则",
    "createdAt": "2026-05-15T10:00:00",
    "updatedAt": "2026-05-15T10:00:00"
  }
}
```

#### 错误码

| code | 说明 |
|------|------|
| 400 | 参数缺失或不合法（eventType/role 枚举值错误） |
| 401 | 未登录 |
| 403 | 非管理员 |
| 409 | 同 eventType+role 的启用规则已存在 |

## 3. 业务规则实现说明

### 幂等键
- 规则创建：`(eventType, role)` 组合唯一性由数据库索引保证（软删除前提下）
- 批量阈值保存：采用 `INSERT ... ON DUPLICATE KEY UPDATE`

### 事务边界
- 创建/更新规则：`@Transactional(rollbackFor = Exception.class)`
- 批量阈值保存：整体一个事务，任一失败全部回滚

### 生效时间机制
- `effectiveAt = null`：立即生效，消费链路每次读取最新配置
- `effectiveAt` 非空：消费链路查询规则时过滤 `effective_at <= NOW() OR effective_at IS NULL`

### 审计字段
- 每次 CREATE/UPDATE 自动记录：`operator`（当前登录用户名）、`reason`（必填）
- 建议通过 `SecurityContextHolder` 获取当前用户名

---

## 5. 风险与限制

- 规则删除（软删）后若存在 `effective_at` 未到期的版本，需在消费链路明确过滤 `deleted_at IS NULL`
- 批量阈值变更会影响所有用户的等级重算，建议在低峰期操作
- `operator` 字段依赖 Spring Security 上下文，接口需确保登录态正确传递
