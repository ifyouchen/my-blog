## 2. 接口清单

### 2.1 解锁文章

**POST** `/api/articles/{id}/unlock`
**Auth**：需要登录

#### 请求示例

```http
POST /api/articles/2005/unlock
Authorization: Bearer <token>
```

（无请求体）

#### 响应示例 — 解锁成功

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "articleId": 2005,
    "unlocked": true,
    "pointsCost": 50,
    "balanceAfter": 300,
    "orderNo": "UNLOCK-1001-2005-1715747025"
  }
}
```

#### 响应示例 — 已解锁（幂等）

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "articleId": 2005,
    "unlocked": true,
    "message": "已解锁"
  }
}
```

#### 错误码

| code | 说明 |
|------|------|
| 400 | 积分不足（POINT_INSUFFICIENT）|
| 400 | 文章无需解锁（need_unlock=0）|
| 401 | 未登录 |
| 404 | 文章不存在 |

## 3. 业务规则实现说明

### 3.1 解锁流程

```
ArticleUnlockAppService.unlock(userId, articleId)  @Transactional
  ├── 查 blog_article（只读，获取 need_unlock / unlock_point_price / author_id）
  │       └── 若 need_unlock=0 → 返回 400
  │
  ├── 查 article_unlock_relation WHERE user_id=? AND article_id=? （已解锁？）
  │       └── 若已存在 → 返回幂等成功
  │
  ├── 查 user_point_account.balance 是否足够
  │       └── 若 balance < unlock_point_price → 返回 400 POINT_INSUFFICIENT
  │
  ├── 生成 orderNo = "UNLOCK-{userId}-{articleId}-{timestamp}"
  ├── INSERT article_unlock_order(PENDING)  — uk_user_article 防并发重入
  │
  ├── PointAppService.deductPoints(userId, pointsCost, "UNLOCK", orderNo)
  │       └── 扣分失败（余额不足）→ 整体回滚
  │
  ├── INSERT article_unlock_relation
  ├── RevenueShareAppService.createShareRecord(orderNo, articleId, authorId, pointsCost)
  │       └── 计算 platform / author 分成并写 revenue_share_journal
  │
  └── UPDATE article_unlock_order SET status=SUCCESS
```

### 3.2 幂等控制

- `uk_user_article(user_id, article_id)` 在 `article_unlock_order` 和 `article_unlock_relation` 两表均有，双重防护
- 多次点击解锁按钮：第一次 INSERT 成功，后续命中唯一键 → 捕获异常 → 查已存在记录返回成功

### 3.3 一致性保证

解锁订单、积分扣减、解锁关系、分账记录**在同一本地事务**中完成，任一步骤失败整体回滚。

## 4. 测试验收点

| 场景 | 预期结果 |
|------|---------|
| 100 并发同一用户解锁同一文章 | 只有一笔扣分，order/relation 各 1 条 |
| 积分不足 | 返回 400，balance 不变，无 order 记录 |
| 已解锁再次调用 | 返回幂等成功，balance 不变 |
| 解锁成功后查询 unlock-status | unlocked=true |
| 解锁成功后分账记录 | revenue_share_journal 有一条，platform+author=total |

## 5. 风险与限制

- `blog_article` 的 `ALTER` 是对现有表的非破坏性变更（加字段），上线时需在低峰期执行，并提前备份
- 成长模块通过 MyBatis 直接查 `blog_article` 表读取 `need_unlock`/`unlock_point_price`/`author_id`，不调用博客应用服务
