## 2. 等级升级规则

### 2.1 升级逻辑

```
每次经验变更后（P1-T2 结束时发布 LevelUpCheckEvent）：

1. 读取 level_threshold_config（按 level ASC，启用的）
2. 从最高级向下遍历，找到 current_exp >= min_exp 的最大 level
3. 若 computed_level != current_level → 执行升级
   - UPDATE user_growth_account SET current_level = computed_level
   - 记录升级日志（可选：user_exp_journal 写一条 event_type=LEVEL_UP，delta=0）
4. 若等级无变化 → 跳过
```

### 2.2 "距下一级差值"计算

```
next_threshold = level_threshold_config[current_level + 1].min_exp
gap = next_threshold - current_exp   （若已是最高级则返回 0）
progress_percent = (current_exp - current_threshold.min_exp)
                   / (next_threshold - current_threshold.min_exp) × 100
```

## 4. 核心流程图

```
LevelUpCheckEvent (userId, newExp)
        │
        ▼
LevelUpCheckEventHandler.onEvent()
        │
        ▼
GrowthQueryAppService.recalcLevel(userId)
        │
        ├── 读 level_threshold_config（缓存 TTL=5min）
        │
        ├── LevelPolicyService.computeLevel(currentExp, thresholds)
        │       └── 返回 computedLevel
        │
        ├── 若 computedLevel != currentLevel
        │       └── GrowthAccount.applyLevelUp(computedLevel)
        │               └── UPDATE user_growth_account（乐观锁）
        │
        └── （可选）发布 LevelUpNotificationEvent
```

## 6. 测试验收点

| 场景 | 预期结果 |
|------|---------|
| exp=0，阈值配置 level1=0 | currentLevel=1 |
| exp=99，level2 阈值=100 | 保持 level1，expToNextLevel=1 |
| exp=100，level2 阈值=100 | 升级为 level2，progressPercent 重算 |
| exp=12000，最高 level8=12000 | currentLevel=8，expToNextLevel=0，progressPercent=100 |
| exp 跨越多级（如从 0 → 1500） | 直接升至 level5，不逐级触发 |
| 并发 50 线程同时写经验 | currentLevel 最终正确，无版本冲突死锁 |

## 7. 风险与限制

- 等级阈值配置变更后需主动刷新缓存，否则升级判断可能延迟
- `GrowthAccount` 使用乐观锁更新，极端并发下可能产生短暂的等级滞后（下次经验写入时会再触发重算）
- 不支持降级：经验减少时不降等级（设计决策）

---

## 8. 回滚方案

- **配置回滚**：修改 `level_threshold_config` 中的 `enabled=0` 可暂停升级检查
- **代码回滚**：关闭 `LevelUpCheckEventHandler` 的 `@EventListener` 注解
- **数据回滚**：手动 `UPDATE user_growth_account SET current_level = ? WHERE user_id = ?`（需谨慎，批量操作需审批）
