# 完整博客平台落地实施方案

## Summary

基于当前项目现状，下一阶段目标不是推倒重来，而是在第一版 MVP 上补齐完整博客平台能力。实施顺序固定为：

1. 平台治理与安全底座
2. 创作者工作台升级
3. 内容组织与发现增强
4. SEO 与公开增长能力
5. 运营位与商业化预留

默认不优先做支付、会员、复杂推荐算法、企业空间。先把平台变成“可公开运营、可持续创作、可治理内容”的完整博客社区。

## Phase 1：平台治理与安全底座

目标：平台可以开放给更多用户使用，不怕垃圾内容、违规评论和后台无据可查。

### 数据库

新增表：

- `blog_report`
  - `id`
  - `reporter_user_id`
  - `target_type`: `ARTICLE | COMMENT | USER`
  - `target_id`
  - `reason_type`: `SPAM | INFRINGEMENT | ABUSE | ILLEGAL | OTHER`
  - `reason_detail`
  - `status`: `PENDING | RESOLVED | REJECTED`
  - `handler_user_id`
  - `handle_note`
  - `handled_at`
  - `created_at`
  - `updated_at`
  - `deleted_at`
  - `version`

扩展字段：

- `blog_article.offline_reason`
- `blog_user.disable_reason`

索引：

- `blog_report(status, created_at desc, id desc)`
- `blog_report(target_type, target_id)`
- `blog_report(reporter_user_id, created_at desc)`

### 后端接口

新增公开举报接口：

- `POST /api/reports`
  - 登录用户可用
  - 请求：`targetType`, `targetId`, `reasonType`, `reasonDetail`
  - 限制：同一用户对同一对象未处理举报不可重复提交
  - 返回：举报 ID

新增后台举报接口：

- `GET /api/admin/reports`
  - 支持 `status`, `targetType`, `reasonType`, `page`, `pageSize`
- `POST /api/admin/reports/{id}/resolve`
  - 请求：`action`, `handleNote`
  - `action`: `REJECT | OFFLINE_ARTICLE | DELETE_COMMENT | DISABLE_USER`
- `GET /api/admin/reports/{id}`
  - 返回举报详情、目标摘要、处理记录

扩展后台接口：

- `POST /api/admin/articles/{id}/offline`
  - 请求：`reason`
- `POST /api/admin/users/{id}/disable`
  - 请求：`reason`

保留现有状态接口兼容，但后台新 UI 优先用语义化接口。

### 后端实现步骤

1. 建 `Report` 领域模型和 `ReportRepository`。
2. 增加 MyBatis `ReportMapper.xml`，包含 `BaseResultMap` 和 `Base_Column_List`。
3. 新增 `ReportAppService`，负责创建举报、分页查询、处理举报。
4. 新增 `ReportController` 和 admin 下的举报接口。
5. 举报处理时调用现有文章下架、评论删除、用户禁用逻辑。
6. 所有后台处理动作写入 `AdminLog`。
7. 给创建举报接口加基础频率限制，第一版可用 Caffeine 本地计数。

### 前端实现步骤

1. 新增通用举报弹窗 `ReportDialog.vue`。
2. 在文章详情页增加“举报文章”入口。
3. 在评论项增加“举报评论”入口。
4. 在用户主页增加“举报用户”入口。
5. 新增后台举报页 `/admin/reports`。
6. 后台举报页包含筛选、列表、详情抽屉、处理弹窗。
7. 处理成功后静默刷新列表，失败显示错误提示。

### 验证点

- 登录用户可以举报文章、评论、用户。
- 游客点击举报会触发登录引导。
- 同一用户重复举报同一对象时返回明确提示。
- 管理员可以驳回举报。
- 管理员可以通过举报处理下架文章、删除评论、禁用用户。
- 处理后举报状态变为 `RESOLVED` 或 `REJECTED`。
- 后台日志能查到处理动作。
- 被下架文章前台不可见。
- 被删除评论前台不可见。
- 被禁用用户不能登录和执行互动操作。

## Phase 2：创作者工作台升级

目标：让作者知道自己的内容表现，并持续写作。

### 数据库

新增统计表：

- `blog_article_daily_stats`
  - `id`
  - `article_id`
  - `stat_date`
  - `view_count`
  - `like_count`
  - `favorite_count`
  - `comment_count`
  - `created_at`
  - `updated_at`

- `blog_creator_daily_stats`
  - `id`
  - `user_id`
  - `stat_date`
  - `view_count`
  - `like_count`
  - `favorite_count`
  - `comment_count`
  - `follower_count`
  - `article_count`
  - `created_at`
  - `updated_at`

第一阶段可以先不做定时任务表回填，允许接口实时聚合近 7/30 天数据；但表结构先预留。

### 后端接口

新增创作者接口：

- `GET /api/dashboard/overview`
  - 返回总文章数、草稿数、已发布数、总阅读、总点赞、总收藏、总评论、粉丝数
- `GET /api/dashboard/trends?range=7d|30d`
  - 返回日期维度趋势
- `GET /api/dashboard/articles/performance`
  - 支持 `sort=view|like|favorite|comment|updated`
  - 返回文章表现列表
- `GET /api/dashboard/interactions`
  - 返回最新评论、被点赞、被收藏、被关注摘要

### 后端实现步骤

1. 在 `UserAppService` 或新增 `DashboardAppService` 中实现创作者聚合。
2. 仓储层新增聚合查询，避免 `findAll()` 后内存计算。
3. 文章表现接口不查 `content` 大字段。
4. 趋势接口先基于现有 `created_at`、文章统计字段做轻量聚合。
5. 后续再用日统计表替换实时聚合。

### 前端实现步骤

1. 重构 `/dashboard/articles` 上方概览区，改成创作者工作台。
2. 增加 `/dashboard/overview` 或在现有 dashboard 中加入概览 Tab。
3. 增加数据卡：总阅读、总点赞、总收藏、总评论、粉丝数。
4. 增加趋势图：近 7/30 天阅读和互动。
5. 增加表现最佳文章列表。
6. 增加互动反馈列表，点击能回到文章或通知详情。
7. 我的文章页保留现有表格，但入口从概览页进入。

### 验证点

- 作者能看到自己的总体数据。
- 草稿数、发布数、下架数与我的文章列表一致。
- 近 7/30 天切换后趋势数据变化正确。
- 文章表现排序正确。
- 表现列表不加载正文大字段。
- 空数据作者显示友好引导，不出现空白。
- 后端接口在 1000 篇文章级别仍不明显变慢。

## Phase 3：内容组织与发现增强

目标：让平台不只是文章列表，而是有专题、推荐、作者、专栏的内容组织能力。

### 数据库

新增专题表：

- `blog_topic`
  - `id`
  - `title`
  - `summary`
  - `cover_url`
  - `status`: `PUBLISHED | OFFLINE`
  - `sort_order`
  - `created_at`
  - `updated_at`
  - `deleted_at`
  - `version`

- `blog_topic_article`
  - `id`
  - `topic_id`
  - `article_id`
  - `sort_order`
  - `created_at`
  - `updated_at`
  - `deleted_at`
  - `version`

扩展字段：

- `blog_column.recommended`
- `blog_column.recommend_weight`
- `blog_user.recommended`
- `blog_user.recommend_weight`
- `blog_article.featured`
- `blog_article.featured_at`

### 后端接口

新增前台接口：

- `GET /api/topics`
- `GET /api/topics/{id}`
- `GET /api/topics/{id}/articles`
- `GET /api/recommendations/authors`
- `GET /api/recommendations/columns`
- `GET /api/recommendations/articles`

新增后台接口：

- `GET /api/admin/topics`
- `POST /api/admin/topics`
- `PUT /api/admin/topics/{id}`
- `DELETE /api/admin/topics/{id}`
- `POST /api/admin/topics/{id}/articles`
- `DELETE /api/admin/topics/{id}/articles/{articleId}`
- `POST /api/admin/articles/{id}/feature`
- `POST /api/admin/articles/{id}/unfeature`

### 后端实现步骤

1. 新增 Topic 领域模型、仓储、Mapper。
2. 前台专题只返回 `PUBLISHED` 且未删除数据。
3. 专题文章只返回已发布文章。
4. 推荐作者、推荐专栏先走后台配置权重，不做算法推荐。
5. 精选文章由管理员手动设置。
6. 推荐接口使用 Caffeine 缓存，TTL 1 到 5 分钟。

### 前端实现步骤

1. 新增专题列表页 `/topics`。
2. 新增专题详情页 `/topics/:id`。
3. 首页右侧“热门专题”接真实专题接口。
4. 首页推荐作者、推荐专栏接真实推荐接口。
5. 文章详情页增加相关推荐。
6. 后台新增专题管理页。
7. 后台文章管理页增加“设为精选/取消精选”。

### 验证点

- 后台可以创建、编辑、下架专题。
- 专题页只展示已发布文章。
- 首页热门专题来自真实接口。
- 推荐作者和推荐专栏可由后台控制排序。
- 文章设为精选后能进入精选流。
- 下架专题后前台不可见。
- 删除专题不物理删除。
- 缓存命中后推荐接口响应更稳定。

## Phase 4：SEO 与公开增长能力

目标：让公开内容能被搜索引擎收录和传播。

### 数据库

扩展文章字段：

- `blog_article.slug`
- `blog_article.seo_title`
- `blog_article.seo_description`

索引：

- `uk_blog_article_slug(slug)`

### 后端接口

新增：

- `GET /sitemap.xml`
- `GET /rss.xml`
- `GET /robots.txt`

扩展文章接口：

- 创建/更新文章支持 `slug`, `seoTitle`, `seoDescription`
- 文章详情返回 SEO 字段
- 支持 `/articles/{id}-{slug}` 或 `/articles/{slug}`，推荐第一阶段用 `/articles/{id}-{slug}` 降低冲突风险

### 后端实现步骤

1. 文章创建时自动生成 slug。
2. 作者可在编辑器中修改 slug，但必须全站唯一。
3. sitemap 只输出已发布文章、专栏、专题、作者主页。
4. RSS 输出最近发布文章。
5. robots 输出允许抓取公开内容，禁止抓取 admin、dashboard、settings。

### 前端实现步骤

1. 文章详情动态设置 title、description。
2. 文章编辑器增加 SEO 设置折叠区。
3. 文章分享链接优先使用 slug 地址。
4. 作者主页、专栏详情、专题详情补充页面 title。
5. 前端路由兼容旧 `/articles/:id`。

### 验证点

- 新文章自动生成 slug。
- slug 重复时后端返回明确错误。
- 旧文章仍可通过 `/articles/:id` 打开。
- `/sitemap.xml` 可访问并包含文章链接。
- `/rss.xml` 可访问并包含最近文章。
- 文章详情浏览器标题正确。
- 分享到社交平台时有标题、摘要和封面。

### 前端实现步骤

1. 首页右侧栏支持广告位。
2. 文章详情右侧栏支持广告位。
3. 搜索结果页支持广告位。
4. 广告必须显示“广告”或“推广”标签。
5. 曝光事件懒触发，进入视口后上报。
6. 点击广告先上报点击，再跳转目标链接。
7. 后台新增广告管理页和统计概览。

### 验证点

- 广告只在配置时间范围内展示。
- 禁用广告不展示。
- 曝光只在广告进入视口后上报。
- 点击广告会记录点击。
- 外链打开新窗口，并加 `noopener noreferrer`.
- 广告样式不伪装成普通文章卡。

## Implementation Order

执行顺序固定：

1. Phase 1 举报治理和安全底座
2. Phase 2 创作者数据工作台
3. Phase 3 专题和推荐
4. Phase 4 SEO
5. Phase 5 广告运营位

每个 Phase 的执行方式固定：

1. 先改数据库 schema。
2. 再补 domain/repository/application/controller。
3. 再补前端 API。
4. 再补页面和交互。
5. 最后补测试和验收。

不要跨 Phase 同时开太多战线。每个 Phase 完成后必须保证：

- `mvn -q -DskipTests compile` 通过。
- `npm run build` 通过。
- 关键页面手工验收通过。
- 不引入原生 alert/confirm。
- 不恢复列表刷新闪烁。

## Test Plan

### 后端基础测试

- 举报创建、重复举报、举报处理。
- 管理员权限校验。
- 下架文章、删除评论、禁用用户。
- 创作者概览统计。
- 专题创建、文章关联、前台可见性。
- slug 唯一性。
- sitemap/rss 输出。
- 广告展示和事件记录。

### 前端验收

- 文章详情举报入口可用。
- 评论举报入口可用。
- 用户主页举报入口可用。
- 后台举报处理完整。

- 评论、回复、删除。
- 点赞、收藏、关注。

- 我的文章和我的收藏。
- 管理后台用户、文章、评论、分类、标签、日志。

## Assumptions

- 产品方向是完整技术博客平台，不是个人博客站。
- 第一阶段不做支付、会员、积分、企业空间。
- 推荐先采用后台配置和轻量排序，不做复杂算法。
- 搜索先保留 MySQL 方案，后续再替换全文检索。
- 广告先做自营广告位，不做竞价投放。
- 所有新增表继续遵循 `deleted_at`, `version`, `created_at`, `updated_at` 约定。
- 后端继续遵循 DDD 分层，Controller 不直接访问 Mapper。
