# 博客平台产品完善方案

## Summary

在 `11-设计v2.md` 规划的五个 Phase 全部落地后，平台已具备完整的内容生产、社交互动、内容发现、SEO 和运营位能力。本文档聚焦"从完整到好用"的跃迁：补齐用户在核心路径上的体验断点，打磨互动细节，提升内容消费效率，强化创作者留存能力，以及完善平台可运营性。

不做的事：支付、会员体系、积分商城、App 客户端、复杂推荐算法、多租户空间。

实施顺序：

1. 核心路径体验修补
2. 创作者能力增强
3. 内容消费与发现优化
4. 社区互动深化
5. 平台运营与数据能力

---

## Phase 1：核心路径体验修补

目标：消除当前用户在注册登录、文章阅读、移动端浏览、设置项等关键路径上的体验断点。

### 1.1 账号与认证

**问题：**
- 注册后没有欢迎引导，用户不知道下一步做什么。
- 忘记密码没有找回入口（当前无此功能）。
- 没有"保持登录"能力（Token 过期后被强制退出）。
- 用户无法修改密码。
- 无邮箱绑定，无法接收重要通知（举报处理结果、系统公告）。

**数据库扩展：**

```sql
-- blog_user 新增字段
ALTER TABLE blog_user
    ADD COLUMN email        VARCHAR(200)  NULL COMMENT '登录邮箱（唯一，可选绑定）',
    ADD COLUMN email_verified TINYINT(1) NOT NULL DEFAULT 0 COMMENT '邮箱是否已验证',
    ADD COLUMN password_reset_token  VARCHAR(128) NULL COMMENT '密码重置 Token',
    ADD COLUMN password_reset_expire DATETIME     NULL COMMENT '密码重置 Token 过期时间',
    ADD COLUMN last_login_at DATETIME NULL COMMENT '最近登录时间',
    ADD COLUMN last_login_ip VARCHAR(64) NULL COMMENT '最近登录 IP';
```

**后端接口：**

```
POST /api/auth/password/forgot
  - 请求：email
  - 逻辑：生成重置 token，发送邮件（第一阶段可 log 到控制台代替真实发送）
  - 限流：每个 IP / 邮箱每小时最多 5 次

POST /api/auth/password/reset
  - 请求：token, newPassword
  - 校验：token 有效且未过期

POST /api/users/me/password
  - 需登录
  - 请求：currentPassword, newPassword

POST /api/users/me/email
  - 需登录
  - 请求：email, password（确认身份）

GET /api/users/me/security
  - 需登录
  - 返回：lastLoginAt, lastLoginIp, emailBound, emailVerified
```

**前端：**

1. 登录页增加"忘记密码"链接，跳转至 `/auth/forgot-password`。
2. 新增 `/auth/forgot-password` 页面（邮箱输入 + 提交）。
3. 新增 `/auth/reset-password?token=xxx` 页面（新密码输入 + 确认）。
4. 个人资料设置页 `/settings/profile` 新增"账号安全"Tab：
   - 修改密码（当前密码 + 新密码 + 确认密码）
   - 绑定邮箱（支持更换）
   - 显示最近登录时间和 IP
5. 注册成功后显示欢迎引导弹窗：
   - 提示完善头像和简介
   - 引导关注几个推荐作者
   - 引导写第一篇文章或探索内容

### 1.2 个人资料完善

**问题：**
- 当前 `/settings/profile` 仅支持昵称、头像、简介，没有 username（@名）修改、个人主页链接、社交账号等信息。
- 用户头像没有裁剪功能，上传后比例可能变形。
- 设置页只有"个人资料"一项，账号安全、通知偏好没有入口。

**数据库扩展：**

```sql
ALTER TABLE blog_user
    ADD COLUMN website     VARCHAR(500) NULL COMMENT '个人网站',
    ADD COLUMN github      VARCHAR(200) NULL COMMENT 'GitHub 用户名',
    ADD COLUMN twitter     VARCHAR(200) NULL COMMENT 'Twitter/X 用户名',
    ADD COLUMN location    VARCHAR(200) NULL COMMENT '所在地';
```

**后端接口扩展：**

```
PUT /api/users/me/profile
  - 新增字段支持：website, github, twitter, location
  - username 修改：每 30 天只能改一次（在 last_username_changed_at 字段记录）
```

**前端：**

1. `/settings/profile` 页面扩展字段：个人网站、GitHub、Twitter/X、所在地。
2. 头像上传改为"上传后裁剪"流程（裁剪组件可使用 `cropperjs`）。
3. `/settings` 路由下增加三个 Tab：个人资料 / 账号安全 / 通知偏好。
4. 用户主页展示新增字段（网站链接、GitHub、所在地）。

### 1.3 移动端响应式修补

**问题：**
- 文章编辑器在手机上几乎无法使用（工具栏溢出、预览区撑满）。
- 后台管理页表格在移动端列数溢出。
- 首页侧边栏在移动端仍然渲染（被隐藏了，但接口仍然请求）。
- 搜索结果页筛选器在移动端交互不流畅。

**前端：**

1. 编辑器工具栏在移动端自动折叠为两行，最常用操作保留第一行（粗体、斜体、链接、图片、代码块）。
2. 编辑器移动端默认隐藏预览区，通过底部 Tab 切换"编辑"和"预览"。
3. 管理后台表格在 ≤768px 时改为卡片列表布局，每条记录用竖排卡片展示。
4. 首页侧边栏在移动端通过 CSS media query 隐藏，同时用 `v-if` 在小屏幕下不渲染 DOM（通过 `useWindowSize` composable 实现）。
5. 搜索页筛选抽屉在移动端改为全屏底部滑出（bottom sheet）。

### 1.4 页面加载体验

**问题：**
- 文章详情页初次加载内容区空白时间过长，没有骨架屏。
- 列表页刷新时数据全部清空后重新填充，有闪烁。
- 图片未做懒加载和渐进加载，首屏大量图片同时请求。

**前端：**

1. 文章详情页添加内容骨架屏（标题占位符 + 段落线条）。
2. 文章列表骨架屏：每个列表项显示灰色矩形占位块，与真实卡片尺寸一致。
3. 所有 `<img>` 标签统一补充 `loading="lazy"` 和 `decoding="async"`（全局 ESLint 规则检查）。
4. 封面图和头像增加 `blur-up` 渐进加载效果（先展示低质量缩略图，加载完成后淡入）。
5. 路由切换添加顶部进度条（`nprogress`）。

## Phase 2：创作者能力增强

目标：让作者有更强的写作工具、更清晰的数据反馈、更顺畅的发布流程。

### 2.1 Markdown 编辑器增强

**问题：**
- 当前编辑器不支持表格插入（只能手写 Markdown 语法）。
- 不支持数学公式（KaTeX）。
- 不支持代码块语言切换（只能手写 ` ``` python`）。
- 没有全局字数/阅读时间估算。
- 插入图片只支持上传，不支持粘贴图片（`paste` 事件处理）。
- 没有"撤销/重做"快捷键提示。
- 长文章 TOC（目录）不能折叠。

**前端：**

1. 工具栏新增"插入表格"操作，弹出行列选择网格，插入标准 Markdown 表格模板。
2. 支持粘贴图片自动上传：监听 `paste` 事件，检测 `clipboardData.files`，调用上传接口后插入图片链接。
3. 工具栏代码块按钮改为下拉菜单，可选常用语言（JavaScript、Python、Java、Go、Bash、SQL、JSON、YAML）。
4. 底部状态栏新增：字数统计（中文字符计 1，英文单词计 1）+ 预估阅读时间（按 300 字/分钟）。
5. 编辑器右侧 TOC 支持折叠/展开，默认在文章超过 10 个标题时折叠三级以下。
6. 快捷键提示 Tooltip，悬停工具栏按钮时显示快捷键（`Ctrl+B`、`Ctrl+I` 等）。

### 2.2 文章草稿与版本

**问题：**
- 草稿只保存在 localStorage，清除浏览器数据后丢失。
- 没有"版本历史"，文章发布后无法回滚到之前的版本。
- 文章保存后没有明确的"已保存"反馈（只有本地草稿）。

**数据库：**

```sql
CREATE TABLE blog_article_version (
    id          BIGINT UNSIGNED  NOT NULL AUTO_INCREMENT,
    article_id  BIGINT UNSIGNED  NOT NULL,
    version_no  INT              NOT NULL COMMENT '版本号，从 1 开始递增',
    title       VARCHAR(500)     NOT NULL,
    content     LONGTEXT         NOT NULL,
    summary     VARCHAR(500)     NULL,
    saved_by    BIGINT UNSIGNED  NOT NULL COMMENT '保存操作的用户 ID',
    created_at  DATETIME         NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    KEY idx_article_version (article_id, version_no DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='文章版本历史';
```

**后端接口：**

```
GET /api/articles/{id}/versions
  - 需登录且为作者
  - 返回版本列表（version_no, 预览标题, saved_by, created_at）

GET /api/articles/{id}/versions/{versionNo}
  - 返回该版本完整内容

POST /api/articles/{id}/versions/{versionNo}/restore
  - 将文章内容回滚到指定版本（创建新草稿）
```

**后端实现：**
1. 文章每次"保存草稿"或"发布"操作后，在 `blog_article_version` 中写入当前内容快照。
2. 仅保留最近 20 个版本，超出时自动删除最旧版本。
3. 恢复版本操作创建一个新草稿（不直接覆盖当前已发布版本）。

**前端：**

1. 编辑器工具栏右侧增加"历史版本"按钮，点击打开侧边抽屉。
2. 版本列表展示：版本号、保存时间、标题预览；点击某个版本可在右侧预览内容。
3. 预览版本后，提供"恢复到此版本"操作（确认后回填编辑器并提示"已恢复为草稿，请确认后保存"）。
4. 编辑器顶部增加"已自动保存"状态标识（定时 30 秒自动保存到服务端草稿，并在状态栏显示"已自动保存 xx:xx"）。

### 2.3 专栏管理（创作者侧）

**问题：**
- 目前专栏只能由管理员在后台创建，普通创作者无法自主创建专栏。
- 创作者没有"我的专栏"管理入口。

**后端接口（新增，由作者调用）：**

```
GET /api/users/me/columns
  - 需登录
  - 返回当前用户创建的所有专栏

POST /api/columns
  - 需登录
  - 请求：name, description, coverUrl
  - 每个用户最多创建 5 个专栏（可配置）

PUT /api/columns/{id}
  - 需登录且为专栏作者
  - 更新名称、描述、封面

DELETE /api/columns/{id}
  - 需登录且为专栏作者
  - 软删除，同时清除关联文章

POST /api/columns/{id}/articles
  - 需登录且为专栏作者
  - 请求：articleId（文章必须是本人所有）

DELETE /api/columns/{id}/articles/{articleId}
  - 需登录且为专栏作者
```

**前端：**

1. Dashboard 创作概览新增"我的专栏"卡片，显示已创建专栏数 + "管理专栏"入口。
2. 新增 `/dashboard/columns` 页面（我的专栏列表），支持创建、编辑、删除专栏，以及管理专栏内文章。
3. 文章编辑器"高级设置"区新增"归入专栏"下拉选择（只展示自己的专栏）。
4. 文章详情页作者区块显示该文章所属专栏名称 + 跳转链接。

### 2.4 创作者数据增强

**问题：**
- 当前趋势图没有"分文章"的数据钻取。
- 没有"受众画像"（访问来源、读者地区）。
- 没有"最佳发布时间"建议。

**后端接口（增量）：**

```
GET /api/dashboard/articles/{id}/stats
  - 需登录且为作者
  - 返回单篇文章详细统计：每日阅读趋势（最近 30 天）、点赞趋势、收藏趋势、评论趋势
  - 顺带返回：总阅读、点赞、收藏、评论
```

**前端：**

1. Dashboard 文章表现列表点击某篇文章，右侧滑出详情抽屉，展示该文章的趋势折线图。
2. 创作概览增加"最活跃读者时段"展示（基于 `blog_article_view.created_at` 按小时聚合的热力图，取最近 30 天数据）。

## Phase 3：内容消费与发现优化

目标：让读者更容易发现好内容，更愿意长时间停留在平台上。

### 3.1 文章详情页体验

**问题：**
- 文章正文宽度在大屏上偏宽，影响阅读舒适度（最优行长 60-80 字符）。
- 阅读进度没有反馈（不知道读了多少）。
- 没有"字体大小"和"阅读模式"切换。
- 文章分享能力缺失：没有复制链接、生成分享卡片等功能。
- 文章底部没有"上一篇/下一篇"导航（在同专题或同作者范围内）。
- 代码块没有"复制代码"按钮。
- 长文章没有"回到顶部"浮动按钮。

**前端：**

1. 正文区最大宽度限制为 `720px`，并在阅读时自动居中对齐。
2. 文章顶部增加阅读进度条（固定在页面顶部的细线，随滚动变化宽度）。
3. 文章工具栏（点赞/收藏栏）新增"分享"按钮：
   - 复制链接（带 slug）
   - 分享到微博（拼接 `weibo.com/share` 链接）
   - 生成分享图片（Canvas 渲染标题 + 作者 + 站名，导出 PNG）
4. 文章底部增加相关文章导航，展示同专题或同作者的"上一篇/下一篇"。
5. `MarkdownPreview` 组件内的代码块右上角增加"复制"图标按钮，点击后显示"已复制"反馈（2 秒后恢复）。
6. 当页面滚动超过 300px 时，显示右下角"回到顶部"浮动按钮（带动画）。
7. 文章右下角增加"沉浸阅读"按钮，激活后隐藏侧边栏和顶部导航（ESC 退出）。

### 3.2 首页内容分发优化

**问题：**
- 首页只有按时间排序的文章流，没有"推荐"或"精选"优先展示。
- 首页对未登录用户没有差异化内容（应展示热门而非最新）。
- 精选文章 `featured` 字段已存在，但首页没有精选文章专区。

**数据库：**

无需新增表，复用 `blog_article.featured` 和 `blog_article.featured_at`。

**后端接口扩展：**

```
GET /api/articles
  - 新增 sort 选项：featured（精选优先 + 时间倒序）
  - 新增 sort 选项：recommended（综合热度 = view*1 + like*3 + comment*5，取最近 7 天）
```

**前端：**

1. 首页 Feed 顶部增加 Tab 切换：最新 / 热门 / 精选。
   - 未登录：默认"热门"Tab。
   - 已登录：默认"关注"Tab（当前 `/following` 功能整合进来）；如果关注列表为空则降级到"热门"。
2. 精选文章在 Feed 中显示特殊标识（右上角 `⭐ 精选` 角标）。
3. 首页侧边栏增加"本周精选"小组件，展示 3 篇精选文章标题链接。

### 3.3 搜索能力增强

**问题：**
- 搜索接口只支持 MySQL `LIKE`，关键词分词不准确。
- 搜索结果没有高亮关键词。
- 没有"空搜索"状态的引导（有热门关键词，但缺少"猜你想搜"）。
- 搜索结果页切换 Tab 时，URL 不携带 Tab 信息（刷新后默认回到文章 Tab）。

**后端接口扩展：**

```
GET /api/articles
  - keyword 搜索增加 MATCH...AGAINST 全文索引支持（MySQL 8.0 FULLTEXT）
  - 如果全文索引命中为空，回退到 LIKE 模糊匹配
```

**数据库：**

```sql
ALTER TABLE blog_article
    ADD FULLTEXT INDEX ft_article_title_content (title, content)
    WITH PARSER ngram;
```

**前端：**

1. 搜索结果中文章标题和摘要中的关键词用 `<mark>` 高亮（后端在摘要字段返回 `highlightSummary`，或前端用正则替换）。
2. 搜索结果 URL 增加 `tab=articles|users|columns` 参数，刷新后保持 Tab 状态。
3. 搜索无结果时，下方增加"换个词试试"提示 + 5 个热门关键词快速入口。

### 3.4 标签页与分类页

**问题：**
- 有分类和标签数据，但没有独立的"分类页"和"标签页"，点击分类/标签只是触发搜索过滤器。
- 缺少"按标签浏览内容"的专属页面体验。

**数据库：**

无需变更（分类、标签表已有）。

**后端接口（新增）：**

```
GET /api/categories/{id}/articles
  - 查询某分类下已发布文章（分页，支持 sort）

GET /api/tags/{id}/articles
  - 查询某标签下已发布文章（分页，支持 sort）

GET /api/tags/hot
  - 返回使用次数 top 30 的热门标签
```

**前端：**

1. 新增路由 `/categories/:id` → `CategoryView.vue`（分类详情页）：展示分类名称、描述、下属文章列表。
2. 新增路由 `/tags/:id` → `TagView.vue`（标签详情页）：展示标签名称、文章数量、下属文章列表。
3. 文章卡片中的分类和标签均改为可点击链接，跳转到对应详情页。
4. 搜索页筛选区"分类"标签改为链接样式，点击可直接跳转到分类详情页。
5. 新增 `/explore` 页面（内容探索页）：展示所有分类 + 热门标签，作为发现内容的"目录"入口，加入导航。

## Phase 4：社区互动深化

目标：让用户之间的互动更有温度，增加平台黏性和留存。

### 4.1 通知系统增强

**问题：**
- 通知只有列表，没有实时推送（刷新页面才能看到新通知）。
- 通知类型缺少"被收藏了你的文章"（当前只有点赞、评论、关注）。
- Header 的通知角标只在页面刷新时更新，不会实时变化。
- 没有"系统通知"类型（平台公告、举报处理结果等）。

**数据库扩展：**

```sql
-- blog_notification.type 新增枚举值（通过应用层控制，无需 DDL）：
-- SYSTEM_NOTICE      -- 系统通知/平台公告
-- REPORT_RESOLVED    -- 举报处理结果通知

-- 新增系统公告表
CREATE TABLE blog_announcement (
    id           BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
```

## Implementation Order

执行顺序固定：

1. Phase 1（核心路径修补）— 直接影响用户留存，优先级最高
2. Phase 2（创作者能力增强）— 影响内容供给质量和作者留存
3. Phase 3（内容消费优化）— 影响读者留存和平台口碑
4. Phase 4（社区互动深化）— 增加平台黏性
5. Phase 5（运营与数据）— 支撑平台长期可运营

每个 Phase 执行方式：

1. 先改数据库 schema（仅新增，不修改现有列）。
2. 再补 domain / repository / application / controller。
3. 再补前端 API 层。
4. 再补页面和交互。
5. 最后补验收点自查。

每个 Phase 完成后保证：

- `mvn -q -DskipTests compile` 通过。
- `npm run build` 通过。
- 关键路径手工验收通过。
- 不引入原生 `alert` / `confirm`。
- 新增路由有正确的 `useHead` SEO 设置。
- 新增组件有 `data-testid` 属性用于 E2E 测试。

## Key Constraints

与 `11-设计v2.md` 相同，延续以下约束：

- 后端严格遵循 DDD 四层架构：Controller 不调 Mapper，Application Service 不含 SQL。
- 所有新增表继续遵循 `deleted_at`、`version`、`created_at`、`updated_at` 约定。
- 物理删除禁止，一律软删除。
- Application Service 写操作必须加 `@Transactional(rollbackFor = Exception.class)`。
- Mapper XML 必须定义 `BaseResultMap` 和 `Base_Column_List`。
- 前端 API 层统一走 `@/api/http.js` 封装，不允许组件直接 `fetch`。
- 不引入新的重量级依赖（如 ElasticSearch、Kafka、Redis Streams），优先用现有 MySQL + Redis 解决问题。

## Assumptions

- 忘记密码的邮件发送：第一阶段只 log，第二阶段再接入真实邮件服务（JavaMail / 第三方 SMTP）。
- SSE 通知推送：不引入 WebSocket，用 Spring 的 `SseEmitter` 实现，适合中低并发场景。
- 全文搜索：MySQL 8.0 FULLTEXT + ngram 分词器，不引入 ElasticSearch。
- 图片裁剪：前端引入 `cropperjs`（轻量，gzip 后约 50KB）。
- 文章版本历史：仅保存文本快照，图片不重复存储。
- 邀请码：第一阶段不强制邀请码注册，作为可选增长工具。
- 数据导出：同步生成 ZIP，超过 100 篇文章时返回 202 + taskId 轮询（简化异步任务，不引入 MQ）。
