# 博客网站项目开发规范

## 1. 概述

本文档是 `my-blog` 项目的强制开发规范，适用于后续所有 Spring Boot 后端开发、数据库设计、测试、代码审查和部署运维工作。

项目后端采用基于领域驱动设计的单体 Java 应用架构，并遵循《阿里巴巴 Java 开发手册》的核心规范。后续所有代码实现、重构和文档补充都必须优先遵循本文档。

## 2. 技术基线

| 类型 | 规范 |
|---|---|
| 架构风格 | DDD 单体应用，前后端分离 |
| 后端框架 | Spring Boot 2.7.18 |
| Java 版本 | JDK 8 |
| 持久层 | MyBatis 或 MyBatis Plus，优先 MyBatis |
| 数据库 | MySQL 8.0+ |
| 对象转换 | MapStruct 1.5.5.Final |
| 测试框架 | JUnit 5、AssertJ、Spring Boot Test |

## 3. 架构规范

### 3.1 分层架构

项目后端采用 DDD 四层架构：

```text
┌─────────────────────────────────────────────────────────────┐
│ 用户接口层 interfaces                                       │
│ Controller / REST API / DTO / 数据转换 / 参数校验 / 权限入口 │
├─────────────────────────────────────────────────────────────┤
│ 应用层 application                                          │
│ AppService / 用例编排 / 事务控制 / 领域事件发布              │
├─────────────────────────────────────────────────────────────┤
│ 领域层 domain                                               │
│ Entity / ValueObject / DomainService / Repository接口        │
│ DomainEvent / Specification / Factory                       │
├─────────────────────────────────────────────────────────────┤
│ 基础设施层 infrastructure                                   │
│ Repository实现 / 持久化 / 外部服务 / 消息 / 缓存 / 配置       │
└─────────────────────────────────────────────────────────────┘
```

依赖方向必须保持单向：

```text
interfaces -> application -> domain
infrastructure -> domain
infrastructure -> application
```

禁止事项：

- 领域层禁止依赖接口层。
- 领域层禁止依赖基础设施实现类。
- Controller 禁止直接调用 Mapper。
- Controller 禁止直接编写业务逻辑。
- 应用服务禁止堆积领域规则，业务规则应进入领域模型或领域服务。

## 5. Java 编码规范

### 5.1 代码格式

- 使用 4 个空格缩进，禁止使用 Tab。
- 单行不超过 120 个字符。
- 左大括号不换行，右大括号换行。
- 方法之间、逻辑块之间保留空行。
- 禁止通配符导入，例如 `import java.util.*`。
- 所有 `public` 类和 `public` 方法必须有 Javadoc。
- 类注释必须包含 `@author` 和 `@since`。

### 5.2 OOP 规范

- 抽象类以 `Abstract` 或 `Base` 开头。
- 接口实现类以 `Impl` 结尾。
- 禁止字段注入，必须使用构造器注入。
- 方法参数过多时使用 DTO、Command 或 Builder。
- 优先使用不可变对象。
- 包装类型比较必须使用 `equals`，禁止使用 `==`。
- 字符串常量比较时常量放左边，例如 `"PUBLISHED".equals(status)`。

### 5.3 集合规范

- 集合必须指定泛型，禁止裸类型。
- 集合初始化时尽量指定容量。
- 遍历 `Map` 时优先使用 `entrySet`。
- 返回空集合而不是 `null`。
- 空集合使用 `Collections.emptyList()`、`Collections.emptySet()`。

### 5.4 控制语句

- 所有 `if`、`while`、`for` 必须使用大括号。
- 嵌套层级不得超过 3 层，复杂逻辑使用提前返回。
- `switch` 必须有 `default`。
- 每个 `case` 必须 `break`、`return` 或 `throw`。
- 禁止嵌套三目运算符。

### 5.5 并发规范

- 线程池必须显式使用 `ThreadPoolExecutor` 或配置明确的 `ThreadPoolTaskExecutor`。
- 禁止使用 `Executors` 快捷方法创建线程池。
- 优先使用对象锁，避免类锁。
- 单例 Bean 的成员变量必须考虑线程安全。
- 并发 Map 使用 `ConcurrentHashMap`，禁止使用 `Hashtable`。

## 6. DDD 编码规范

### 6.1 实体和聚合根

- 聚合根负责维护聚合内部不变性。
- 领域对象应采用充血模型，业务行为写在领域对象中。
- 禁止为领域对象提供随意修改状态的 Setter。
- 外部对象引用优先使用 ID，不直接持有其他聚合根对象。
- 状态变更必须通过业务方法表达，例如 `publish()`、`offline()`、`delete()`。
- 领域事件由聚合根注册，应用服务负责发布。

示例：

```java
/**
 * 文章聚合根
 *
 * @author author
 * @since 1.0.0
 */
public class Article extends AbstractAggregateRoot<ArticleId> {

    private ArticleId id;
    private UserId authorId;
    private ArticleTitle title;
    private ArticleStatus status;
    private LocalDateTime updatedAt;

    private Article() {
    }

    public void publish() {
        if (ArticleStatus.DELETED.equals(this.status)) {
            throw ArticleException.statusError(this.status, ArticleStatus.PUBLISHED);
        }
        this.status = ArticleStatus.PUBLISHED;
        this.updatedAt = LocalDateTime.now();
        registerEvent(new ArticlePublishedEvent(this.id, this.authorId));
    }

    public ArticleId getId() {
        return id;
    }
}
```

### 6.2 值对象

- 值对象必须不可变。
- 字段优先使用 `final`。
- 通过构造器或静态工厂方法完成校验。
- 值对象不得暴露修改状态的方法。
- 金额必须使用 `BigDecimal`，禁止使用 `double` 或 `float`。
- 值对象相等性通过业务值判断。

### 6.3 领域服务

- 领域服务处理无法自然归属于单个实体或值对象的领域逻辑。
- 领域服务只依赖其他领域对象、领域服务或仓储接口。
- 领域服务禁止依赖 Controller、DTO、Mapper、DO。
- 领域服务方法必须表达业务概念。

### 6.4 应用服务

- 应用服务负责用例编排、事务边界、权限上下文协调和领域事件发布。
- 应用服务不写核心业务规则。
- 变更类用例必须添加 `@Transactional(rollbackFor = Exception.class)`。
- 查询类用例优先走 QueryService 或查询模型，不强制开启事务。
- 应用服务入参优先使用 Command 或 Query。

### 6.5 仓储

- 仓储接口定义在领域层。
- 仓储实现定义在基础设施层。
- 仓储接口返回领域对象，不返回 DO。
- 查询结果可能为空时返回 `Optional`。
- 仓储实现负责 DO 与领域对象转换。
- 禁止在 Mapper 或 XML 中写业务逻辑。

## 7. Controller 与接口规范

- Controller 只处理 HTTP 入参、参数校验、DTO 转换和响应包装。
- Controller 禁止直接访问 Mapper。
- Controller 禁止直接处理领域规则。
- 请求 DTO 放在 `interfaces.rest.dto.request`。
- 响应 DTO 放在 `interfaces.rest.dto.response`。
- DTO 与 Command、Query、Response 的转换放在 `interfaces.rest.mapper`。
- 所有请求参数必须使用 JSR-380 注解校验。
- 统一返回 `Result<T>`。

Controller 示例：

```java
/**
 * 文章 REST 接口
 *
 * @author author
 * @since 1.0.0
 */
@RestController
@RequestMapping("/api/articles")
public class ArticleController {

    private final ArticleAppService articleAppService;
    private final ArticleDtoMapper articleDtoMapper;

    public ArticleController(ArticleAppService articleAppService,
                             ArticleDtoMapper articleDtoMapper) {
        this.articleAppService = articleAppService;
        this.articleDtoMapper = articleDtoMapper;
    }

    @PostMapping
    public Result<ArticleCreateResponse> createArticle(
            @RequestBody @Valid CreateArticleRequest request) {
        PublishArticleCommand command = articleDtoMapper.toCommand(request);
        ArticleId articleId = articleAppService.createArticle(command);
        return Result.success(new ArticleCreateResponse(articleId.getValue()));
    }
}
```

## 8. 数据库规范

### 8.1 表设计

- 表名全小写，下划线分隔。
- 字段名全小写，下划线分隔。
- 主键统一使用 `id`。
- 主键类型使用 `BIGINT UNSIGNED`，自增或雪花 ID。
- 禁止使用业务字段作为主键。
- 所有表必须使用 InnoDB。
- 所有表字符集必须使用 `utf8mb4`。
- 所有字段必须有 `COMMENT`。

所有业务表必须包含：

```sql
id          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
deleted_at  DATETIME DEFAULT NULL COMMENT '删除时间(软删除)',
version     INT NOT NULL DEFAULT 0 COMMENT '版本号(乐观锁)'
```

软删除统一使用 `deleted_at IS NULL` 判断，禁止使用物理删除。

### 8.2 索引规范

- 主键必须存在。
- 业务唯一字段必须建立唯一索引。
- 索引名使用 `idx_表名_字段名`。
- 唯一索引名使用 `uk_表名_字段名`。
- 联合索引中区分度高的字段放左边。
- 单表索引数量原则上不超过 5 个。
- 禁止在频繁更新且区分度低的字段上滥建索引。
- `WHERE`、`ORDER BY`、`JOIN` 高频字段必须评估索引。

### 8.3 MyBatis 规范

- Mapper 只做数据访问，禁止业务逻辑。
- XML 必须定义 `BaseResultMap`。
- XML 必须定义 `Base_Column_List`。
- 查询必须过滤 `deleted_at IS NULL`。
- 更新必须处理乐观锁版本号。
- 删除必须使用逻辑删除。
- SQL 参数必须参数化，禁止字符串拼接。

## 9. 异常处理规范

### 9.1 异常体系

```text
Throwable
  Error
  Exception
    Checked Exception
    RuntimeException
      SystemException
      BusinessException
        DomainException
        ApplicationException
```

### 9.2 异常规则

- 禁止直接抛出裸 `RuntimeException`。
- 业务异常继承 `BusinessException`。
- 领域异常继承 `DomainException`。
- 应用异常继承 `ApplicationException`。
- 禁止捕获 `Throwable`。
- 捕获 `Exception` 后必须处理或重新包装。
- 禁止 `e.printStackTrace()`。
- 全局异常处理器统一返回 `Result<T>`。
- 业务异常记录 WARN 日志。
- 系统异常记录 ERROR 日志并包含堆栈。

## 10. 日志规范

- 日志门面使用 SLF4J。
- 日志实现使用 Logback 或 Log4j2。
- 日志必须使用 `{}` 占位符，禁止字符串拼接。
- 敏感信息必须脱敏。
- 日志文件至少保存 15 天。

日志级别：

| 场景 | 级别 | 要求 |
|---|---|---|
| 方法入口 | INFO | 记录方法名和关键参数 |
| 方法出口 | INFO | 记录结果摘要 |
| 关键步骤 | DEBUG | 记录中间状态 |
| 业务异常 | WARN | 记录业务原因和上下文 |
| 系统异常 | ERROR | 记录异常堆栈和上下文 |

推荐配置：

```yaml
logging:
  level:
    root: INFO
    com.myblog: DEBUG
    com.myblog.infrastructure.repository: DEBUG
  file:
    name: logs/application.log
    max-size: 100MB
    max-history: 30
```

## 11. 安全规范

- 所有外部输入必须校验。
- 请求 DTO 使用 JSR-380 注解，例如 `@NotBlank`、`@Size`、`@Email`。
- 密码必须加密存储，禁止明文保存。
- JWT 密钥禁止提交到代码仓库。
- 日志禁止打印密码、Token、身份证、手机号、邮箱等敏感信息原文。
- SQL 必须参数化，禁止拼接用户输入。
- 文件上传必须校验类型、大小和存储路径。
- 前端权限控制只用于体验，后端必须做最终权限校验。

## 12. 性能规范

- 事务粒度尽量小。
- 事务内禁止远程调用。
- 循环内禁止数据库查询，避免 N+1 问题。
- 批量写入使用批处理，单批次不超过 1000 条。
- 深分页必须使用游标或延迟关联优化，禁止直接 `LIMIT 100000, 10`。
- 热点数据可使用缓存，缓存键必须有统一命名。
- 异步处理必须使用自定义线程池。
- 数据库连接池使用 HikariCP，并配置合理连接数。

第一版暂不强制引入 Redis。若引入缓存，必须明确缓存失效策略和一致性边界。

## 13. 测试规范

### 13.1 测试分层

```text
        /\
       /  \
      / E2E\        少量端到端测试
     /------\
    /Integration\   中等数量集成测试
   /------------\
  / Unit Test    \  大量单元测试
 /----------------\
```

### 13.2 单元测试

- 使用 JUnit 5。
- 使用 AssertJ 断言。
- 测试方法名必须描述业务场景。
- 领域模型的核心业务方法必须有单元测试。
- 测试遵循 Given / When / Then 结构。
- 测试数据必须独立，不能依赖执行顺序。

### 13.3 集成测试

- 仓储、复杂查询和关键接口需要集成测试。
- 集成测试优先使用真实 MySQL 环境或 Testcontainers。
- 数据准备和清理必须明确。
- 核心功能测试覆盖率目标不低于 80%。

## 14. 部署与运维规范

- 必须提供健康检查接口或 Actuator 健康检查。
- 生产环境配置和密钥必须外置。
- 日志必须落盘并按时间滚动。
- 关键业务指标建议使用 Micrometer 收集。
- 后端服务必须支持优雅停机。
- 上传目录、日志目录、临时目录必须明确权限。

健康检查至少覆盖：

- 应用存活状态。
- MySQL 连接状态。
- 磁盘或上传目录可写状态。
