# My-Blog 项目亮点与技术难点

> 版本：1.0.0 · 更新时间：2026-05-13

---

## 一、架构设计亮点

### 1.1 严格的 DDD 四层分层架构

**亮点描述**：项目严格按照领域驱动设计（DDD）的四层架构组织代码，依赖方向单向内聚，无循环依赖。

```
Interfaces（接口层）
    ↓ 只能调用
Application（应用层）
    ↓ 只能调用
Domain（领域层）← Infrastructure（基础设施层）实现 Domain 接口
```

**技术难点**：
- **Repository 接口在 Domain 层定义，实现在 Infrastructure 层**——领域层完全不依赖 MyBatis 框架，使得核心业务逻辑与持久化技术解耦。即便将来换成 JPA 或其他 ORM，Domain 层代码无需修改。
- **应用层不包含业务规则**——所有 AppService 只做用例编排（`find → validate → call aggregate method → save → emit event`），真正的不变式校验都在聚合根内部（如评论 10 分钟编辑窗口、文章发布必填校验等），保证业务规则集中在领域层，便于测试。

**代码示例**：
```java
// Domain 层定义接口 —— 完全无框架依赖
public interface ArticleRepository {
    Optional<Article> findById(ArticleId id);
    void save(Article article);
}

// Infrastructure 层实现，依赖 MyBatis
@Repository
public class ArticleRepositoryImpl implements ArticleRepository {
    @Autowired ArticleMapper mapper;
    ...
}
```

### 1.2 工厂方法 + 私有构造函数保护聚合根不变式

**亮点描述**：所有 18 个聚合根的构造函数均为 `private`，只能通过 `create()` 和 `restore()` 两个静态工厂方法实例化。

**设计意图**：
- `create()`：用于新建场景，包含完整的业务规则校验（如用户名长度、密码非空）
- `restore()`：用于从数据库还原场景，跳过新建校验，直接赋值

**技术难点**：这种设计需要打破传统 JPA `@Entity` 映射的无参构造函数要求。项目采用 MyBatis 作为持久层，在 `PersistenceConverter`（转换器）中手动调用 `restore()` 完成实体→聚合根的映射，既保留了领域模型的纯净性，又支持了数据库还原。

**实际收益**：任何非法状态的聚合根无法被构造出来（例如不可能创建一个 content 为 null 的已发布文章），从根本上排除了一类 bug。

---

### 1.3 值对象封装主键，消除原始类型混用

**技术难点**：在大型系统中，`Long userId` 和 `Long articleId` 类型相同，编译器无法阻止参数传错位置，且方法签名可读性差。

**解决方案**：为每个聚合根的 ID 创建独立的值对象（18 个 VO），并实现 `equals/hashCode` 的语义等值：
```java
// 编译时就能发现错误，不再依赖运行时
user.follow(followerId: UserId, followingId: UserId)
// vs 传统写法，可能传反：
user.follow(followerId: Long, followingId: Long)
```

特殊值对象 `Email` 还内置了格式校验，确保邮箱格式合法性在领域层兜底。

## 二、安全机制亮点

### 2.1 自实现 JWT，无第三方依赖，防算法混淆攻击

**亮点描述**：`JwtTokenProvider` 完全从零实现 JWT 的生成与验签，只依赖 JDK 的 `javax.crypto.Mac` 和 `Base64`，无 jjwt / nimbus-jose-jwt 等第三方库。

**技术难点 —— 算法混淆攻击（Algorithm Confusion Attack）防护**：

JWT 规范允许 `alg` 字段由客户端指定。攻击者可将 Header 中的 `"alg":"HS256"` 改为 `"alg":"none"` 或 `"alg":"RS256"`，部分 JWT 库会相信这个字段而跳过验签。

**防护实现**：
```java
// 解析时显式校验 alg 字段，拒绝接受任何非 HS256 的 Token
String headerJson = Base64.getUrlDecoder().decode(parts[0]);
if (!headerJson.contains("\"HS256\"")) {
    throw new ApplicationException(UNAUTHORIZED, "不支持的Token算法");
}
// 随后才进行签名比对
String expectedSig = sign(parts[0] + "." + parts[1]);
if (!expectedSig.equals(parts[2])) {
    throw new ApplicationException(UNAUTHORIZED, "Token签名无效");
}
```

### 2.2 宽松/严格双模式认证（Lenient Authentication）

**亮点描述**：同一个认证拦截器支持两种工作模式，无需为公开接口和受保护接口写两套拦截逻辑。

**设计背景**：博客类平台需要支持「登录状态增强体验」——例如文章列表对匿名用户展示，但登录用户看到的同一列表中应附带自己的点赞/收藏状态。

**实现方案**：
```java
if (isPublicRequest(request)) {
    // 宽松模式：token 解析失败 → 静默回退为匿名，不抛异常
    authenticateIfPresent(header, queryToken, lenient: true);
    return true;
} else {
    // 严格模式：token 无效 → 直接抛 UNAUTHORIZED，拒绝访问
    authenticateIfPresent(header, queryToken, lenient: false);
}
```

**SSE 场景兼容**：浏览器的 `EventSource` API 不支持设置自定义请求头，无法携带 `Authorization: Bearer xxx`。项目支持通过 URL 参数 `?token=xxx` 传递 JWT，拦截器同时检查 Header 和 QueryParam，解决了 SSE 实时推送的认证问题。

---

### 2.3 用户状态实时校验，防止 Token 内禁用失效

**技术难点**：JWT 设计为无状态，Token 只要未过期就有效。如果管理员刚刚禁用了某个用户，该用户在 Token 有效期内（最长 24 小时）仍可正常访问。

**解决方案**：`ensureUserAvailable()` 在每次认证后都查库确认用户状态：
```java
private void ensureUserAvailable(Long userId) {
    User user = userRepository.findById(new UserId(userId))
        .orElseThrow(() -> new ApplicationException(UNAUTHORIZED, "请先登录"));
    if (!UserStatus.NORMAL.equals(user.getStatus())) {
        throw new ApplicationException(FORBIDDEN, "账号不可用");
    }
}
```

**权衡考量**：每次请求多一次 DB 查询，但 user 表数据简单（仅查 status 字段），且现代数据库主键查询命中缓冲池，延迟极低（< 1ms）。对于博客场景来说这个代价完全可以接受，换来的是「管理员禁用后立即生效」的可靠安全保障。

## 三、并发与性能亮点

### 3.1 点赞/收藏/关注的「软删除+复用」设计，避免高频写放大

**技术难点**：点赞是极高频操作，如果每次点赞/取消都执行 INSERT/DELETE，会导致：
1. 表碎片化严重（大量死行）
2. 索引膨胀（唯一索引频繁增删）
3. 无法保留操作历史

**解决方案**：所有互动关系（点赞/收藏/关注/专栏订阅）使用「软删除+记录复用」模式：
- 初次点赞 → INSERT 一条记录（`deleted_at IS NULL`）
- 取消点赞 → UPDATE `deleted_at = now()`（软删除，记录保留）
- 再次点赞 → `reactivate()`，UPDATE `deleted_at = NULL`（复用已有记录，避免重复 INSERT）

```java
public void reactivate() {
    this.deletedAt = null;
    this.updatedAt = LocalDateTime.now();
    this.version++;
}
```

**核心索引**：唯一索引 `uk_pair(article_id, user_id)` 加在软删除字段上，防止同一用户对同一文章出现多条非删除记录。

---

### 3.2 乐观锁版本控制，防止并发写丢失

**技术难点**：高并发场景下（如热门文章瞬间大量点赞），多个事务同时读到相同的 `like_count=100`，都执行 +1，最终写入时可能只有一次被保留，导致计数错误。

**解决方案**：全部聚合根表含 `version` 字段（乐观锁），每次更新时在 WHERE 子句中加入版本号校验：
```sql
UPDATE blog_article
SET like_count = like_count + 1, version = version + 1
WHERE id = ? AND version = ?  -- 版本不匹配说明被并发修改，更新0行
```
若更新 0 行则重试，确保计数最终一致。

---

### 3.3 精心设计的多级缓存，差异化 TTL 策略

**亮点描述**：项目使用 Caffeine 本地缓存，为不同访问模式的数据设置合理 TTL：

| 数据类型 | TTL | 设计理由 |
|----------|-----|----------|
| 未读通知数 | **30秒** | 角标强时效性，允许30秒延迟 |
| 首页引导 | **45秒** | 首屏高频，短TTL兼顾性能与新鲜度 |
| 分类/标签 | **3分钟** | 变化极少，较长缓存节省大量查询 |
| 推荐 Feed | **1小时** | 个性化计算代价高，结果相对稳定 |
| 排行榜 | **2小时** | 统计类数据变化慢，长缓存省资源 |

**容量保护**：热点数据缓存设置 `maximumSize`（精选文章128条、推荐feed 512条），防止缓存无限增长导致 OOM。

**主动失效**：文章发布/删除时通过 `HomePortalCacheInvalidator` 主动清除首页相关缓存，保证首页数据实时性。

---

### 3.4 CallerRunsPolicy 线程池，任务不丢失的优雅降级

**技术难点**：异步任务线程池队列满时，通常选择丢弃任务（`DiscardPolicy`）或抛异常（`AbortPolicy`），这对于「发送通知」等需要可靠执行的操作是不可接受的。

**解决方案**：采用 `CallerRunsPolicy`——队列满时，将任务退回给提交线程同步执行，本质上实现了「有损降级但不丢任务」：
```java
executor.setRejectedExecutionHandler(
    new ThreadPoolExecutor.CallerRunsPolicy() // 队列满 → 调用方线程执行
);
executor.setWaitForTasksToCompleteOnShutdown(true); // 优雅关闭：等待存量任务完成
executor.setAwaitTerminationSeconds(30);             // 最长等待30秒
```

## 四、业务设计亮点

### 4.1 评论二级嵌套的数据结构设计

**技术难点**：评论系统通常面临「无限嵌套 vs 二级嵌套」的权衡。无限嵌套实现复杂、查询递归耗时；完全扁平则无法展示对话上下文。

**解决方案**：采用 `rootCommentId + parentId` 双指针设计：
```
一级评论 A (rootCommentId=null, parentId=null)
  └─ 回复 B (rootCommentId=A.id, parentId=A.id)
  └─ 回复 C (rootCommentId=A.id, parentId=B.id) ← 楼中楼，但 root 始终是 A
```

**查询优势**：
- 一级评论列表：`WHERE root_comment_id IS NULL`（无递归查询）
- 楼中楼列表：`WHERE root_comment_id = {id}`（单次查询取全部回复）
- 前3条预览：`ORDER BY created_at ASC LIMIT 3`

**索引设计**：
```sql
idx_root_page_v2 ON blog_comment
    (root_comment_id, status, pinned DESC, pinned_at DESC, like_count DESC, created_at DESC)
-- 覆盖了「置顶优先 → 热门排序」的完整排序需求，无需二次排序
```

---

### 4.2 私信会话参与者的有序化设计

**技术难点**：用户 A 和用户 B 之间的对话，从 A 侧发起和从 B 侧发起，应该是同一个会话，不能重复创建。若以 `(sender_id, receiver_id)` 作为唯一键，则同一对用户会产生两行记录。

**解决方案**：参与者写入时强制按 ID 大小排序：
```java
// participantA 始终是 ID 较小的用户
Long a = Math.min(senderId, targetUserId);
Long b = Math.max(senderId, targetUserId);
Conversation.create(a, b)
```

唯一索引 `uk(participant_a_id, participant_b_id)` 确保两个用户之间有且仅有一条会话记录，完全消除重复问题。

---

### 4.3 文章定时发布机制

**技术难点**：定时任务需要在精确的时间点将 SCHEDULED 状态的文章改为 PUBLISHED，且要处理服务重启、任务积压等边缘情况。

**实现方案**：
1. 文章存 `scheduled_publish_at` + `status=SCHEDULED`
2. 专用索引 `idx_scheduled_publish(status, scheduled_publish_at)` 加速扫描
3. 后台定时任务每分钟扫描：
   ```sql
   SELECT * FROM blog_article
   WHERE status = 'SCHEDULED'
     AND scheduled_publish_at <= NOW()
     AND deleted_at IS NULL
   ```
4. 逐条调用 `article.publish()` + 持久化

**服务重启兜底**：即使服务在定时时间点发生重启，下次启动后定时任务仍会扫描到这些记录并发布，不会永久漏发。

---

### 4.4 文章版本历史与回滚

**亮点描述**：每次文章发布/更新都自动拍摄一份不可变快照（`blog_article_version`），支持任意版本回滚，类似 Git 的 commit 历史。

**设计要点**：
- `version_no` 自增，`(article_id, version_no)` 联合唯一
- 快照记录全量内容（title + content + summary + tags + ...），而非 diff，查询简单
- 回滚时以历史版本内容覆盖当前文章，并新增一条「回滚版本」的快照
- 按 `version_no DESC` 排序展示，最新版本排在最前

### 4.5 广告系统的精确时段控制

**亮点描述**：`AdCampaign.isActiveAt(LocalDateTime time)` 方法封装了广告有效性判断，支持任意时刻的判断：

```java
public boolean isActiveAt(LocalDateTime time) {
    if (!Boolean.TRUE.equals(enabled)) return false;
    if (deletedAt != null) return false;
    if (startAt != null && time.isBefore(startAt)) return false;
    if (endAt != null && time.isAfter(endAt)) return false;
    return true;
}
```

**实际意义**：
- 广告可精确设置分钟级的开始/结束时间（如活动从周六 20:00 开始）
- `startAt/endAt` 均可为空（null = 永久有效）
- 逻辑在聚合根内封装，应用层和接口层不需要知道时段判断细节

## 五、可观测性亮点

### 5.1 全链路 TraceId 追踪，精确到 12 位短 ID

**技术难点**：分布式日志中，单次请求会触发多个模块的日志输出（Controller、AppService、Repository），如何将这些分散的日志关联到同一次请求？

**解决方案**：
```java
// RequestTimingFilter 在请求进入时注入 traceId
MDC.put("traceId", traceId);

// Logback Pattern 配置，所有日志自动携带 traceId
"%d{HH:mm:ss} [%thread] %-5level [%X{traceId}] - %msg%n"

// 响应头回传，便于前端透传到下一个请求
response.setHeader("X-Trace-Id", traceId);
```

**12 位短 ID 的选型**：UUID 全长 32 位，人工排查日志时难以区分，且复制粘贴麻烦。12 位 UUID 前缀在单服务单机场景下碰撞概率极低（1/16^12 ≈ 1/281万亿），且足够简洁，排查日志时眼睛一扫即可识别。

**分布式兼容**：优先从请求头 `X-Trace-Id` 读取上游传入的 traceId，支持微服务链路追踪横向扩展。

### 5.2 BizLogHelper 自动脱敏的统一日志格式

**技术难点**：业务日志中频繁出现密码、邮箱、IP 等敏感信息，若开发者手动过滤容易遗漏，且格式不统一导致日志可读性差。

**解决方案**：`BizLogHelper` 提供了一套强制规范的日志工具方法，自动识别敏感字段并脱敏：

```java
// 使用方（只需按规范调用，无需关心脱敏细节）
log.info("{} | {} {} | 入参({}) | 结果({}) | {}",
    BizLogHelper.trace(),
    BizLogHelper.who(userId, nickname),
    "用户登录",
    BizLogHelper.params("account", account, "password", rawPwd),  // password 自动→****
    BizLogHelper.loggedIn(userId),
    BizLogHelper.elapsed(startTime));

// 输出：
// traceId=a1b2c3d4e5f6 | 用户[1001][张三] 用户登录 | 入参(account=test@qq.com, password=****) | 结果(tokenIssued=true, userId=1001) | 耗时(8.42ms)
```

**脱敏规则引擎**（`safeParamValue()`）：
- 精确匹配键名（password/token/code/jwt 等）→ `****`
- 键名含 email 或值含 `@` → 邮箱脱敏（`tes***@example.com`）
- 键名含 ip 或值符合 IPv4 格式 → 保留前两段（`192.168.*.*`）
- 大文本（> 200 字符）→ 只输出长度（`length=1234`）

---

### 5.3 管理员操作完整审计日志（含 before/after JSON 快照）

**亮点描述**：每次管理员操作（创建/更新/删除广告、禁用用户、处理举报等）都在 `blog_admin_log` 中记录：
- 操作者身份（userId + username）
- 操作 HTTP 方法和 URI
- 客户端 IP（支持反代场景读取 X-Forwarded-For 首个 IP）
- User-Agent（便于识别来自管理台还是 API 调用）
- **变更前快照（beforeSnapshot）**：操作前对象的完整 JSON 序列化
- **变更后快照（afterSnapshot）**：操作后对象的完整 JSON 序列化

**价值**：可以精确还原「管理员在某个时间点看到的数据」以及「改成了什么」，为运营纠纷提供可信的操作证据，远比只记录「谁在什么时间操作了什么」更有价值。

## 六、工程质量亮点

### 6.1 统一的 Result 包装与全局异常处理

**异常层次体系**：
```
RuntimeException
    ├── DomainException     ← 领域规则校验失败（如评论超时编辑）
    ├── ApplicationException ← 应用层用例错误（如文章不存在）
    └── BusinessException   ← 统一业务异常基类（被 GlobalExceptionHandler 捕获）
```

`GlobalExceptionHandler` 统一处理三类情况：
1. `BusinessException` → 提取 code + message，返回 `Result.fail(code, message)`（HTTP 200）
2. `MethodArgumentNotValidException` → 提取所有字段的校验错误，拼接为 `fieldName:errorMsg` 格式
3. `AsyncRequestTimeoutException`（SSE 超时）→ 返回 void，不输出响应体（防止 SSE 客户端收到乱码）
4. `Exception`（兜底）→ 打印完整堆栈，返回统一的「系统繁忙」提示，不暴露内部细节

### 6.2 邮件系统的内存队列 + 异步消费 + 限流 + 重试

**亮点描述**：邮件发送是耗时的 IO 操作（SMTP 连接延迟可达几百毫秒到数秒），若同步发送会显著拖慢注册 API 响应时间。项目设计了一个完整的异步邮件子系统：

```
发送请求线程               邮件队列消费者（每3秒轮询）
─────────────           ─────────────────────────────
1. 限流检查              4. 取出任务
2. 生成验证码            5. 渲染模板
3. 入队（非阻塞）──►  [LinkedBlockingQueue]  ──► 6. JavaMailSender.send()
   立即返回                                        7. 失败 → retry（最多3次）
                                                   8. 仍失败 → 记录日志，丢弃
```

**三层限流防刷**：
- IP 级别：20 秒内只能触发 1 次（防止同一 IP 循环发送）
- 邮箱级别：30 秒内只能向同一邮箱发送 1 次
- 每日上限：同一邮箱每天最多 10 封（防止被用来骚扰他人邮箱）

### 6.3 全局参数校验 + 类型安全的 DTO 设计

## 七、数据库设计亮点

### 7.1 MySQL ngram 全文索引支持中文搜索

**技术难点**：MySQL 默认全文索引（MyISAM `FULLTEXT` 或 InnoDB `FULLTEXT`）使用英文分词规则，对中文支持极差——中文词没有空格分隔，会被当作整体索引。

**解决方案**：使用 MySQL 8 内置的 `ngram` 分词插件：
```sql
ALTER TABLE blog_article ADD FULLTEXT ft_fulltext(title, summary, content) WITH PARSER ngram;
ALTER TABLE blog_column  ADD FULLTEXT ft_knowledge(title, summary, intro) WITH PARSER ngram;
ALTER TABLE blog_topic   ADD FULLTEXT ft_knowledge(title, summary, intro) WITH PARSER ngram;
```

`ngram` 将文本切割为 n 个字符的 gram（默认 n=2，即二元切割），使得「Java 框架」可以匹配「Java」「框架」「架构」等片段，实现了**无外部组件依赖的中文全文搜索**（不需要 Elasticsearch）。

### 7.2 冗余计数字段，避免昂贵的 COUNT 子查询

**技术难点**：`SELECT article.*, COUNT(like) FROM article LEFT JOIN like ON ...` 在百万级数据下耗时极长，且每次请求都执行会有性能问题。

**解决方案**：在 `blog_article` 表直接维护 `view_count`、`like_count`、`favorite_count`、`comment_count` 冗余字段，每次互动操作更新该字段（乐观锁保证最终一致）。

**代价与收益**：写操作略重（需要更新计数字段），但读操作极轻（无 JOIN，单行数据直接返回计数），对读多写少的博客场景非常合适。

### 7.3 覆盖索引消除回表查询

**技术难点**：评论列表查询需要复合排序（置顶 → 热度 → 时间），普通索引无法覆盖完整的 ORDER BY 条件，会产生大量回表。

**解决方案**：为评论表设计了专门覆盖查询的联合索引：
```sql
CREATE INDEX idx_root_page_v2 ON blog_comment
    (root_comment_id, status, pinned, like_count, created_at)
-- 注：pinned DESC, like_count DESC, created_at DESC 排序字段完全包含在索引中
```

该索引实现了「置顶优先 → 热门排序 → 时间排序」的完整查询，无需 filesort，查询性能显著优于普通索引。
